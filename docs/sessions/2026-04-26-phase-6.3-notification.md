## Work Summary

Closed Phase 6 by adding the notification subsystem in
`src/proposal/notification.py`. The dispatcher / notifier split mirrors
how `FeedbackLoop` decouples orchestration from auditing тАФ it lets a
single `notify_proposal(...)` call fan out across console, file, and
(future) Slack/push backends without coupling the engine to any one
runtime.

Components:

- `Notification` model тАФ UUID + timestamp + level + embedded
  `Proposal` + short message. Self-contained so a JSONL log on disk
  doesn't depend on the live ProposalHistory.
- `Notifier` Protocol тАФ `runtime_checkable` so test doubles work
  without subclassing.
- `ConsoleNotifier` тАФ writes a one-line banner to a configurable
  text stream (default `sys.stdout`).
- `FileNotifier` тАФ append-only JSONL at
  `data/notifications/proposals.jsonl`. Same shape as
  `src.feedback.audit.AuditLog`.
- `NotificationDispatcher` тАФ fan-out + `min_score` quality gate (the
  "good trading opportunities" half of FR-015) + per-channel failure
  isolation so one bad backend never silences the others.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| `min_score` filter at dispatcher level, not per-notifier | "Good opportunity" is a cross-cutting policy. Putting it at the dispatcher means callers can build a stricter dispatcher for paging vs a permissive one for the audit log without changing notifier code. |
| Per-channel exception isolation (`try/except Exception`) | If the file backend fills the disk we still want the user paged; if Slack 5xx's we still want the durable JSONL log. Logging + continuing is safer than aborting. |
| Embed full `Proposal` inside `Notification` (not a foreign-key id) | Self-contained log lines stay debuggable even if the underlying `ProposalRecord` is later deleted or rewritten. Matches the pattern in `AuditEvent`. |
| `Notifier` as a `Protocol` (not an ABC) | Lets tests use simple recording classes, lets the dashboard pass any object with `async send`, and avoids forcing subclassing on third-party adapters (e.g. a hypothetical Slack SDK wrapper). |
| Default `level=GOOD_OPPORTUNITY` from `notify_proposal` | The dispatcher is gated by `min_score` already тАФ anything passing the gate by definition is the "good opportunity" event FR-015 calls for. Callers who want to log routine proposals override to `INFO`. |

  notifications. Worth surfacing in the dashboard config UI when 7.x
