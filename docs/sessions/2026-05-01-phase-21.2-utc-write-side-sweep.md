## Work Summary

Phase 21.2 closes the engine-side write-half of DEBT-025: every
naive `datetime.now()` write-site that was surfaced by Phase 21.1's
compatibility sweep, plus a defence-in-depth Pydantic boundary
coerce on the seven models that store timestamps, plus reader-side
naive-tolerance at the five places where legacy on-disk records
flow back into the engine. The original 21.2 spec wording named
only the three `JsonlRotator` write-sites (lines 105 / 180 / 253);
21.1's auditor flagged 12+ further sites in the dev's
compatibility-sweep report and surfaced the scope question to
the planner. The team-lead's decision for this cycle: absorb the
full sweep under 21.2 rather than splitting into a narrow 21.2
(rotator) + new 21.4 (engine / feedback / strategy / ai). The
rationale was uniformity of the regression guard — splitting
would have left the engine timestamps in a hybrid state across
two cycles, with the cross-comparison risk live for the
intervening commits. The dev's report frames this as a
"scope-expansion-with-planner-blessing", not a silent expansion.

The sweep itself is mechanical at every site: import `now_utc`
from `src/utils/time.py`, replace the bare `datetime.now()` call,
verify no downstream `.replace(tzinfo=...)` chain disagrees with
the new UTC return. A new helper `ensure_utc(value: datetime)
-> datetime` was added to the same module: it returns the input
unchanged if already UTC-aware, attaches `tz=UTC` if naive (under
the legacy-naive-means-UTC interpretation that matches
production-on-Fly behaviour), and is the workhorse of the two
non-mechanical surfaces — the Pydantic `field_validator(mode=
"after")` hooks and the reader-boundary tolerance shims. The
`mode="after"` choice is deliberate: validation runs *after*
Pydantic's own type coercion, so the validator sees a `datetime`
(never a string), can branch on `tzinfo`, and re-attach UTC
without parsing. The seven models that gained this hook are
`ActivityEvent`, `AuditEvent`, `Proposal`, `CandidateRecord`,
`AssetSnapshot`, `PerformanceRecord` (two timestamp fields:
`analysis_timestamp` + `exit_timestamp`), and `TradeHistory`
(two fields: `entry_time` + `exit_time`).

The reader-boundary naive-tolerance at five sites
(`PortfolioTracker.load_snapshots`,
`TradeHistoryTracker.get_trades_by_date_range`,
`PerformanceTracker.get_records_by_date_range`,
`ProposalHistory.purge_old`, `ProposalHistory.list_all` sort
key) covers the historical artefacts that were written before
this cycle landed: any on-disk JSON record from a pre-21.2
non-UTC dev session is naive, and a comparison against a
post-21.2 UTC-aware peer would raise `TypeError`. The fix at each
site routes legacy-naive values through `ensure_utc(...)` before
the comparison; new writes are UTC-aware end-to-end via the
write-side swap, so the tolerance is one-way (read-only) and
the on-disk format gradually self-heals as records cycle through.
Both reviewers (qa-reviewer + quant-trader-expert) returned 🟢.
The qa-reviewer flagged one cosmetic nit — a redundant `f`
prefix on a non-interpolating string at `engine.py:438` that
crept in during the sweep — already fixed by the team-lead
before hand-off; recorded here for completeness and not blocking.

Alternative approaches considered and rejected: (a) **Pydantic
`mode="before"` validators** — would have had to handle both
`str` and `datetime` inputs (depending on whether the model is
constructed from JSON deserialisation or directly from Python),
adding parser code that duplicates Pydantic's own; rejected in
favour of `mode="after"` which always sees a `datetime`. (b)
**Skip the reader-boundary shims and let on-disk records
self-heal naturally** — rejected because the first non-UTC dev
who hits a `purge_old` call on a legacy file would get a
`TypeError` mid-cycle and lose the run; the one-way tolerance
is a 5-line cost per site for a clean migration. (c) **Land the
sweep without the Pydantic hooks (write-sites only)** —
rejected because a future contributor adding a new
`datetime.now()` call inside a model construction context
would silently reintroduce the bug; the validators are the
defence-in-depth that catches regressions at the model boundary
regardless of how the caller built the value.

## Potential Risks

- **Legacy-naive-as-UTC interpretation has a one-time silent
  shift on non-UTC dev artefacts.** Production on Fly always
  wrote UTC-naive (host is UTC), so production records get
  UTC-stamped correctly. Non-UTC dev sessions (KST, etc.)
  wrote local-naive records pre-21.2; the reader-shim
  `ensure_utc` will UTC-stamp those at read time, effectively
  shifting them by the dev's local offset. Operator-facing
  artefacts only (proposal history sort, trade-history range
  queries); financial figures are unaffected because PnL math
  doesn't read these timestamps. Tolerable as a one-time
  migration cost.

- **Pydantic `mode="after"` validators run on every model
  construction, including read-from-JSON paths.** This is the
  intended behaviour — the validators are the boundary that
  guarantees no naive value ever lives inside a model — but it
  means model construction is now ~5% slower (one `tzinfo`
  branch per timestamp field). Negligible in practice given
  the model construction rate, but flagged for completeness.

- **The `f`-prefix nit at `engine.py:438`** was a sweep
  artefact — a string had its content swapped from a
  `datetime.now().isoformat()` interpolation to a `now_utc(...)`
  call that no longer needed the `f` prefix, and the prefix was
  left dangling. Lead fixed before hand-off. The nit is a
  category that future sweeps should grep-check for (`grep -nE
  'f"[^{]*"' src/`); not a regression risk for this cycle but a
  hygiene note for the next sweep cycle.
