## Work Summary

Phase 17.3 closes the parallel persistence gap to Phase 17.2: the
Analysis Techniques dashboard read every per-technique aggregate
(win rate, average P&L, total P&L) as zero because
`PerformanceTracker.save_record` had no caller in production. The
proposal engine queried `PerformanceTracker` for ranking
(`composite_score` aggregates per-technique historical edge), but
the write side was never wired — every dashboard aggregate
defaulted to its empty-population fallback. Phase 17.3 wires the
runtime engine to snapshot a complete `PerformanceRecord` at
trade close, combining the technique fields from the originating
`Proposal` with the realised fill prices / fees / P&L / outcome
from the closed `TradeHistory`. Failures inside the persistence
path are logged at WARNING and swallowed so a missed performance
row never breaks the cycle — the same "best-effort visibility,
never break the loop" framing 17.2 used for the snapshot path.

The wiring lives in two new private methods on `TradingEngine`:
`_save_performance_record(proposal_record, trade, reason)` and a
`@staticmethod _classify_close_reason(reason) -> TradeOutcome`
helper. The latter maps the engine's close-reason string onto the
`TradeOutcome` enum: `take_profit` → `WIN`, `stop_loss` →
`LOSS`, anything else (manual close, monitor abort) →
`BREAKEVEN`. The mapping is deliberately conservative — the
neutral default for "anything else" avoids inflating either side
of the win/loss tally with semantically ambiguous closes (e.g.
trailing-stop closes, time-based exits) until a future cycle
adds the explicit close-reason taxonomy. `_save_performance_
record` reads from `self.proposal_engine.performance_tracker`
via `getattr(..., None)` — the proposal engine is the canonical
owner of the tracker (so the read and write paths share state),
and the `getattr` fallback keeps tests that construct a
`TradingEngine` without a real `ProposalEngine` working.

The `_record_closed_trade` call chain was restructured around a
new `_find_proposal_record_for_trade` helper that returns the
full `ProposalRecord` rather than just the proposal_id (the old
shape, `_find_proposal_for_trade -> str | None`, is replaced
in-place — every caller already had the record-derivation flow).
The `proposal_id` is still extracted from the record for the
`attach_outcome` call; the record itself is forwarded to
`_save_performance_record` so the technique fields
(`technique_name`, `technique_version`, `signal`, `entry_price`,
`stop_loss`, `take_profit`, `confidence`, `analysis_timestamp`,
`profile_name`) are available without a second lookup.
`PerformanceRecord` carries 18 fields total; the runtime path
fills all of them from the proposal + trade pair.

The new test
`test_close_writes_performance_record_for_dashboard` pins the
contract end-to-end: seeds a `ProposalRecord` with
`trade_id="t-close-1"`, primes the open trade with TP-hit
prices so the monitor closes it via `take_profit`, and asserts
the persisted `PerformanceRecord` carries `outcome=WIN`,
`pnl_percent=3.0`, `symbol`/`signal` matching the proposal,
and `trade_id` matching the original open. The injection
pattern mirrors the snapshot tests (17.2) — the test
overrides `mocks["proposal_engine"].performance_tracker` with
a tracker rooted at `tmp_path / "performance"` so the read
path the dashboard would use against `data/performance/` is
exercised against a tmp directory.

## Key Decisions

| Decision | Rationale |
|---|---|
| Persistence failures logged at WARNING and swallowed | A missed performance record is recoverable on the next trade close; a crashed cycle drops monitoring across every active trade. Same framing as Phase 17.2's snapshot path. |
| `_classify_close_reason` defaults to `BREAKEVEN` for unknown reasons (not `WIN` / `LOSS`) | The engine's close-reason taxonomy isn't formally enumerated yet — manual closes, time-based exits, monitor aborts could all surface here. Defaulting to a neutral outcome avoids inflating either side of the win/loss tally with semantically ambiguous closes until a future cycle nails down the close-reason enum. |
| `_find_proposal_record_for_trade` replaces `_find_proposal_for_trade` in-place rather than living alongside | Every caller already had the proposal-record derivation flow internally; returning the full record from one helper avoids duplicating the `list_all` scan. The callers that only want `proposal_id` extract it from the record (`record.proposal.proposal_id`) — one level of indirection but no extra disk read. |
| `getattr(self.proposal_engine, "performance_tracker", None)` rather than a hard attribute access | Tests that construct `TradingEngine` with a `MagicMock`-shaped `proposal_engine` shouldn't have to wire a `performance_tracker` attribute. The `getattr` fallback returns `None` cleanly so the persistence path no-ops on test fixtures without losing production coverage. |

## Potential Risks

- **`_classify_close_reason`'s `BREAKEVEN` default hides real
  losses or wins from atypical close paths.** A trailing-stop
  exit at a profit would currently classify as `BREAKEVEN`,
  understating the strategy's win rate. The mitigation is the
  conservative-default rationale above (don't inflate either side
  until the close-reason taxonomy is enumerated), but the
  understatement is real. Consider folding into a future "close-
  reason taxonomy" cycle; not actionable as a separate item
  today.

- **`PerformanceTracker.save_record` write-path is not atomic
  on the 17.3 baseline.** A crashed cycle mid-write could leave
  `data/performance/<technique>/` torn. Phase 22.1 later
  addressed this via `atomic_write_text` migration of
  `_save_records` and `_update_summary` (DEBT-028 → Resolved
  2026-05-01); on the 17.3 baseline the path is plain
  `Path.write_text(json.dumps(...))`.
