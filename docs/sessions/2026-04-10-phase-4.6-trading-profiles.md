## Work Summary

Added a second axis to trade construction: the *trading profile*.
A profile captures the risk-management and execution style —
risk_percent, max/default leverage, position-size cap, minimum R/R,
signal confidence floor, order type, and confirmation preference —
independent of the analysis technique that produced the signal.
Profiles load from YAML/JSON files under `trading_profiles/`, combine
with any `AnalysisResult` via `create_position_from_profile`, and
feed into `PerformanceTracker` so the same technique can be compared
across profiles (FR-005).

Shipped four bundled profiles: `conservative`, `moderate`,
`aggressive`, `scalping`. All four load and smoke-test on startup.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| TradingProfile is a separate dimension from analysis techniques | A technique answers "what is the market doing?"; a profile answers "how aggressively should I act?". Mixing them would force the same technique to be copied per risk preset. Keeping them orthogonal matches how traders actually talk about strategy. |
| `profile.accepts_signal()` is the only built-in filter (confidence + neutral rejection) | R/R is already validated by `TradingStrategy` downstream; having the profile duplicate that check would double-emit the same error. Keeping the profile's filter narrow makes the composition predictable. |
| `create_position_from_profile` returns None for a filtered signal, raises for a failed validation | None means "skip, no error" — this is the normal outcome for low-confidence signals. Raising is reserved for user-visible configuration problems (R/R too low, price inconsistency). |
| `profile_name` on `PerformanceRecord` is optional (defaults to None) | Preserves backwards compatibility with existing records that were written before profiles existed. Query methods handle None as its own bucket. |
| Added `model_post_init` cross-field check for `default_leverage <= max_leverage` | Pydantic field-level constraints can't express this; catching it at construction time stops a class of config bugs before runtime. |
| `require_confirmation` on the profile is a declared preference, not a hard override | NFR-012 says live trading *always* requires confirmation. The `LiveTrader` honors that regardless of the profile. Profiles encode intent; dashboard/CLI layers can read it to choose between interactive and batch callbacks in paper mode. |
| YAML precedence over JSON when both exist for a name | YAML is the more expressive "authoring" format; JSON files are likely machine-generated exports. Picking YAML first means hand edits always win in a mixed directory. |
| `ProfileLoader.load_all_profiles` logs and skips bad files | One corrupt profile file shouldn't prevent the dashboard from showing the rest. Errors still surface as logs for debugging. |

## Potential Risks

- **Confirmation preference is advisory, not enforced end-to-end.** If
  a future orchestrator layer reads `profile.require_confirmation ==
  False` and decides to inject an auto-approve callback into
  `LiveTrader`, NFR-012 compliance would weaken. The current
  `LiveTrader` always calls its callback, which is the correct place
  to enforce NFR-012 — keep it that way.
- **`profile_name` is just a free-form string on `PerformanceRecord`.**
  There's no referential integrity between stored performance records
  and the profile files on disk — if a profile is renamed or deleted,
  old records still carry the old name. This is deliberate
  (historical audit value), but dashboards should treat the field as
  opaque metadata.
- **`load_all_profiles` swallows errors silently from the caller's
  perspective.** The log line is the only signal; if the dashboard
  cares about bad files, it can call `list_profiles()` +
  `load_profile()` individually.
