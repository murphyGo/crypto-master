## Work Summary

Built the user-interaction layer that sits between the headless
`ProposalEngine` (Phase 6.1) and the user. New module
`src/proposal/interaction.py` provides:

- `format_proposal()` — pure renderer that produces a CLI banner
  similar in shape to `LiveTrader.default_confirmation`, including
  proposal id, prices, R/R, and the full score breakdown.
- `default_decision_prompt()` — async stdin prompt asking
  accept / reject (and an optional reason on reject); uses
  `asyncio.to_thread` to keep the event loop unblocked.
- `ProposalRecord` / `ProposalDecision` — the persisted shape, with
  seams (`trade_id`, `outcome_pnl_percent`, `outcome_recorded_at`)
  for FR-014's "actual performance" once trades close.
- `ProposalHistory` — JSON-per-proposal repository under
  `data/proposals/`. Mirrors `FeedbackLoop.save_state`/`load_state`.
  Provides `save`, `load`, `list_all(decision=...)`,
  `attach_outcome(...)`.
- `ProposalInteraction` — orchestrator with `present()` and
  `present_batch()` so a CLI driver (or the eventual dashboard) is
  one call away from a fully persisted decision.

The persistence layer is decoupled from the CLI: a future Streamlit
dashboard can pass its own `decision_callback` and reuse the same
`ProposalHistory`.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Three loosely-coupled units (display / prompt / history) instead of one class | Future dashboard reuses persistence without inheriting CLI prompt. |
| JSON-per-proposal under `data/proposals/`, not append-only JSONL | Read-mostly access pattern (load + update on outcome). Matches `FeedbackLoop.save_state` pattern. JSONL would force replay just to update an outcome. |
| `attach_outcome(proposal_id, trade_id, pnl_percent)` exposed but not called from this layer | Phase 6.2 records decisions; wiring it up to `LiveTrader.close_position` belongs to whoever assembles the CLI driver in a later phase. Keeps the seam without coupling layers prematurely. |
| `ProposalDecisionInput` returned by callback (not a bare bool) | Lets the callback bundle a rejection reason atomically — no second async hop in the orchestrator just to fetch reasons. |
| `present` does not auto-execute accepted proposals | Execution (LiveTrader / PaperTrader) has its own NFR-012 confirmation flow; double-prompting would be hostile. The accepted record is the trigger; downstream wiring decides whether to execute. |

## Potential Risks

- **Concurrent writers** — two processes saving the same proposal id
  would race. Not a concern today (single-process CLI), but
  noteworthy if the dashboard ever runs alongside an automated loop.
- **`data/proposals/` growth** — unbounded; needs a retention policy
  before live deployment. Out of scope for 6.2.
