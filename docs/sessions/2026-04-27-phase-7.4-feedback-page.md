## Work Summary

Closed Phase 7 with the Feedback Loop page. The page surfaces:

- **Summary cards** — total, awaiting approval, promoted, discarded,
  errored. Counts derived from every `CandidateRecord` snapshot
  under `data/feedback/state/`.
- **Candidates table** — newest-first, with the robustness verdict
  rolled up to a `PASS` / `FAIL` / `—` badge.
- **Per-candidate detail** — full record fields (source path,
  backtest run id, robustness summary, failed gates, decision
  reason).
- **Audit timeline** — `AuditLog.filter(candidate_id=...)` rendered
  as a chronological table.

The page is **read-only by design**. CON-003 enforces user approval
inside `FeedbackLoop.approve(...)` itself; surfacing approve/reject
buttons in the dashboard would either bypass that path or duplicate
its logic. Operators continue to use the loop's API for promotion;
the dashboard just shows them what's pending.

A standalone `load_candidate_records(state_dir)` function (mirrors
`FeedbackLoop.list_pending` minus the pending filter) lets the page
read state without instantiating a full loop, which would require
all of the improver / backtester / analyzer / gate dependencies just
to glance at JSON snapshots.

Phase 7 is now complete; cross-check generated at
`docs/cross-checks/phase-7-dashboard.md`.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Standalone `load_candidate_records` (not via `FeedbackLoop.list_pending`) | The loop constructor requires improver / backtester / analyzer / gate. The dashboard only needs to read JSON snapshots — instantiating those dependencies just to call `list_pending()` would couple the page to the entire feedback subsystem. The standalone loader stays in the page module since it's a presentation concern, not core domain logic. |
| Read-only page (no approve / reject UI) | CON-003 requires user approval; routing it through `FeedbackLoop.approve` keeps the audit log + state file rewrites + frontmatter-status flip atomic. A dashboard button would have to reimplement that path, which both increases risk and divides the truth. Operators run `approve()` from a CLI / script; the dashboard tells them *what* to approve. |
| Robustness rendered as `PASS` / `FAIL` / `—` | A boolean is too compact (which side is true?), and the full summary is too long for a table cell. Three-value badge is the right granularity for the at-a-glance row; the per-candidate detail block shows the full summary. |
| Audit timeline sorted oldest-first | Reads as a story: GENERATED → BACKTESTED → GATE_PASSED → APPROVED → PROMOTED. Newest-first ordering would force operators to read the lifecycle backward. |

## Potential Risks

- **Audit log read on every render.** Each navigation re-reads
  `data/audit/feedback.jsonl` from scratch. That's fine while the
  log is small but linear in event count. Same caching candidate as
  the other 7.x pages.
- **Per-candidate detail pulls the full audit log just to filter.**
  `AuditLog.filter` reads everything and filters in memory. With
  thousands of events this becomes wasteful; if it bites,
  `AuditLog` can grow a streaming filter or an index.
- **No approve/reject UI.** Intentional (CON-003), but operators
  driving the loop via the dashboard now have to switch to a CLI
  for the final decision. A future phase could surface a "copy this
  command" button (`feedback-loop approve <id> --by alice`) without
  taking on the actual mutation responsibility.
