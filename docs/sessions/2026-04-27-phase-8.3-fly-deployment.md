## Work Summary

Packaged the runtime engine + Streamlit dashboard for Fly.io. Single
machine, single container, single volume — both processes run from
`start.sh` so they share local disk (which the dashboard needs to
read what the trader writes). Phase 8 closes with this; cross-check
generated at `docs/cross-checks/phase-8-runtime.md`.

Key shipping pieces:

- `Dockerfile` — Python 3.13-slim base + Node 18 (for the Claude
  Code CLI per NFR-002) + Python deps from `requirements.txt`. Uses
  `tini` as PID 1 for clean signal forwarding.
- `start.sh` — bash supervisor that launches both processes, traps
  SIGTERM/SIGINT to propagate to children, and exits when either
  child exits so Fly recycles the machine. Acceptable supervision
  strategy for a paper-trading deploy; cheaper than supervisord.
- `fly.toml` — `app = "crypto-master"`, `primary_region = "nrt"`
  (Tokyo, close to Binance + Bybit), single shared-cpu-1x VM at
  1 GB. HTTP service maps to Streamlit's port 8080 with the
  built-in `/_stcore/health` healthcheck. `auto_stop_machines =
  false` because the trader must run 24/7.
- `.dockerignore` — strips dev artifacts (.venv, __pycache__,
  pytest/coverage caches, docs/, tests/, .env, .git) so the build
  context stays small.
- `docs/deployment.md` — operator runbook covering prerequisites,
  first-time setup (`fly launch --no-deploy`, `fly volumes create`,
  `fly secrets set`), Cloudflare Access for dashboard auth, region
  picks, cost estimate (~$3-4 USD/mo), rollout/rollback, and live
  trading switch-over notes.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| One container, both processes (not Fly's `[processes]` block) | Fly volumes are tied to a single machine — they can't be mounted by two process groups. The dashboard reads what the trader writes (proposals, trades, audit log, activity log, snapshots), so they must share local disk. One container is the cheapest correct shape. |
| `start.sh` + `wait -n` (not supervisord) | Recovery latency is ~one machine boot when either process dies — acceptable for paper. Saves ~3 MB image space and one runtime dep. supervisord would auto-restart individual processes but adds complexity that's overkill for personal use. |
| `tini` as PID 1 | Standard container best practice — reaps zombie children and forwards POSIX signals correctly. Without it `wait -n` works but signal-driven graceful shutdown doesn't propagate cleanly. |
| Claude CLI installed via npm (not curl-script) | The `@anthropic-ai/claude-code` npm package is the documented install. Debian Bookworm's apt `nodejs` is 18.x (Claude Code's minimum). One layer, predictable. |
| Region `nrt` (Tokyo) | Close to Binance + Bybit primary endpoints; latency matters for the eventual live mode. Easy to swap via `primary_region` + `fly volumes fork` if cheaper regions matter more than latency. |
| Cloudflare Access (not Tailscale serve, not basic auth) | Free tier covers personal use; one-time PIN to email needs no IdP setup. Tailscale is also fine but requires sidecar config; basic auth in front of Streamlit is brittle (no rotation, no audit). |
| Image size accepted at ~700-900 MB | Python + Node + npm + numpy/pandas wheels carry weight. Multi-stage builds could trim ~200 MB but at the cost of legibility. Premature optimisation. |

## Potential Risks

1. **Build verification is local-only.** The first `fly deploy` may
   hit issues that don't reproduce on macOS Docker Desktop (arch
   mismatch, missing system lib). Run `fly deploy` in a test
   environment first; deployment.md flags this with a "rollback
   first deploy" note.
2. **Claude CLI auth path is assumed.** The container relies on
   `ANTHROPIC_API_KEY` being honored by `claude -p` in headless
   mode. If Anthropic changes the auth path, the engine will start
   logging `cycle_errored` events with Claude-related strings —
   monitor the activity log on first deploy.
3. **Single machine = single point of failure.** Fly machines are
   reliable but not redundant. Acceptable for paper; live should
   add a hot standby in a second region.
4. **Image size ~700-900 MB.** Slow first deploy; subsequent deploys
   benefit from layer caching on requirements.txt.
5. **`auto_stop_machines = false` means we always pay for the
   machine.** That's the point (the trader must run 24/7), but it
   does mean cost is fixed at ~$3-4/mo even when nothing is
   happening. Cheaper than the alternative (state in S3 + serverless
   trader).
