# Session Log: 2026-04-28 - Phase 11.2 - OHLCV Cache for Multi-Technique Scan

## Work Summary

Phase 11.2 resolves DEBT-002 (Low): Phase 10.6's multi-technique
scan re-fetched OHLCV per technique, producing N×M `get_ohlcv`
calls per symbol per cycle (N = techniques, M = timeframes) versus
1×M previously. Quant flagged temporal-consistency drift (technique
A could see candle T while technique B sees T+δ if a candle rolled
mid-cycle) and rate-limit pressure at scale. The fix is a per-call
OHLCV cache keyed by `(symbol, tf)` threaded through the proposal
engine's call chain — local dict per call, no module/class state.

The non-obvious choice was the threading shape. Two approaches were
considered: a hoist (compute the OHLCV dict above the technique
loop in `_propose_all_for_symbol` and pass the dict down — cheaper
to read but requires every caller to know which timeframes will be
needed before strategy selection) versus a per-call cache threaded
as a parameter through the entry-point chain (Option A — more
parameter plumbing but lets each `_build_proposal_for_strategy`
fetch lazily and share through the dict). Dev took Option A:
`propose_bitcoin` / `propose_altcoins` instantiate the cache,
`_propose_for_symbol` / `_propose_all_for_symbol` thread it through,
and `_build_proposal_for_strategy` checks-then-fetches per
`(symbol, tf)`. Both single-TF and multi-TF branches consume it.
The legacy `_select_best_technique` path (kept live for op-emergency
rollback per Phase 10.6) also threads the cache for consistency, so
there is no per-path divergence in fetch behaviour.

Fetch counts verified end-to-end: 3 sym × 4 tech drops 12 → 3
fetches; multi-TF (2 strategies sharing 3 TFs) drops 6 → 3;
sequential 2× `propose_bitcoin` stays at 2 (no leak between calls,
confirming cache lifetime is exactly one public-method invocation);
legacy 3 sym × 1 tech stays at 3 (no regression). 4 new tests
extend `tests/test_proposal_engine_multi_technique.py` (1083 →
1087). ruff clean; mypy zero new errors on `engine.py`. The
existing 23 `test_proposal_engine.py` tests pass unchanged. Type
hints use PEP 604 union syntax. QA grep-verified no global state
landed; the cache is genuinely per-call.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Option A (thread cache as parameter) over the hoist alternative | Hoisting requires every caller to know which timeframes will be needed before strategy selection — fine for single-TF, awkward for multi-TF where the TF set is per-strategy. Threading as a parameter lets `_build_proposal_for_strategy` fetch lazily and dedupe through the shared dict, and naturally extends to the multi-TF branch. The parameter plumbing cost is one-time; the lazy semantics are recurring. |
| Per-call cache, not per-engine | Strategy decisions need fresh data each cycle. A per-engine cache would risk stale-data correctness defects across cycles. Lifetime is exactly one `propose_bitcoin` / `propose_altcoins` invocation; verified by the sequential 2× test (2 fetches, not 1). |
| Legacy `_select_best_technique` path also uses the cache | Phase 10.6 kept the legacy single-best-technique path live as op-emergency rollback. Threading the cache through it too means there is no per-path divergence in fetch behaviour — operators flipping the rollback flag get the same fetch count as the new path on the legacy code's 1-technique-per-symbol shape. |
| PEP 604 union syntax for the cache type hints | Project convention post-11.1; consistent with the rest of the engine module. Keeps mypy clean without `from __future__ import annotations` workarounds. |

## Potential Risks

- **Cache lifetime contract is implicit** — the cache is local to
  the public-method frame. If a future cycle factors `_propose_for_symbol`
  / `_propose_all_for_symbol` out into a longer-lived helper (e.g. a
  reusable `ProposalRound` class), the lifetime contract has to be
  re-asserted explicitly or the per-cycle freshness invariant
  silently breaks. Tested today by the sequential 2× call test;
  future refactors must keep that test green.
- **Legacy path coverage is incidental** — the legacy
  `_select_best_technique` path threads the cache but its fetch
  count was already 1×M (best technique = 1 technique). The
  3 sym × 1 tech regression test confirms no new regression; it
  does not exercise the cache's hit path on the legacy shape, so
  any future legacy-path change that needs cache hits has to add
  its own coverage.
