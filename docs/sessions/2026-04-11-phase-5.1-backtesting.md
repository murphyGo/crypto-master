## Work Summary

Built a walk-forward backtester that runs any ``BaseStrategy``
against a series of historical OHLCV candles and produces a
structured ``BacktestResult``. The engine reuses the existing
``TradingStrategy`` for sizing / R/R validation and plugs directly
into ``TradingProfile`` so a single technique can be evaluated across
multiple risk styles. Intra-candle SL/TP checks, per-side slippage,
and taker fees are modeled explicitly. Results persist under
``data/backtest/{run_id}/result.json`` with round-trip JSON encoding
of Decimals and datetimes.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Strategy only ever sees `ohlcv[:i+1]` at candle `i` | Eliminates look-ahead bias — the canonical mistake in naive backtests. The `ControllableStrategy` test double asserts this by recording the call indices. |
| SL/TP checked *before* running the next analysis | Matches live trading where pending risk orders fire before the next decision. |
| Pessimistic tie-breaker: SL wins when both SL and TP fall inside the same candle | Backtests that assume TP wins produce optimistic numbers. A conservative bias here is safer for capital-preservation decisions. |
| Entries fill at the current candle's close after slippage (not next open) | Simpler and matches the semantics callers get from live paper/live traders, which also execute at current close. The strategy at candle `i` already knows close[i], so the signal's trigger is consistent with the fill. |
| Slippage always works *against* the trader (both entry and exit) | Avoids accidentally making slippage a free subsidy. Slippage is a cost, not noise. |
| End-of-data force-close with `close_reason="end_of_data"` | Forgetting to close lingering trades would silently hide drawdown at the end of the series. |
| Reuse `TradingStrategy.create_position` for sizing | Keeps R/R + position-size-cap + leverage validation in one place. Backtest and live both inherit the same rules. |
| Profile integration mirrors `create_position_from_profile` | A profile-driven backtest should behave identically to a profile-driven live run. Code path uses the same profile helpers. |
| `_close_trade` returns `(BacktestTrade, balance_delta)` where delta *excludes* the entry fee | Entry fee was already deducted when the trade opened; returning delta = raw_pnl − exit_fee keeps the running balance tracking exact without re-counting. |
| `allow_concurrent_positions=False` default | Simplest sensible semantics for a first release; can be revisited when multi-symbol portfolio backtests arrive. |
| Results under `data/backtest/{run_id}/result.json` (one dir per run) | Gives future backtest artifacts (plots, equity curves, per-candle balance logs) a place to live alongside the primary result without changing the load/save contract. |

## Potential Risks

- **Fill model is optimistic on limit-order-style signals**: The
  engine treats every fill as if a market order filled exactly at the
  candle's close (plus slippage). Techniques that signal a limit-price
  entry which the market never actually touches will still "fill"
  in this simulation. Revisit when limit-order semantics become
  important.
- **Position quantity is computed against the signal's entry price,
  not the post-slippage fill price**: `TradingStrategy.calculate_position_size`
  uses `analysis.entry_price` for the risk-per-unit calculation. The
  actual slipped fill is slightly different, so the realized risk
  per trade deviates from `risk_percent` by the slippage delta.
  Documented here; acceptable for a v1.
- **Balance goes negative in pathological edge cases**: The engine
  deducts the entry fee as long as it's ≤ balance, but large leveraged
  losses can drive `balance` below zero without triggering
  liquidation semantics. A proper margin-call simulation is a future
  enhancement; for now the backtester reports the number it computes.
- **No multi-symbol or multi-strategy support yet**: One
  `Backtester.run` call = one strategy + one symbol. Portfolio-level
  backtests are out of scope for 5.1.
