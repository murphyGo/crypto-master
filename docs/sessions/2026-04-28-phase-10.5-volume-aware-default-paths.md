# Session Log: 2026-04-28 - Phase 10.5 - Volume-Aware Default Paths

## Overview

- **Date**: 2026-04-28
- **Phase**: 10 - Operational Maturation
- **Sub-task**: 10.5 - Volume-Aware Default Paths

## Work Summary

Closed the persistence-loss defect Cycle 1's Fly diagnosis surfaced: `fly.toml`
mounts the persistent volume at `/data` but the Dockerfile sets `WORKDIR=/app`,
so any module that defaulted its storage to a relative `Path("data/...")`
literal was writing to the ephemeral container root (`/app/data/...`) instead
of the volume. Every machine recycle (auto-deploy, OOM, host migration) wiped
the activity log, audit log, feedback-loop state, proposal history, file
notifier output, and portfolio history — producing dashboard timeline holes
and breaking the audit trail Phase 5.5 was designed to provide.

The fix is mechanical: replicate the `PerformanceTracker` /
`TradeHistoryTracker` pattern across the six offending components. Each
`__init__` now accepts a `data_dir: Path | None = None` kwarg and, when no
explicit path is supplied, derives the default from `Settings.data_dir` at
construction time. The pattern preserves backward compatibility — existing
callers that pass an explicit `tmp_path` (the test convention) hit the same
path branch they always did; the `data_dir` branch only fires when no path is
supplied. `src/trading/portfolio.py` was already correct; the dev added a
documenting comment so a future grep for the pattern lands consistent results.

The fidelity decision worth flagging: the new `data_dir` kwargs are
keyword-only after `*`, which prevents accidental positional misuse and keeps
the call shape self-documenting. Each touched component picked up one new
"respects `Settings.data_dir`" test that uses `monkeypatch.setenv` +
`reload_settings` + a double-assert in `try/finally` to guarantee the global
settings cache stays clean between tests.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Replicate the `PerformanceTracker` / `TradeHistoryTracker` pattern verbatim | Both already thread `Settings.data_dir` correctly. Inventing a new abstraction (e.g. a `DataDirMixin` or a dedicated path-resolution helper) would introduce a third pattern across the codebase for marginal gain. The audit trail value is in pattern repetition — future readers grep `data_dir: Path \| None = None` and find six identical implementations. |
| Keyword-only `data_dir` kwarg (after `*`) | Prevents accidental positional misuse against callers that already pass other arguments positionally. Self-documents the optional nature at the call site. |
| Resolve `Settings.data_dir` at construction time, not at write time | Matches the existing pattern. Lazy resolution would mean the path could change mid-run if `Settings` reloaded; eager resolution makes the path immutable for the lifetime of the component, which is the property the rest of the codebase already relies on. |
| `src/trading/portfolio.py` left unchanged except for a comment | The pattern was already correct. Changing it for cosmetic uniformity would be churn without benefit. The doc comment makes the consistency explicit so a future audit doesn't re-flag it as a gap. |
| `DEFAULT_*_PATH` module-level constants kept | The dev flagged these as "now documentation-only — could rename to `_LEGACY_*` in a future sweep". Skipped this cycle: renaming would require updating every internal reference and test fixture, the constants don't actively mislead (they still represent the project-relative default), and the auditor judged the noise-to-value ratio as too high to record as debt. |

## Potential Risks

- **Existing Fly deployments still hold the empty `/app/data/...` files from before this fix.** This cycle changes the *default* on fresh construction; it does not migrate any historical data sitting in the ephemeral container root. The first redeploy after this lands will start writing to the volume, but anything written before the redeploy is gone. Acceptable: Cycle 1's diagnosis already established that those files were being wiped on every machine recycle anyway, so there is no historical data to migrate.
- **`DEFAULT_*_PATH` module-level constants are now documentation-only.** Any external code (tests, scripts) that imports those constants directly and uses them as the actual write path will write to the wrong location. No such external code exists today — `grep` across the repo confirms the constants are referenced only inside the module that defines them. Flagged here so a future grep that surfaces a new caller knows the constant is a legacy artefact, not a contract.
- **The settings-cache double-assert pattern in tests is load-bearing.** If a future test forgets the `try/finally` + `reload_settings()` cleanup, the global settings cache will leak across tests and produce flaky failures depending on test ordering. The six new tests are consistent; a lint/check that enforces the pattern would be over-engineering for now, but worth revisiting if a seventh component picks up the same kwarg.
