## Work Summary

Phase 15.1 closes the diagnostic gap that produced the 2026-04-28
misdiagnosis cycle: while monitoring the Phase 12 redeploy, the
`crypto_master.trading.strategy` logger emitted lines like
`Created position: short BTC/USDT @ 76750.0` that read as "a trade
was opened" — but the line is actually emitted from
`TradingStrategy.create_position` during proposal sizing
(`src/trading/strategy.py:473`), called from
`ProposalEngine._propose_for_symbol`, **before** the threshold gate
runs. The actual trade-open log (`Opened paper position: ...`) lives
in `PaperTrader.open_position` at `src/trading/paper.py:546` and
never fired because every proposal was rejected at
`auto_approve_threshold = 1.0` while composite scores topped out
around 0.35. Net result: an hour of mistaken "trades are happening"
reads on logs that turned into "why does the dashboard show 0?" —
both assumptions wrong. The fix is two safe, mechanical changes plus
the operator action (setting `ENGINE_AUTO_APPROVE_THRESHOLD=0.30` via
Fly secrets) that unblocks actual execution. The operator action is
out-of-scope for 15.1; only the code clarity follow-up is in scope.

The first change is a verb rename at `src/trading/strategy.py:474`:
`"Created position: ..."` → `"Sized position candidate: ..."`. Same
fields, same verbosity, only the verb changes — `position.side`,
`symbol`, `position.entry_price`, `position.quantity`, `leverage`,
`position.stop_loss`, `position.take_profit` all preserved bytewise.
The `PaperTrader.open_position` "Opened paper position" log
(`src/trading/paper.py:546`) stays unchanged so the two events are
clearly distinct in `fly logs` greps — operators can pin "sized
candidate but didn't open" vs "actually opened" with a verb match.

The second change extends the dashboard so the *why* of an empty
trade table is visible. `TradingSummaryMetrics` (the Phase 13.1
TypedDict in `src/dashboard/pages/trading.py`) gains
`proposals_rejected_threshold_count: int`. `build_summary_metrics`
accepts an optional `proposal_history: ProposalHistory | None = None`
(defaults to `ProposalHistory()` so existing callers don't need to
wire it up — backward-compat), reads via `history.list_all()`, and
counts records where `decision == "rejected"` and `rejection_reason`
matches `^composite \d+\.\d+ below threshold \d+\.\d+$` — the exact
format from `RuntimeEngine._auto_decide` at
`src/runtime/engine.py:586` (`f"composite {composite:.4f} below
threshold {threshold:.4f}"`). Cap-rejected records (Phase 12.1,
reason starts with `"symbol "`) are explicitly excluded so the
metric stays interpretable — they represent a different cause (cap
saturation) and lumping the two together would hide which gate is
biting. A new private helper `_count_threshold_rejections` wraps
`history.list_all()` in `try/except` so a malformed proposals dir
warns + returns 0 rather than crashing the whole Trading page
render. Layout: `st.columns([3, 1])` next to "Active Positions" so
an operator seeing 0 open positions immediately sees how many
proposals were rejected and why; `render(...)` accepts a
`proposal_history=` injection point for tests, defaulting to
`ProposalHistory()`.

No ADR — log-string rename + one new dashboard field is mechanical
clarity, not a component-shape decision. Phase 15 stays open: the
phase exists for diagnostic-clarity items as they emerge from
production observation, and 15.1 is the only sub-task today; future
operational misdiagnoses may surface additional 15.x items, so the
phase remains in-progress rather than sealed.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Log verb rename only — same fields, same verbosity (vs deleting + recreating the line, vs adding a structured event_type field) | The misdiagnosis was caused by the *verb* reading as past-tense execution ("Created"). Replacing it with "Sized position candidate" makes the intent unambiguous in raw log greps without disturbing field shape, log level, or downstream parsers. A structured `event_type` would be more rigorous but is out-of-scope for a 4-line clarity fix; the log-aggregation layer (Phase 8.x) is the right home for that if it's ever needed. |
| Anchored regex `^composite \d+\.\d+ below threshold \d+\.\d+$` (vs substring match `"below threshold"`, vs structured rejection-cause enum on `ProposalRecord`) | The threshold-gate reason is emitted from a single call site (`RuntimeEngine._auto_decide`) with exact `f"composite {composite:.4f} below threshold {threshold:.4f}"` formatting. Anchoring the regex pins the metric to that emit and excludes cap-rejected records (Phase 12.1, reason starts with `"symbol "`) which represent a structurally different cause. A structured enum would be cleaner long-term but would require a `ProposalRecord` schema migration; deferred until a third rejection cause emerges. |
| `proposal_history: ProposalHistory \| None = None` injection (vs requiring callers to construct one, vs reading directly from disk inside `build_summary_metrics`) | Keeps Phase 13.1's `build_summary_metrics(trades, snapshots)` signature backward-compatible — existing callers and unit tests (other than the ones explicitly updated) don't need to change. `None` resolves to `ProposalHistory()` at call time, which uses `Settings.data_dir` so the metric reads the same dir the dispatch loop writes to. Tests inject a `tmp_path`-based history for hermetic counting. |
| Defensive `try/except` in `_count_threshold_rejections` warns + returns 0 (vs propagating, vs returning `None`) | A malformed proposals dir must not crash the entire Trading page — operators rely on the trade table and equity curve regardless of proposal-history health. Returning 0 with a warning log surfaces the issue without blocking the render. `None` would propagate into the metric card and require another None-handling layer; integer 0 is the cleanest contract. |
| Cap-rejected records explicitly excluded from the count | Phase 12.1's per-symbol cap is a structurally different gate than the composite threshold. An operator looking at the metric for "why is the trade table empty?" wants to disambiguate "scores too low" from "cap saturated" — lumping the two would hide which gate is biting. If cap-rejected count becomes operationally relevant, it's a separate metric. |
| `st.columns([3, 1])` beside "Active Positions" (vs a fourth summary card at the top) | The semantic relationship is "open positions vs proposals-rejected-by-threshold" — both answer "what's in the table right now?". Putting the metric next to the table where the answer matters is more discoverable than burying it in the top-of-page summary row. |
| No ADR | Log-string rename + one new dashboard field is mechanical clarity, not a component-shape decision. The `Notifier`, `ProposalHistory`, and dashboard component contracts are all unchanged. ADRs are for component-boundary decisions, not for narrating diagnostic-clarity follow-ups. |

## Potential Risks

- **The metric counts rejection records *cumulatively* across the
  proposals retention window** (Phase 11.4's purge horizon, default
  90 days). An operator glancing at the card sees lifetime-window
  threshold rejections, not "this hour" or "since the last open
  trade". For the 2026-04-28 cycle's diagnostic intent (139 rejected
  in one hour producing the misread) this is fine — any non-zero
  count answers "no, the table isn't empty because of a bug; here's
  the count of proposals the threshold gate caught". A sliding-window
  variant (last hour / last 24h) would be a Phase 15.x follow-up if
  the operator workflow needs it.
- **The anchored regex pins the metric to the exact format string
  from `RuntimeEngine._auto_decide`**. If a future cycle tweaks the
  reason format (e.g. extra precision, different wording, structured
  reason), the metric will silently return 0 instead of the real
  count. Not catastrophic — the page still renders, the trade table
  is still correct — but the misdiagnosis-prevention value of 15.1
  evaporates until the regex is updated. Mitigation lives in the
  test seed: `test_summary_metrics_counts_threshold_rejections` uses
  the verbatim `"composite 0.4500 below threshold 1.0000"` string,
  so a format change in the producer would surface as a test failure
  on the consumer side. Tight coupling acknowledged; format is
  stable since Phase 6.x.
- **Verb-rename is grep-visible but not enforced by the test suite**
  — no test asserts the literal log text. A future refactor that
  reverts the verb to "Created position" would not be caught by CI;
  it would only be caught by an operator reading `fly logs`. Adding a
  log-text regression test was considered but rejected as
  over-engineered for a one-line fix; the dev plan's change-history
  row preserves the rationale for any future archaeologist.
