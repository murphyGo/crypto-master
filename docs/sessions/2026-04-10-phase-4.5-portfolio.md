## Work Summary

Added portfolio aggregation on top of the existing
`TradeHistoryTracker`. The new `PortfolioTracker` is stateless with
respect to balances — callers (PaperTrader / LiveTrader / backtester)
supply balances and current prices on each call, and the tracker
combines them with realized P&L summed from closed trades plus
unrealized P&L computed from open trades. Snapshots are persisted
under `data/portfolio/{mode}/snapshots.json`, keeping NFR-008's
per-mode separation consistent with how trades are already stored.

Key types: `AssetSnapshot` (persisted point-in-time state),
`Portfolio` (live view, not persisted), `PortfolioTracker`
(orchestrator + persistence).

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| PortfolioTracker is stateless w.r.t. balances — caller passes them in | PaperTrader owns `PaperBalance` and LiveTrader reads from the exchange. Centralizing balance state in PortfolioTracker would either duplicate or fight the trader-specific views. Stateless aggregation is simpler and the existing `PaperTrader.get_all_balances()` / `exchange.get_balance()` APIs fill that role. |
| Reuse `TradeHistoryTracker` for P&L roll-ups instead of new storage | Closed-trade P&L already lives in `data/trades/{mode}/`; realized P&L is a sum, not a new source of truth. Avoids drift between `portfolio/` and `trades/`. |
| Positions with no current price are skipped (not zeroed) in unrealized P&L | A missing ticker from a stale price feed should not silently zero-out a whole portfolio. Skipping keeps the surviving positions' marks accurate; the caller can flag missing symbols separately. |
| Unrealized P&L uses leverage but not fees | Fees are charged on close and already deducted inside `TradeHistory.calculate_pnl()` for closed trades. Mark-to-market of open trades represents "what we'd get at market price before close fees" — matches how exchanges display unrealized P&L. |
| `Portfolio` is a plain view model, not persisted | Portfolio is computed on demand from balances + prices + trades; persisting it would duplicate the sources. `AssetSnapshot` is the persisted form. |
| `load_snapshots` returns `[]` on corrupt JSON | Keeps dashboards resilient to a bad file write. A log line records the failure. |
| Decimal round-tripping via explicit `_snapshot_to_dict` | Pydantic handles load-time coercion, but JSON encoding of Decimals is not native — manual str() conversion keeps precision exact. |

## Potential Risks

- **Realized P&L is global-to-mode**, not time-windowed. A snapshot's
  `realized_pnl` reflects *all* closed trades in that mode up to now,
  not just those closed before the snapshot timestamp. This matches
  dashboard intent ("lifetime realized P&L in this mode") but means
  historical snapshots are informational, not auditable after the fact.
- **No automatic snapshot trigger** — snapshots only exist when the
  orchestrator (future loop code) calls `record_snapshot`. Phase 7
  will wire that into the dashboard / feedback loop.
- **Quote currency assumption**: `total_equity` only counts the quote
  currency balance. Non-quote balances (e.g. BTC in a USDT-quoted
  portfolio) are recorded but not converted to quote. Multi-currency
  valuation would need per-asset price fetches and is out of scope
  for this phase.
