## Overview

## Work Summary

Phase 21.1 closes the adapter read-side of DEBT-025 (High): the
`datetime.fromtimestamp(ms / 1000)` no-tz construction at four
sites each in `src/exchange/binance.py` and `src/exchange/bybit.py`,
and the `JsonlRotator._coerce_timestamp` read-side normalisation.
A single helper module `src/utils/time.py` was added with two
functions: `from_unix_ms(ms: int) -> datetime` returning a UTC-aware
`datetime` (`tz=UTC`), and `now_utc() -> datetime` wrapping
`datetime.now(tz=UTC)`. Every adapter call-site that previously
constructed a tz-naive `datetime` from exchange-returned millisecond
timestamps now routes through `from_unix_ms`; on a non-UTC host
(e.g. KST) the returned `datetime` carries `tzinfo=UTC` and the
wall-clock UTC value matches the input ms — verified by
`time_machine.travel(..., tz_offset=9)` regression tests in both
adapter test modules. The fix is mechanical at every site: import
the helper, replace the bare `fromtimestamp` call, leave the
downstream `timestamp.replace(...)` chain untouched (the existing
chain is now a no-op on the UTC-aware return rather than a silent
local-tz reinterpretation).

The senior-dev's pre-cycle inspection surfaced one non-obvious
detail: `JsonlRotator._coerce_timestamp` is the read-side coercer
(parses incoming records' string / numeric timestamps back into
`datetime`), distinct from the write-side `datetime.now()` calls
at lines 105 / 180 / 253 that derive the active-month token. 21.1's
spec scope explicitly covers the read-side coercer; the write-side
swap is Phase 21.2's territory and was deliberately left for the
next cycle to avoid scope creep. The dev verified this split by
re-reading the rotator and confirmed the read-side coercer was the
only `JsonlRotator` site touched. Both reviewers (qa-reviewer +
quant-trader-expert) returned 🟢 with no follow-ups required for
21.1 itself.

The compatibility sweep the dev ran while in there flagged 12+
remaining naive `datetime.now()` write-sites across the codebase
that go beyond the original 21.2 spec wording (which mentions only
`JsonlRotator`). Verbatim from the dev's report:
`src/runtime/engine.py:649`, `src/feedback/loop.py` (~6 sites),
`src/proposal/interaction.py` (~3 sites),
`src/strategy/performance.py:247/380/966/1002`,
`src/ai/improver.py:334`. These are dormant-on-Fly-UTC, live-on-KST-
dev failure modes of the same shape as DEBT-025 — the tz-naive write
either lands in the wrong rotation bucket or compares incoherently
against a UTC-aware peer. The auditor surfaces this as a scope
expansion for Phase 21.2's planner cycle rather than editing the
21.2 spec text directly (planner territory). Whether 21.2 absorbs
all 12+ sites or splits into a 21.2 (rotator narrow) + 21.4 (engine /
feedback / strategy / ai sweep) is a planner call.

Alternative approaches considered and rejected: (a) **broader fix
in 21.1** — would have absorbed Phase 21.2 + part of 21.3 into one
cycle; rejected because the audit-trail value of the three-sub-task
split is precisely that each closure is independently verifiable
and reviewable; (b) **leverage `datetime.UTC` import and skip the
helper** — rejected because a typed helper makes the intent
explicit at every call site (`from_unix_ms` reads as "convert
exchange ms to UTC-aware datetime", whereas
`datetime.fromtimestamp(ms / 1000, tz=UTC)` reads as boilerplate
that a future contributor might mechanically copy without the
`tz=` argument); (c) **backfill all 12+ write-sites in 21.2 itself
without a planner pass** — rejected because the spec wording is
narrow and silently expanding scope inside an implementation cycle
is exactly the drift this project's three-role split is designed
to prevent.

## Key Decisions

| Decision | Rationale |
|---|---|
| Single helper module `src/utils/time.py` (not `src/exchange/_time.py`) | DEBT-025's suggested resolution offered both paths; `src/utils/time.py` keeps the helper exchange-agnostic so `JsonlRotator` (and Phase 21.2's runtime / feedback / strategy / ai callers) consume the same module without an `exchange` package import dependency. |
| `from_unix_ms` does NOT take a `tz` parameter | Forces UTC at the helper boundary. A future caller that needs a different tz must do the `astimezone()` conversion explicitly downstream; the helper's contract is "exchange-ms → UTC-aware datetime", full stop. |
| Read-side coercer fix in 21.1, write-side `datetime.now()` deferred to 21.2 | Maintains the three-sub-task split's audit-trail value (each closure independently verifiable). Mixing read + write in 21.1 would have ballooned the diff and made the reviewer's job harder. |
| Compatibility sweep findings surfaced as planner-territory follow-up, not absorbed | Spec wording for 21.2 mentions only `JsonlRotator`; silently expanding scope inside an implementation cycle is the drift the three-role split prevents. Planner decides whether 21.2 absorbs all 12+ sites or splits. |
| Helper named `from_unix_ms` (not `from_exchange_ms` / `from_ms`) | "Unix ms" is the canonical name for the `int` epoch-ms format the exchanges return; "exchange" implies the domain is exchange-only (which Phase 21.2 will violate). |

## Potential Risks

- The `JsonlRotator._coerce_timestamp` read-side now returns
  UTC-aware `datetime`s while the write-side at lines 105 / 180 /
  253 still calls bare `datetime.now()` (tz-naive). Any
  comparison between a freshly-written rotator timestamp and a
  read-back coerced timestamp will now mix naive-vs-aware and
  raise `TypeError`. The dev confirmed no such cross-comparison
  exists in the current codebase (audited
  `JsonlRotator.read_with_retention` and the rotator's internal
  callers), but Phase 21.2 should land before any new caller is
  added that does this comparison. Surfaced for 21.2's
  implementation cycle.

  `src/runtime/engine.py`, `src/feedback/loop.py`,
  `src/proposal/interaction.py`, `src/strategy/performance.py`,
  and `src/ai/improver.py` go beyond the original 21.2 spec wording.
  If Phase 21.2 lands as written (rotator-only), DEBT-025 stays
  Active for those sites and the bug remains live on non-UTC dev

  cycle.

- `freeze_time` / `time_machine` test infrastructure is now load-
  bearing for the regression guard; if the upstream library
