## Motivation

Phase 5 produced a closed-loop technique-evolution system, so the
project now has a population of validated, performance-tracked
analysis techniques. What it lacked was a way to *use* those
techniques opportunistically — to scan the market on demand, decide
which technique fits which symbol, and surface a ranked list of
trade ideas to the user.

Phase 6.1 fills that gap. The engine is intentionally headless: it
returns data structures, not user-facing output. CLI display,
accept/reject handling, and proposal history are 6.2's job;
notifications are 6.3's. Keeping 6.1 free of I/O concerns makes it
trivially testable and reusable across the eventual CLI, dashboard
(Phase 7), and notification paths.

## Work Summary

### `src/proposal/engine.py` (new)

The single new module hosts four public types and one engine class:

- **`ProposalScore`** — surfaces every factor that fed the ranking
  (`confidence`, `win_rate`, `sample_size`, `expected_value`,
  `sample_factor`, `edge_factor`, `composite`). Operators can
  explain *why* a proposal ranks where it does without recomputing.
- **`Proposal`** — the actual trade idea: symbol, timeframe,
  technique name/version, signal, fully-priced entry/SL/TP, sized
  quantity, leverage, R/R ratio, score, and the analysis reasoning
  string. UUID and timestamp are auto-assigned at construction.
- **`ProposalEngineConfig`** — tunables (timeframe, OHLCV limit,
  default balance, leverage, risk percent, sample-size threshold,
  no-history score factor). All have conservative defaults.
- **`ProposalEngineError`** — base exception for misuse (e.g.
  `top_k < 1`).
- **`ProposalEngine`** — the orchestrator. Two public entry points
  (`propose_bitcoin` for FR-011, `propose_altcoins` for FR-012)
  share a private `_propose_for_symbol` worker.

### Best-technique selection

`_select_best_technique(symbol)` filters strategies down to those
whose `info.symbols` includes the target symbol (or has no symbol
filter), then ranks them by:

1. **Edge factor** — `max(0, avg_pnl_percent)`, descending.
   Negative-EV techniques contribute zero, so they tie at the bottom
   and never beat a marginal positive-EV technique.
2. **Sample size** — `total_trades`, descending. Wins ties.
3. **Lex-first name** — deterministic last-resort tiebreaker.

If no technique has *any* history, the engine falls back to the
lex-first applicable strategy with `perf=None`. This lets a brand-new
system still produce proposals (priced by analysis confidence alone,
discounted by `no_history_score_factor`) instead of returning an
empty list and silently doing nothing.

### Score formula

```
sample_factor = min(1, sample_size / min_trades_for_full_score)
edge_factor   = max(0, avg_pnl_percent)            # in % units
composite     = confidence × edge_factor × sample_factor   if perf
              = confidence × no_history_score_factor       if perf is None
```

The formula deliberately avoids combining confidence with win-rate
or expected-return through some opaque non-linearity. Each factor is
a clearly-defined gate: high confidence + proven edge + enough
history all multiply together. A zero in any of the three drops the
proposal to the bottom.

### Sizing

`TradingStrategy.create_position` already does R/R validation,
position sizing, and max-position capping. The engine delegates
entirely; no math is duplicated. R/R floor enforcement (default 1.5)
naturally rejects low-quality analyses without the engine having to
implement a parallel filter.

### Error handling

- **Exchange errors** propagate from `propose_bitcoin` (single-symbol
  intent) but are caught + logged + skipped in `propose_altcoins`
  (multi-symbol scan keeps going).
- **Strategy errors** (`StrategyError`) — same: single-symbol path
  returns None; multi-symbol path keeps scanning.
- **`TradingValidationError`** (e.g. R/R below floor, neutral signal)
  — always returns None for that symbol; the caller learns about it
  through the absence of a proposal in the result list.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Engine is headless (returns `Proposal`, never prints/persists) | Lets 6.2 (CLI / accept-reject), 6.3 (notification), and Phase 7 (dashboard) all consume the same engine without it being coupled to any one of them. |
| Promotion target = `TradingStrategy.create_position` | Sizing logic + R/R floor enforcement already lives there; duplicating would be a regression. The engine is a thin orchestrator over existing primitives. |
| Score = product of three normalized factors, not weighted sum | Multiplicative score has the nice property that any one factor going to zero drops the proposal — exactly what we want for "no edge", "no confidence", or "no history". A weighted sum would let a technique with zero edge still rank high on confidence alone. |
| Cold-start picks lex-first name (not random) | Deterministic = testable + reproducible. Important when several "fresh" techniques exist and there's no signal to prefer one. |
| Exchange/strategy errors skip in altcoin scan, propagate in single | Single-symbol caller wants to know what went wrong; multi-symbol caller wants a best-effort batch result. Different intents, different policies. |
| `info.symbols=[]` (empty list) treated as "any symbol" | Some strategies are designed to be symbol-agnostic. Existing convention defaults to `["BTC/USDT"]`, but an explicit empty list reads as "no filter" rather than "no symbols supported." |
| Exchange not opened by the engine | Matches `Backtester` and `LiveTrader` — the lifecycle belongs to the caller (`async with`). The engine just consumes a connected exchange. |

## Potential Risks

- **Score formula is heuristic, not validated.** The defaults
  (`min_trades_for_full_score=20`, `no_history_score_factor=0.5`)
  are educated guesses. Should be tuned once we have real proposal
  history to evaluate against — Phase 6.2 will start producing that.
- **No portfolio-level risk caps.** If three altcoin proposals all
  recommend long positions, taking all three concentrates risk in
  one direction. The caller decides how many proposals to act on;
  the engine doesn't track aggregate exposure. Acceptable for now;
  Phase 7 dashboard may surface this as a UI concern.
- **`avg_pnl_percent` is a noisy edge proxy.** A technique with 20
  trades averaging +2% per trade ranks above one with 100 trades
  averaging +0.5% per trade — even though the latter has more
  evidence of edge. A future refinement could use Sharpe or a
  Bayesian shrinkage estimator. Out of scope for 6.1.
- **No caching of OHLCV or performance data.** Each scan re-fetches
  from the exchange; each `_select_best_technique` re-reads
  performance summaries. Not a problem at current scan rates; if
  Phase 7 dashboards start polling, add a TTL cache.
