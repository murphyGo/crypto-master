## Work Summary

Phase 16.1 closes two prod-observed chasulang failures from the
2026-04-28 redeploy: (a) every successful Claude return parsed with a
`KeyError: 'signal'` because the chasulang_ict_smc.md template returns
the actionable trade nested under `trade.*` rather than flat
top-level, so the parser never saw a `signal` field; and (b) at
`2026-04-28T15:02:15Z` a chasulang retry timed out at 360s and the
engine then sat silent for 12+ hours until the operator manually
restarted, suggesting the prior `asyncio.create_subprocess_exec` +
`asyncio.wait_for` path raised the timeout but didn't actually kill
the child process — the wrapper's declared timeout was a lie. Both
failures rendered chasulang effectively disabled in production and
posed an open wedge risk for the engine. The fix is two surgical
changes inside `src/ai/claude.py`; no new framework abstractions, no
ADR.

The first change extends `_parse_response` with a
`_normalize_trade_fields` helper that promotes nested `trade.*` keys
(`signal`, `entry_price`, `stop_loss`, `take_profit`, `confidence`,
`reasoning`) to the top level when present. Precedence is "trade
sub-dict wins over top-level when chasulang shape is detected,
top-level preserved for back-compat" — the canonical chasulang shape
nests the trade decision alongside structural analysis frames
(`external_structure`, `liquidity_map`, `order_blocks`,
`wait_conditions`); none of those structural keys are themselves the
trade, so the `trade` block is the single source of truth when it
exists. Take-profit precedence is `trade.take_profit` >
`trade.take_profit_1` > nothing — `take_profit_1` is the closer,
conservative target, deliberately picked over the stretch
`take_profit_2` which is far more likely to give back open profit
than be hit. The original `trade` sub-dict is left intact in the
returned result so downstream callers that want the full nested view
(e.g. to read `take_profit_2`) still can. When neither path carries a
`signal`, the parser raises `ClaudeParseError` with a message that
explicitly names both candidate paths (`'signal'` top-level and
`trade.signal`) so operators can quickly identify which template
needs fixing.

The second change rebuilds `_execute_cli_once` on top of the blocking
`subprocess.Popen` API (run via `asyncio.to_thread` so we don't block
the event loop). The new path drives the timeout via
`proc.communicate(timeout=...)` rather than `asyncio.wait_for`; on
`subprocess.TimeoutExpired` it explicitly calls `proc.kill()`
(SIGKILL — not the soft-terminate of the prior path), then
`proc.wait(timeout=5)` as a bounded reap-confirmation step. If even
SIGKILL fails to reap the child within 5s, the wrapper raises a
distinct `ClaudeTimeoutError` with `"SIGKILL"` in the message so
operators can tell that failure mode apart from a normal timeout in
logs. `ClaudeTimeoutError` continues to carry `attempt_number` (Phase
14.1 propagation) — both the SIGKILL-succeeded and SIGKILL-failed
branches preserve it. `FileNotFoundError` is re-raised unchanged for
the caller to convert into `ClaudeNotFoundError`. The conversion of
the entire test suite from `asyncio.create_subprocess_exec` /
`AsyncMock` mocks to `subprocess.Popen` / `MagicMock(spec=Popen)`
mocks accounts for ~80% of the test churn — behaviour under test is
unchanged, only the mock surface moved.

No ADR — bug fix to existing component. The `ClaudeCLI` public
contract is unchanged: same `analyze` signature, same exception
types, same retry semantics, same `attempt_number` propagation.
Phase 16 is a compact one-sub-task phase that closes both prod
defects together; with no other open 16.x sub-tasks it is sealed in
this cycle and the phase cross-check runs inline.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Promote `trade.*` to top level (vs requiring downstream `StrategyTechnique` consumer to read both shapes) | Single canonical shape downstream is cheaper than a fanout at every consumer; the chasulang nest-vs-flat split is a prompt-template shape concern, not a strategy-framework one. Promotion at the parse layer keeps `src/strategy/loader.py`'s `AnalysisResult` construction unchanged and back-compat for `sample_prompt.md` / `simple_trend_analysis.md` is bytewise (their flat shape has no `trade` sub-dict, so the helper is a no-op). |
| Pick `take_profit_1` over `take_profit_2` when both present | TP1 is the closer, more conservative target; TP2 is the stretch target. In live trading, holding for TP2 trades a much higher chance of giving back open profit (mean reversion past TP1) for a higher payoff in the tail case. Single-target downstream code (`AnalysisResult.take_profit`) only carries one number, so we have to pick one — conservative wins. The original `trade.take_profit_2` is left intact in the returned dict for any caller that wants to read it directly. |
| Clear error mentioning BOTH candidate paths when `signal` missing (vs raising the existing "missing signal" error verbatim) | An operator triaging a failure needs to know whether to fix the prompt's top-level shape or the nested `trade` block; the legacy error didn't tell them. Naming both paths in the error message points the operator at the exact templates to inspect (`sample_prompt.md` / `simple_trend_analysis.md` for top-level, `chasulang_ict_smc.md` for nested) and turns a 30-minute log dive into a 30-second find-and-fix. |
| Rebuild `_execute_cli_once` on `subprocess.Popen` via `asyncio.to_thread` (vs sticking with `asyncio.create_subprocess_exec` and adding extra cleanup) | The pre-Phase-16.1 path was observed in prod (2026-04-28T15:02:15Z) to leave the child alive even after `asyncio.wait_for` raised `TimeoutError`. The async subprocess teardown is subtle — `process.kill()` on an already-zombied child can hang, `process.wait()` after a kill can deadlock if stdout buffer is full. The blocking `subprocess.Popen` API has decades of well-understood semantics; running it via `asyncio.to_thread` keeps the event loop free without inheriting the async-subprocess teardown gotchas. |
| Hard SIGKILL on timeout (vs soft `terminate()` first, then SIGKILL on holdout) | The prod wedge proves the soft-terminate path was insufficient at least once; a graceful-then-hard escalation costs an extra wait period during which the engine can't recover. For a per-attempt 240s leash already exhausted, the operator's preference is "stop wasting time, kill it now and retry" — SIGKILL immediately. Claude CLI is stateless from the engine's perspective; nothing graceful is gained by giving it a few extra seconds. |
| Bounded `proc.wait(timeout=5)` after `proc.kill()` (vs unbounded `proc.wait()`) | An unbounded wait on a child that's already failed to respond to SIGKILL is exactly the wedge we're trying to close. 5s is comfortably above any reasonable kernel-side reap time and bounds the worst-case wedge to 365s (240s timeout + 120s retry communicate + 5s SIGKILL reap) instead of 12+ hours. The new distinct `ClaudeTimeoutError` message ("SIGKILL within 5s") makes the failure mode visible in logs so operators can spot zombie / kernel-stuck children if they ever happen. |
| No ADR | Bug fix to an existing component. `ClaudeCLI` public contract is unchanged: same `analyze` signature, same `ClaudeError` / `ClaudeTimeoutError` / `ClaudeNotFoundError` / `ClaudeExecutionError` exception types, same retry semantics, same `attempt_number` propagation contract from Phase 14.1. ADRs are reserved for component-boundary or constraint decisions; this is mechanical reliability work. |

## Potential Risks

- **Live verification still owed** — the parser fix and the wedge
  mitigation are unit-tested but neither has been observed working
  on Fly. The operator needs to redeploy and check that (a)
  chasulang's responses now produce actual `Proposal` objects in
  the cycle log instead of every cycle ending with the
  `'signal'` `KeyError`, and (b) any subsequent timeout (if 240s ×
  1.5 = 360s isn't enough) terminates the child within 5s of the
  per-attempt timeout instead of wedging. Without the live
  verification, both fixes are unproven against the actual prod
  failure mode they target.
- **Composite scores topping out around 0.35 is still the binding
  constraint on actual trade execution** (Phase 15.1 carry, still
  standing) — even with Phase 16.1 unblocking chasulang's
  proposals, `auto_approve_threshold = 1.0` (default) means
  proposals get rejected at the gate. The operator action set
  `ENGINE_AUTO_APPROVE_THRESHOLD=0.30` via Fly secrets is still
  required for trades to actually execute. Phase 16.1 delivers
  proposals; the threshold setting is what turns proposals into
  trades.
- **TP2 information is silently dropped from the canonical view**
  — when chasulang returns both `take_profit_1` and `take_profit_2`,
  the top-level promoted view only carries `take_profit_1`. A
  future strategy that wants to honour the second target (e.g.
  scale-out at TP1, hold remainder for TP2) would have to read
  `result["trade"]["take_profit_2"]` directly. Acceptable for
  Phase 16.1's bounded scope; if a multi-target strategy
  materialises, the right next move is extending `AnalysisResult`
  with an optional `take_profit_targets: list[Decimal]` field.
- **The `subprocess.Popen` switch is a behaviour change for the
  retry path** — Phase 14.1's `attempt_number` propagation was
  designed against the `asyncio.create_subprocess_exec` path; the
  Popen rewrite preserves the contract end-to-end (locked by
  `test_subprocess_kill_on_timeout` asserting `attempt_number == 1`
  and the existing
  `TestClaudeCLIRetryOnTimeout::test_timeout_error_carries_final_attempt_number`
  asserting through 3 attempts), but a future refactor that
  changes the retry-loop's wrapper shape needs to keep the
  `attempt_number=attempt + 1` forwarding intact. Test coverage
  pins this; intentionally surfacing as a risk so future code
  archaeologists know the invariant.
