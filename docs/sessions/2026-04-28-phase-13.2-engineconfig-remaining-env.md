## Work Summary

Phase 13.2 closes DEBT-003 by promoting the four `EngineConfig`
fields that Phase 10.2 deferred — `monitor_interval_seconds`,
`bitcoin_symbol`, `altcoin_top_k`, `actor` — through `Settings` so
operators can tune them via environment variables instead of editing
source. The change is the third application of the Phase 10.2
pattern (10.2 first, 12.1 second, 13.2 third); the pattern is now
well-established and the cycle was accelerated as a confidence-high
case. No behavioural change in default deployments — every default
matches the pre-13.2 hardcoded `EngineConfig` value bytewise, so
existing deployments without the new env vars set are unaffected.

Four `Settings` fields added to `src/config.py` with the
`engine_*` env-var convention:
`engine_monitor_interval: int = Field(default=60, ge=10)` (env
`ENGINE_MONITOR_INTERVAL`),
`engine_bitcoin_symbol: str = Field(default="BTC/USDT")` (env
`ENGINE_BITCOIN_SYMBOL`),
`engine_altcoin_top_k: int = Field(default=3, ge=1)` (env
`ENGINE_ALTCOIN_TOP_K`),
`engine_actor: str = Field(default="auto-engine")` (env
`ENGINE_ACTOR`). The `ge=10` floor on `monitor_interval` and `ge=1`
floor on `altcoin_top_k` mirror `EngineConfig`'s own validators so
the env-side and config-side reject the same invalid values.
`build_engine` in `src/main.py` passes all four through to
`EngineConfig(...)` alongside the pre-existing four, and the
docstring was rewritten to drop the "not yet env-overridable" note.

`.env.example` and `docs/deployment.md` document the four new env
vars with operator-facing descriptions: monitor interval as
SL/TP poll cadence inside one cycle, bitcoin symbol as the per-cycle
Bitcoin scan target (distinct from `ENGINE_SYMBOLS`'s altcoin list),
altcoin-top-K as the composite-score-ranked truncation cap, and the
actor name as the activity-log identifier (only useful for operators
running multiple engines against the same data dir). `deployment.md`'s
"As of Phase 10.2 the four tunables below" sentence was updated to
"As of Phase 13.2 every EngineConfig field is env-driven through
Settings".

Tests extended both ends: `tests/test_config.py::TestEngineSettings`
got default-value + env-override assertions for every new field plus
validator tests for the two `ge=` floors (4 new test methods +
4 new assertions on the existing default-vs-EngineConfig parity
test). `tests/test_main_dispatch.py` got
`test_build_engine_propagates_all_engine_env_overrides` — a
single smoke test that sets all four env vars to non-default values
and verifies the resulting `engine.config` has them all set
(propagation is verified end-to-end through `build_engine`).

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Repeat the Phase 10.2 pattern verbatim — `engine_*` `Settings` fields, env-var named `ENGINE_*`, `build_engine` constructs `EngineConfig(...)` with all fields | Pattern is now applied three times (10.2, 12.1, 13.2). Consistency means operators don't have to learn a second convention; the field-name → env-var mapping is mechanical. DEBT-003 itself recommended this pattern. |
| Defaults bytewise-match `EngineConfig` (60, BTC/USDT, 3, auto-engine) | Existing deployments must be unaffected without an explicit env setting. The parity is locked by `test_settings_defaults_match_engine_config` which compares each `Settings` field against `EngineConfig()` directly. |
| `ge=10` and `ge=1` validators on `monitor_interval` and `altcoin_top_k` | Mirror `EngineConfig`'s own `Field(ge=...)` floors so env input gets the same validation as direct `EngineConfig(...)` construction; otherwise an env-set `ENGINE_ALTCOIN_TOP_K=0` would slip through `Settings` and fail downstream with a less-specific error. |
| Single end-to-end smoke test in `test_main_dispatch.py` instead of one per field | The per-field defaults + env overrides are already covered in `test_config.py`. The dispatch test only needs to verify the propagation wiring works — one test setting all four to non-default values is sufficient. Mirrors the Phase 10.2 / 12.1 dispatch-test shape. |
| `engine_actor`: no validator | The actor field is a freeform identifier; any non-empty string is operationally valid. No `min_length` constraint added since a deliberately-empty actor (signalling "anonymous") is conceivable and the activity log handles it. |
| `bitcoin_symbol`: no validator | Same reasoning. Symbol format depends on the exchange and is validated downstream when the proposal hits the exchange adapter. |

## Potential Risks

- **Default-parity drift over time** — the parity is locked by
  `test_settings_defaults_match_engine_config`. If a future cycle
  changes one side's default without the other, that test fails
  red. Acceptable; the test is the canonical guard.
- **Env-var sprawl** — `Settings` now exposes 8 `engine_*` env vars
  (4 from 10.2/12.1, 4 from 13.2). The operator surface is wider
  than pre-10.2; `.env.example` and `docs/deployment.md` document
  every one. Worth a periodic review if the count grows further,
  but at 8 it's still tractable.
- **`engine_actor` as a freeform string** — no `min_length`
  validator means an env-set empty string would propagate. The
  activity log handles it (no crash), but a deliberate empty actor
  could surprise an operator inspecting the log. Acceptable; the
  default `"auto-engine"` is a reasonable fallback and operators
  who set this know what they're doing.
