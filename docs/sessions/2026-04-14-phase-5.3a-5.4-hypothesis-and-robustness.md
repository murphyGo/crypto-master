## Overview

- **Date**: 2026-04-14
- **Phase**: 5 - Feedback Loop System
- **Sub-tasks**:
  - 5.3a — Strategy Improver prompt redesign (hypothesis-driven)
  - 5.4 — Robustness Validation Gate (new, inserted before the
    automated feedback loop)
- **Plan change**: The previously planned 5.4 ("Automated Feedback
  Loop") was renumbered to **5.5**. A new 5.4 was inserted because
  the loop is only safe once a robustness gate exists to decide
  what gets promoted.

## Motivation

A discussion about the project's chances of producing a profitable
live system surfaced two structural risks that the prior plan did
not address:

1. **`StrategyImprover` was an overfitting machine by default.** The
   improvement prompt told Claude to "improve win rate," and the
   new-idea prompt suggested combining "RSI, MACD, volume, etc." —
   the exact failure mode quants warn about. LLMs are extremely good
   at fitting noise to past data when asked to optimize a backward-
   looking metric, and indicator-combination strategies have no
   structural edge after fees.
2. **Backtest pass ≠ promotion-worthy.** The plan jumped straight
   from "backtest" to "feedback loop" with no gate between them.
   Without OOS / walk-forward / regime / sensitivity checks, the
   loop would have promoted whichever Claude-generated strategy got
   lucky on a single full-history run.

Both fixes are P0 from the architecture-review discussion captured
this session.

## Work Summary

### 5.3a — Hypothesis-Driven Prompt Redesign

`src/ai/improver.py` rewritten on three axes:

1. **Output format** now requires a `hypothesis:` field in YAML
   frontmatter — a single falsifiable sentence describing the market
   inefficiency the technique exploits. The frontmatter requirement
   itself is the main behavioral change: it forces Claude to commit
   to *why* the strategy should work, which then becomes the anchor
   for the robustness gate's "does this thesis actually hold OOS?"
   question.
2. **Improvement prompt** (`_build_improvement_prompt`) replaces
   "improve win rate" with a mandatory **Failure Analysis** section:
   identify 2–3 structural failure modes from losing trades, state
   the root cause, propose ONE targeted change per cause. Hard cap:
   no more than 2 new conditions added per revision. Explicit
   anti-overfit guardrails: no per-trade thresholds, no calendar-
   based filters tuned to the trades shown.
3. **New-idea prompt** (`_build_new_idea_prompt`) replaces "any
   standard indicators (RSI, MACD, volume, etc.)" with **explicit
   refusal** of generic indicator mashups, plus a curated list of
   the *kind* of market-structure hypothesis that is acceptable
   (funding extremes, liquidation cascades, OI divergence, cross-
   exchange basis, stablecoin supply changes). Includes a `## Data
   Requirements` section requirement so the system knows when a
   proposed strategy needs data we may not have.
4. **User-idea prompt** (`_build_user_idea_prompt`) gained a
   "extract the hypothesis" step and a `## Caveats` section if the
   user's idea has no plausible underlying edge — i.e. honestly
   produce the technique but flag the concern rather than silently
   "fixing" the idea.

`GeneratedTechnique` model gained a `hypothesis: str` field; the
parser populates it from frontmatter (defaults to empty for legacy
techniques without the field).

### 5.4 — Robustness Validation Gate

`src/backtest/validator.py` (new) implements the gate as four
independent checks plus an aggregate report:

| Gate | What it catches |
|------|-----------------|
| OOS | Strategy was secretly fit to the full history |
| Walk-forward | Strategy works in only one slice of time |
| Regime | Strategy works in only one market state |
| Sensitivity | Strategy lives on a single lucky parameter value |

Each gate returns a `GateResult` with `status` (PASSED / FAILED /
SKIPPED), `score`, `threshold`, `reason`, and gate-specific
`details`. SKIPPED is a first-class outcome — it means "you didn't
give me enough data/inputs to check this." SKIPPED gates are
neutral in the aggregate verdict but surfaced in the report so an
operator can't mistake "untested" for "validated."

Defaults are conservative on purpose:
- OOS Sharpe must be ≥ 70% of IS Sharpe.
- ≥ 60% of evaluable walk-forward windows must be profitable.
- Every regime with enough trades must have non-negative expectancy.
- Mean Sharpe across the parameter grid must be ≥ 50% of baseline
  AND ≥ 60% of grid points must be individually profitable.
- Sensitivity grid is hard-capped at 64 combos to prevent accidental
  exponential blowups; exceeding the cap FAILS the gate fast rather
  than silently truncating.

Sensitivity is OPT-IN via `strategy_factory` + `param_grid` — most
prompt-based strategies have no tunable numeric parameters, so the
gate cleanly SKIPs rather than forcing callers to invent a fake
grid.

## Key Decisions

1. **SKIPPED is neutral, not a soft pass.** A gate that did not run
   contributes nothing to the verdict but is loud in the summary
   string. Alternative was treating SKIPPED as PASSED, which would
   silently accept untested strategies — exactly the failure mode
   the gate exists to prevent.
2. **Sensitivity gate is opt-in.** A `param_grid` is the kind of
   input only a code-based, parameterized strategy has. Forcing
   prompt-based techniques to fabricate one would produce noise.
   Better to be explicit: "no grid → SKIPPED" with a clear reason.
3. **Hypothesis field is optional on the model, mandatory in the
   prompt.** Pre-existing techniques don't have a hypothesis field
   in their frontmatter; failing to parse them would break the
   loader. New Claude-generated techniques are *strongly steered* to
   include one via the prompt boilerplate, but the model layer
   tolerates absence with an empty string.
4. **Improvement prompt caps new conditions at 2.** The most common
   form of LLM overfitting on trade data is "stack five filters
   that would have avoided every losing trade in the sample." The
   cap is an explicit countermeasure; a numerical limit is harder
   for the model to argue around than vague "be careful."
5. **Phase 5.4 (gate) sequenced before Phase 5.5 (loop).** A
   feedback loop without a promotion gate is worse than no loop —
   it institutionalizes overfitting at scale. The gate has to exist
   before any orchestrator can be safely written.

## Known Limitations & Follow-ups

- **Walk-forward is rolling-OOS, not parameter-refit.** Our
  strategies are static (prompts/rules), so the textbook "refit on
  each window" version doesn't apply directly. If we add genuinely
  parametric strategies later, the gate should grow a `refit_fn`
  hook.
- **Regime classification is SMA-only.** ADX-based trending/ranging
  classification, or volatility-regime tagging, would catch
  additional failure modes. Worth adding once we have any strategy
  that materially differs by regime.
- **No deflated Sharpe / multiple-testing correction yet.** When the
  feedback loop starts generating dozens of candidates, we need a
  meta-gate that accounts for the fact that the best of N random
  strategies looks great by chance. This belongs in Phase 5.5 or
  a new Phase 5.6.
- **No persistence for `RobustnessReport`.** Reports should land
  under `data/backtest/{run_id}/robustness.json` so the loop has an
  audit trail. Deferred to the loop integration in 5.5.
