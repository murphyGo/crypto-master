## Work Summary

Phase 21.3 closes the last surface of DEBT-025 with the narrowest
cycle of the three: verification + type-tightening, no behaviour
change. The 21.2 sweep had already routed the engine-side
`datetime.now()` call inside `_record_stale_quote_rejection` to
`now_utc()`, and the candle timestamp pulled from the ticker is
already UTC-aware via 21.1's `from_unix_ms` adapter swap. What
21.3 adds on top of that is the *contract* — a formal "Timestamp
coherence contract" section appended to the function's docstring
that names every timestamp source flowing into the rejection
payload, declares each one's UTC-aware tzinfo, and points at the
21.1 / 21.2 / 21.3 history that established it. The function
body below the new docstring section is byte-identical; the
contract is metadata, not logic. The point is to make the
guarantee legible at the call-site so a future contributor adding
a new timestamp into the payload sees the contract in the
docstring and routes through `now_utc()` / `from_unix_ms` / the
existing `ensure_utc(...)` path rather than reintroducing a naive
`datetime.now()`.

The verification half of the cycle is three new regression tests
in `tests/test_runtime_engine.py`, each pinning one corner of the
contract that was implicit in 21.1+21.2's coverage but never
asserted directly at the rejection-payload boundary.
`test_stale_quote_rejection_payload_timestamps_are_utc_aware`
(line 992) is the headline: under `time_machine.travel(...,
tz_offset=9)` (KST), trigger the rejection path and assert every
timestamp field on the resulting event carries `tzinfo=UTC`.
`test_stale_quote_rejection_decision_at_minus_candle_ts_is_aware_math`
(line 1033) goes one step deeper — the rejection event records a
`drift` value computed as `decision_at - candle_ts`; this test
asserts the subtraction succeeds (no `TypeError` from
naive-vs-aware mixing) and the resulting `timedelta` is the
correct UTC-anchored value. `test_stale_quote_rejection_tolerates_legacy_naive_record_on_disk`
(line 1082) covers the edge case 21.2's reader-boundary shims
introduced: a pre-21.2 on-disk rejection record (legacy-naive
timestamps) flowing back into the engine on read should be
tolerated by the same `ensure_utc(...)` shim that protects the
five other reader boundaries — locks the contract that the
rejection payload's *read* path also self-heals to UTC.

## Key Decisions

| Decision | Rationale |
|---|---|
| Land 21.3 as docstring contract + 3 regression tests, no body change | The 21.2 sweep already routed `datetime.now()` → `now_utc()` at every site, including the one inside `_record_stale_quote_rejection`. 21.3's spec framed the work as "verification + type-tightening, not new behaviour" and the 21.2 follow-up confirmed there was no body change left to make. ## Potential Risks

- **Docstring contract is only as load-bearing as future
  contributors honour it.** The "Timestamp coherence contract"
  section is metadata, not enforcement — a future contributor
  adding a new timestamp into the rejection payload could
  technically reintroduce a naive `datetime.now()` and the three
  regression tests would catch it (they assert tzinfo on every
  field of the resulting event, not just the four current
  fields, via dict iteration), but only if the new field lands
  in the event payload rather than being computed off to the
  side. Defence-in-depth via the Pydantic `field_validator(mode=
  "after")` UTC-coerce hooks added in 21.2 covers the model
  boundary regardless. Tolerable as a residual risk.

- **Three tests cover three corners; a fourth corner — JSON
  round-trip via on-disk serialisation — relies on 21.2's
  Pydantic-validator coverage rather than a 21.3-specific test.**
  This is intentional (the 21.2 validators are the
  defence-in-depth at the model boundary) but worth flagging:
  the 21.3 read-side test `test_stale_quote_rejection_tolerates_
  legacy_naive_record_on_disk` exercises the reader-boundary
  shim, not the validator. If a future change drops the validator
  coverage on the rejection-record model, the 21.3 read-side
  test would still pass against the shim path but the model-
  construction path would silently go un-coerced. Cross-
  referenced in the 21.2 session log; not a 21.3-specific gap.
