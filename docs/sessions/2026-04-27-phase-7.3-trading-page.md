## Work Summary

Added the second content page to the dashboard. `Trading` shows, for
the selected mode (paper or live):

- **Summary cards** — open position count, closed trade count, win
  rate, realized P&L. A caption underneath surfaces the latest
  recorded equity / unrealized P&L and snapshot timestamp.
- **Active positions** (FR-029) — open trades sorted newest first.
- **Recent trade history** — last 25 trades, closed + open mixed,
  sorted by close-or-entry time descending.
- **Equity curve** (FR-031) — line chart of `total_equity` vs.
  timestamp from `PortfolioTracker.get_equity_curve`.

The page is purely a reader: `TradeHistoryTracker.load_trades(mode)`
and `PortfolioTracker.load_snapshots(mode)` produce all of its data.
No exchange calls, no recomputation of unrealized P&L from live
prices — that's the trader components' job, and the dashboard reflects
their last persisted state.

Four pure helpers (`build_open_positions_dataframe`,
`build_trade_history_dataframe`, `build_equity_curve_dataframe`,
`build_summary_metrics`) keep the data-shaping logic out of the
Streamlit runtime so unit tests don't need `AppTest` for those paths.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| `url_path="strategies"` / `url_path="trading"` on each `st.Page` | Streamlit derives URL paths from the callable name by default. Both page modules expose a function called `render`, which collided with `StreamlitAPIException("Multiple Pages specified with URL pathname render")`. Explicit `url_path` is the documented escape hatch and keeps the helper-function name consistent across pages. |
| Read-only page — no exchange calls | The dashboard runs in the Streamlit process, which doesn't own exchange credentials in this codebase. Pulling fresh prices for unrealized-P&L re-computation would require a connected exchange and async glue we don't want to bolt onto the chassis. The persisted snapshots are accurate enough for the operator-facing summary, with a caption flagging the snapshot's timestamp. |
| Mode is a runtime `st.radio`, not two separate pages | Operators flip between paper and live constantly during validation; one page with a top-of-page toggle is fewer clicks than two pages at the cost of one extra widget per render. Mirrors how `LiveTrader` / `PaperTrader` are picked at the trader level. |
| Trade history sorted by `exit_time or entry_time` | A closed trade's "interesting moment" is when it closed; an open trade's is when it opened. Same definition as how a brokerage account statement orders trades. |
| 25-trade default history limit | Visual: anything bigger needs scrolling, anything smaller cuts off the user's recent activity. Caller can override via the helper's `limit` argument when reused elsewhere (e.g. an export view). |
| Empty DataFrames returned with declared columns | Lets the Streamlit `st.dataframe(...)` call render an empty header row without an extra branch in the render function, and lets tests assert on column presence without populating data. |

## Potential Risks

- **Stale unrealized-P&L numbers.** The summary card shows the latest
  *recorded* unrealized P&L, which can lag live prices badly during
  long quiet periods between snapshots. Mitigation: the timestamp is
  printed alongside the number so the operator sees how fresh it is.
  A future iteration could either (a) wire a "refresh from exchange"
  button, or (b) auto-trigger a snapshot on page load.
- **No mode-mix view.** Operators wanting backtest data mixed with
  paper/live data have to switch between modes manually. Out of scope
  for FR-029 / FR-031, which both mention "paper/live"; revisit if
  the backtest dashboard becomes its own page or if cross-mode
  reporting is requested.
- **No caching.** Every nav click re-reads the trade history JSON.
  Same risk as 7.2 — fine at current scale, candidate for
  `@st.cache_data(ttl=...)` later.
