## Work Summary

Closed the single-strategy lockout that Cycle 1's runtime verification exposed:
`ProposalEngine._select_best_technique` returned exactly one strategy per
symbol per cycle with an alphabetic-by-name tiebreaker in cold-start. On the
live Fly deployment this meant only `bollinger_band_reversion` ever ran —
every other strategy (`rsi`, `ma_crossover`, `chasulang_ict_smc`,
`simple_trend_analysis`, `sample_prompt`) was loaded but never analysed a
candle. Bollinger reversion has a low-base-rate signal (price piercing the
bands), so most cycles produced zero proposals and the threshold gate never
fired. The Phase 9.2 stated goal of "side-by-side LLM-vs-deterministic
comparison + degraded-mode safety net" was structurally broken by this
single-selection design.

`ProposalEngine` now iterates every applicable technique per symbol; per-
symbol dedup keeps the highest-composite winner; the legacy
`multi_technique_per_symbol=False` opt-out preserves bit-for-bit behaviour for
op-emergency rollback. Approach (a) was chosen: a sibling
`_select_all_techniques`, a sibling `_propose_all_for_symbol`, a
`_dedup_by_symbol` helper, and `_build_proposal_for_strategy` factored from
`_propose_for_symbol`. The legacy path's `_select_best_technique` is kept as
live code (not retired) so a config flag flip is enough to roll back if the
multi-technique path ever produces a runtime surprise.

The cycle's process highlight is worth recording. Quant ran in the design
phase before dev, which caught two 🔴 blockers (per-symbol dedup, long+short
conflict) that the original spec had missed. Without dedup, the runtime
engine would have called `trader.open_position` once per technique per symbol
per cycle, opening N positions on the same pair at N× the intended
`risk_percent` — a real-money defect. The spec was patched, dev implemented
against the corrected spec, and quant's post-review found zero correctness
issues. For trading-domain sub-tasks, design-phase quant review is high-
leverage. Cycle 1's diagnosis was wrong (threshold lockout) but the runtime
verification corrected it before any fix code was written — the lesson is to
read the actual logs before declaring a (b)/(c) verdict.

## Potential Risks

- **Per-technique OHLCV re-fetch** — `_propose_all_for_symbol` re-fetches OHLCV per technique, producing N×M `get_ohlcv` calls per symbol per cycle (vs 1×M previously). Not a correctness issue, but two operability concerns: (1) temporal-consistency drift if a candle rolls mid-cycle (technique A sees candle T while B sees T+δ); (2) API rate-limit pressure at scale. Current envelope (5 symbols × 5 strategies × 4 timeframes = 100 calls/cycle) is fine for single-machine deployments. Recorded as DEBT-002 with a recommended fix: cache OHLCV per `(symbol, timeframe)` for the duration of one `propose_*` call.
- **Legacy `_select_best_technique` kept as live code** — chosen for op-emergency rollback. The trade-off is dead-code drift: the function will rot if no one exercises the `multi_technique_per_symbol=False` path. The legacy-path test guarantees the rollback behaviour stays callable; the function will be retired in a future cycle once the multi-technique path has accumulated production miles.
- **Design-phase quant review is now a soft expectation for trading-domain sub-tasks** — this cycle's process win sets the bar. Without it, two 🔴 blockers would have shipped to dev. There is no enforcement mechanism today; the lesson is in the cycle's outcome, not in tooling.
