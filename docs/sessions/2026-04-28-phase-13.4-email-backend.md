## Work Summary

Phase 13.4 ships the third push backend behind Slack (Phase 11.3)
and Telegram (Phase 12.4). `EmailNotifier` lives in
`src/proposal/notification.py` and uses the stdlib `smtplib.SMTP` +
`email.message.EmailMessage` pair wrapped in `asyncio.to_thread` so
the dispatcher's event loop is not blocked while the SMTP handshake
runs. Zero new dependency — the same shape Slack and Telegram
already follow. STARTTLS (port 587) is the default; SMTP_SSL (port
465) is deliberately deferred (recorded as DEBT-012, Low).

The subject is the operator-facing one-liner per spec —
`"Crypto Master: {symbol} {side} score={c:.2f}"` — short enough to
render cleanly in mail-client preview panes. The body reuses
`_build_telegram_text` via a thin `_build_email_body` helper so all
three push backends carry identical content; `test_build_email_body_matches_telegram_text`
locks the invariant. The Markdown bold + triple-backtick fence
renders in Markdown-aware clients (Apple Mail, modern Gmail) and
degrades to readable plain text everywhere else.

Settings exposes 6 SMTP fields — `email_smtp_host`,
`email_smtp_port` (default 587), `email_smtp_user`,
`email_smtp_password`, `email_from`, `email_to`. Five are gated as
strings for activation; `email_smtp_port` has a non-optional default
so only the five string fields decide whether the notifier is
constructed (the dev-plan text "all 6 required" is loose vs the
code's 5-string gate — QA flagged this, code is correct because the
port default cannot fail the `all([...])` test). `EmailNotifier.__repr__`
masks the password unconditionally; host / user / from / to remain
visible because they're operationally useful for log triage. `send`
deliberately does NOT swallow `smtplib` errors — the dispatcher's
existing per-channel failure-isolation contract from Phase 6.3 owns
that responsibility, identical to Slack and Telegram.

`build_engine` appends `EmailNotifier(...)` to the dispatcher's
notifier list when all 5 string fields are set. Activation log line
states presence only; password is never logged. Test coverage adds
11 (1138 → 1149): 9 in `tests/test_proposal_notification.py`
(`test_build_email_subject_matches_spec`,
`test_build_email_body_matches_telegram_text`,

`_FakeSMTP`, `test_email_notifier_repr_masks_password`,

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Stdlib `smtplib` + `EmailMessage` (zero new dep) | Matches Slack (Phase 11.3) and Telegram (Phase 12.4) — all three push backends share the "no new dep, `asyncio.to_thread` blocking I/O" pattern. Adding `aiosmtplib` would be the only "async-native" alternative and isn't justified for one notifier. |
| STARTTLS only — defer SMTP_SSL alternative | Port 587 STARTTLS covers the major operator providers (Gmail, SendGrid, Mailgun, Office 365). Port 465 SMTP_SSL is a real path for some legacy / privacy providers (Yahoo, ATT, ProtonMail) but adds a config branch and a second handshake variant. Recorded as DEBT-012 (Low) — pick up when an operator request lands. |
| Body reuses `_build_telegram_text` verbatim | Spec wants "same content across all push backends"; one shared payload builder makes that mechanical instead of by-hand. Telegram's Markdown bold + code-fenced detail renders fine in Markdown-aware mail clients and degrades to readable plain text elsewhere. |
| 5-string activation gate vs 6-field "all required" | `email_smtp_port` has a default of 587, so it cannot fail an `all([...])` test — gating on it would be dead code. The five string fields (host/user/password/from/to) are the real "is the operator configured for email?" signal. QA-flagged the dev-plan/text "all 6" mismatch; code is correct. |
| `__repr__` masks password only (not host/user/from/to) | Tighter contract than Slack's URL-only redaction would be over-engineering — host / user / from / to are operationally useful for log triage and not secrets in the same sense. The password unconditionally must be hidden. |
| `send` does not catch `smtplib` errors | Phase 6.3's per-channel failure-isolation contract is the single owner of "one bad backend can't silence the others". Catching internally would either swallow the error (test-defeating) or duplicate the dispatcher's logic. Verified by `test_email_smtp_failure_does_not_crash_dispatch` — a raising `_FakeSMTP` does not stop a downstream good notifier. |
| Configurable `timeout: float = 10.0` | A slow SMTP server can't stall the cycle. Test `test_email_notifier_uses_configured_timeout` pins that the value reaches `smtplib.SMTP(timeout=...)`. |

## Potential Risks

- **STARTTLS-only blocks SMTP_SSL providers** — operators on port
  465 (Yahoo, ATT, ProtonMail business plans, some privacy
  providers) currently can't enable email push. Recorded as
  DEBT-012 (Low). Workaround until then: use a relay (Mailgun,
  SendGrid, Postmark) on port 587. Most modern providers offer
  STARTTLS as the recommended path anyway.
- **Body parity invariant is by-hand** — `_build_email_body`
  delegates to `_build_telegram_text` today, but if a future
  Telegram-specific tweak (e.g. an emoji or a Telegram-only deeplink)
  lands, email would silently inherit it. Phase 12 cross-check
  flagged the same risk for Telegram→Slack. No mechanism enforces
  the in-sync invariant; the body-parity test (`test_build_email_body_matches_telegram_text`)
  catches drift but only on the email side.
- **Password is in process memory** — `EmailNotifier._password` is
  a plain `str` field. Operators with very strict secret-handling
  requirements would want a secret-vault integration; that's beyond
  current scope (Slack webhook URL and Telegram bot token both have
  the same characteristic).
