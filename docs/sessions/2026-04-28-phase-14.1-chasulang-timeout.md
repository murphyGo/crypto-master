## Work Summary

Phase 14.1 addresses a prod-observed reliability gap: `chasulang_ict_smc`
kept timing out on every BTC scan cycle on Fly even after Phase 12.3's
retry-with-1.5×-backoff shipped (default `max_retries=1`, base 120s →
180s on retry). Reading recent Fly logs confirmed the retry path was
firing — the `retrying with timeout=180s` Phase 12.3 warning was
present — but both attempts kept timing out, meaning 180s still wasn't
enough for chasulang's prompt-heavy multi-TF ICT/SMC payload on Fly's
shared-CPU / 1 GB RAM machine. Bumping `CLAUDE_CLI_TIMEOUT_SECONDS`
globally would slow every other strategy (`rsi_4h.analyze` finishes
comfortably under 30s); the right fix is a per-strategy override so
chasulang gets a longer leash without taxing the baselines.

Implementation is a four-touch wedge with strict back-compat: (1)
`TechniqueInfo` gains an optional `claude_timeout_seconds: int | None
= Field(default=None, ge=1)` field — `None` keeps existing strategies
on the global `Settings.claude_cli_timeout_seconds` default, integer
overrides go straight to `ClaudeCLI`. (2) `PromptStrategy.analyze`
reads `self.info.claude_timeout_seconds` and instantiates
`ClaudeCLI(timeout=float(override))` when set, otherwise plain
`ClaudeCLI()` so the wrapper's `__init__` resolves Settings as before.
(3) `strategies/chasulang_ict_smc.md` frontmatter gains
`claude_timeout_seconds: 240` — picked so 240 × 1.5 = 360s total worst
case with one retry, comfortably above the observed timeout floor.
(4) The `LLM_TIMEOUT` activity event payload now carries
`attempt_number` (1-indexed attempt that raised) and
`final_timeout_seconds` (alias of `timeout_seconds`, named for clarity
in the dashboard) so operators can distinguish "first attempt failed,
retry path didn't fire" from "every attempt timed out — bump the
per-strategy override". The original `timeout_seconds` key is preserved
for back-compat with downstream readers.

`ClaudeTimeoutError` itself grew an `attempt_number: int = 1` parameter
on `__init__`; `_execute_cli_once` accepts an `attempt_number` keyword
and stamps it onto the raised error, while the retry loop in `analyze`
forwards `attempt + 1` so the error reaching the caller carries the
final attempt's index. Default `attempt_number=1` keeps Phase 12.3's
single-shot semantics intact for any caller that hasn't migrated. No
ADR — extends the existing per-strategy-config pattern with one
override field; nothing about component boundaries changes.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Per-strategy `claude_timeout_seconds` override on `TechniqueInfo` | Bumping `CLAUDE_CLI_TIMEOUT_SECONDS` globally would slow every other strategy (`rsi_4h.analyze` finishes well under 30s) and burn Claude credit on baselines that don't need the headroom. Localising the longer leash to the one strategy that needs it keeps the global default tight. |
| 240s base override (vs 180s, 300s, etc.) | 240 × 1.5 = 360s worst case with one retry — fits comfortably above the observed timeout floor on Fly's shared-CPU/1GB machine. 180s was already proven insufficient (Phase 12.3 retry was firing and still timing out per Fly logs); 300s would tie up the Fly worker too long on a single cycle (5-min monitor interval). |
| `claude_timeout_seconds: int \| None = None` (default `None` falls back to Settings) | Strict back-compat — every existing strategy file is unchanged, every existing test is unchanged, and the override is opt-in per strategy. The `ge=1` validator rejects zero at load time as a config bug rather than silently producing a never-succeeding strategy. |
| `attempt_number` stamped on `ClaudeTimeoutError` (default 1) | Operators triaging Fly logs need to distinguish "first attempt fails, retry path didn't fire" (a wiring bug) from "every attempt timed out" (a leash-too-short problem). The default of 1 preserves Phase 12.3's single-shot semantics for any caller that hasn't migrated. |
| `final_timeout_seconds` alias alongside `timeout_seconds` in the event payload | The dashboard / future log readers should reach for `final_timeout_seconds` (intent-revealing) rather than `timeout_seconds` (ambiguous between "first attempt's timeout" and "final attempt's timeout"). Keeping the legacy key avoids breaking existing readers — additive change only. |
| No ADR | Extends an existing pattern (per-strategy frontmatter config) with one new field. Nothing about component boundaries changes; the `BaseStrategy` / `ClaudeCLI` boundary is unchanged. |

## Potential Risks

- **240s × 1.5 = 360s worst case ties up the Fly worker for a full
  cycle on the failing path** — the engine's monitor interval is 60s
  (default), so a chasulang cycle that hits both attempts and times
  out at 360s blocks 6 monitor ticks. If chasulang continues to fail
  even at 240s, the right next move is either prompt slimming or
  removing chasulang from the BTC scan path entirely; not bumping the
  override further. Worth re-checking Fly logs after the redeploy
  before declaring the issue closed.
- **`final_timeout_seconds` is an alias, not a replacement** —
  existing dashboards / log greps that read `timeout_seconds` keep
  working, but a future cleanup that drops the legacy key would
  silently break them. Not removing the legacy key is the deliberate
  back-compat call; flagged here so a future cycle doesn't decide to
  "tidy up" the duplicate field.
- **Prompt length is the unaddressed root cause** — a per-strategy
  longer leash treats the symptom, not the cause. If chasulang's
  multi-TF ICT/SMC prompt could be slimmed (fewer candle rows, tighter
  instruction wording), the timeout would shrink correspondingly and
  the override could come back down. Out of scope for 14.1; flagged
  for follow-up if 240s proves insufficient.
