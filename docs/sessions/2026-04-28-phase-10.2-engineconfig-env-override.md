# Session Log: 2026-04-28 - Phase 10.2 - EngineConfig Env Override

## Overview

- **Date**: 2026-04-28
- **Phase**: 10 - Operational Maturation
- **Sub-task**: 10.2 - EngineConfig Env Override

## Work Summary

Phase 10.2 closes the operability gap that Phase 8's cross-check carried as a
documented small follow-up: `EngineConfig` (cycle interval, auto-approve
threshold, symbol list, balance) was built from literals in `src/main.py`, so
changing any value required a code edit + redeploy. Settings now drives every
operationally-impactful `EngineConfig` field via env vars. Defaults are
bytewise-equal to the pre-10.2 hardcoded values, so existing deployments
behave identically without any explicit env setting — a redeploy is a no-op
unless the operator opts in by setting one of the new vars.

`Settings.engine_*` fields were added to `src/config.py`
(`engine_cycle_interval`, `engine_auto_approve_threshold`, `engine_symbols`,
`engine_balance`) and `build_engine` in `src/main.py` now constructs
`EngineConfig` from those fields rather than literals. The non-obvious piece
is `engine_symbols`: env vars carry strings, and pydantic-settings normally
JSON-decodes complex types. To accept the natural comma-separated form
(`ENGINE_SYMBOLS=BTCUSDT,ETHUSDT`), the field uses
`Annotated[list[str], NoDecode]` plus a `field_validator(mode="before")` that
splits on commas, trims whitespace, and rejects empty input. This trades a
JSON-array env literal for the operational form operators actually want.

`build_engine` continues to accept an explicit `config: EngineConfig | None`
override — explicit-config-wins back-compat is preserved. The four shipped
fields are the operationally-impactful ones; four other `EngineConfig`
fields (`monitor_interval_seconds`, `bitcoin_symbol`, `altcoin_top_k`,
`actor`) remain hardcoded. Those are tunables operators rarely need to
change, so they're recorded as DEBT-003 (Low) for follow-up if needed
rather than treated as part of this cycle's scope.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| `engine_symbols` uses `Annotated[list[str], NoDecode]` + `field_validator(mode="before")` for comma-separated env parsing | pydantic-settings defaults to JSON-decoding complex types, which would force operators to write `ENGINE_SYMBOLS=["BTCUSDT","ETHUSDT"]` in the env — awkward and error-prone. NoDecode + a before-validator gives the natural `BTCUSDT,ETHUSDT` form with trim/empty handling. The annotation choice is documented inline. |
| Defaults bytewise-equal to pre-10.2 `EngineConfig` literals | A redeploy without any env setting must produce identical behaviour. Operators opt in by setting a var; existing deployments don't change behaviour. QA verified the bytewise equality (cycle 300 / threshold 1.0 / 5-symbol list / Decimal 10000). |
| `build_engine` preserves the explicit `config: EngineConfig \| None` parameter | Tests and any other in-process caller can still inject a fully-constructed `EngineConfig`. The new env-driven path only fires when no explicit config is passed. |
| Ship only the 4 operationally-impactful fields; defer `monitor_interval_seconds`, `bitcoin_symbol`, `altcoin_top_k`, `actor` | These are tunables operators rarely need to change. Including them inflates the `Settings` surface for marginal gain. Recorded as DEBT-003 (Low) follow-up rather than active priority. |
| No ADR needed | Settings + Pydantic env-override is a well-established pattern in the project — `PerformanceTracker` / `TradeHistoryTracker` / Phase 10.5's data_dir routing all use it. No new design ground broken. |

## Potential Risks

- **Four `EngineConfig` fields remain hardcoded** —
  `monitor_interval_seconds`, `bitcoin_symbol`, `altcoin_top_k`, `actor` are
  not env-overridable. Recorded as DEBT-003 (Low). The judgement is that
  these are tunables operators rarely change; promoting them to env vars
  would inflate `Settings` for marginal gain. The risk is that an operator
  hits one of these four needs in production and has to do a code edit +
  redeploy — the same pre-10.2 friction this sub-task partially closed.
  Acceptable given the four shipped today are the operationally-impactful
  ones.
- **`engine_symbols` comma-separated convention is the project's first** —
  pydantic-settings' default is JSON. The NoDecode + validator pattern works
  but is slightly non-standard; future devs unfamiliar with the project may
  reach for JSON env literals. Mitigation: the docstring on the validator
  explains the choice; `.env.example` shows the canonical form.
