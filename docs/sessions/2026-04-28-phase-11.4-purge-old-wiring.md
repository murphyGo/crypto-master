## Work Summary

Phase 11.4 closes the "method exists but nothing calls it" loop on
`ProposalHistory.purge_old` carried forward from Phase 10.4. The
purge logic shipped in 10.4 with full test coverage but no runtime
path invoked it ‚Äî long-running Fly deploys would accumulate proposal
records indefinitely until something called it. The fix is two
complementary entry points: a startup hook in `src/main.py::run`
that runs once per process boot, and an operator CLI
(`python -m src.tools.purge_proposals`) for ad-hoc windows that
differ from the configured retention.

The non-obvious choices were the helper boundary, the empty-result
log policy, and the dual-view `ProposalHistory` instance. Helper:
dev extracted `_purge_old_proposals(history, retention_months)` as a
sibling of `build_engine` so the startup behaviour is testable in
isolation without driving the whole `run()` coroutine. Empty-result:
the helper logs INFO only when records were actually archived and
stays silent on the "0 records" path ‚Äî long-running deploys purge
once early and then have nothing to do for the rest of the retention
window, so logging on the empty path would be noise on every daily
restart. Dual-view: `run()` constructs a fresh `ProposalHistory()`
rather than reusing `engine.proposal_history`. Both default to the
same on-disk dir (`Settings.data_dir / "proposals"` per Phase 10.5),
so the semantics are correct, but it does mean two `ProposalHistory`
objects briefly coexist. QA flagged this as üü° informational (not
blocking) ‚Äî dev's docstring acknowledges the dual-view choice
deliberately.

The CLI follows the same shape as `scripts/backtest_baselines.py`'s
operator-tooling pattern but lives under `src/tools/` rather than
`scripts/` because it imports only project code (no external network
dependencies) and benefits from the same packaging / typing as the
runtime. `main(argv)` accepts an explicit `argv` for tests; the
`__main__` guard wraps `SystemExit(main())` for command-line use.
Output is informative on both branches ‚Äî "Purged N proposal record(s)
older than X months." or "No proposal records older than X months;
nothing to purge." Always exits 0; "nothing to purge" is a valid
steady-state outcome, not an error.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Extract `_purge_old_proposals` as a sibling helper rather than inlining in `run()` | Testable in isolation without driving the whole `run()` coroutine. The 4 tests in `TestPurgeOldProposalsHook` exercise forwarding / count / log policy against a `MagicMock` plus an end-to-end smoke against a real `ProposalHistory` rooted at `tmp_path`. Inlining would have forced async-coroutine harnessing for trivial assertions. |
| INFO log only when records were archived | Long-running deploys purge once early then have nothing to do for the rest of the retention window. Logging on the empty path would emit a noisy "0 records" line on every daily restart. The pytest `caplog`-via-`crypto_master.main`-handler test (same pattern as `test_ai_claude.py` because `get_logger` disables propagation) pins the silent-on-empty branch. |
| Construct a fresh `ProposalHistory()` in `run()` rather than reusing `engine.proposal_history` | Both default to the same on-disk dir (`Settings.data_dir / "proposals"` per Phase 10.5) so the semantics agree. Dual-view briefly coexists for the duration of the purge call. Dev's docstring acknowledges the choice deliberately; QA flagged üü° informational, not blocking. |
| CLI lives under `src/tools/` not `scripts/` | `src/tools/` modules import only project code; `scripts/` is reserved for tools that talk to external services (e.g. `scripts/backtest_baselines.py` paginates Binance public OHLCV). Same packaging / typing as the runtime. |
| `main(argv: list[str] \| None = None)` accepts explicit argv | Tests pass a list (`main(["--retention-months", "6"])`) without depending on the real `sys.argv`. The `__main__` guard wraps `SystemExit(main())` for command-line use. |
| Exit 0 on both "purged" and "nothing to purge" | "Nothing to purge" is a valid steady-state outcome, not an error. Operators wiring the CLI to cron get a green run regardless of whether records were old enough this invocation. |

## Potential Risks

- **Dual-view `ProposalHistory` instances** ‚Äî `run()` constructs a
  fresh `ProposalHistory()` for the purge while the engine has its
  own via `build_engine`. Both default to the same on-disk dir
  (`Settings.data_dir / "proposals"`) so the semantics agree, but
  any future change that introduces per-instance state (e.g. an
  in-memory index) would silently diverge. QA flagged üü°; dev's
  docstring acknowledges the dual-view choice deliberately.
- **Startup-only hook, no in-loop periodic purge** ‚Äî the hook runs
  once per process boot. Operators on long-running deploys (months
  between restarts) have to either `fly machine restart` or invoke
  the CLI manually if proposal volume grows enough that monthly
  records pile up before the next deploy. Phase 11.4 considered the
  in-loop variant and chose startup-only as the simpler shape;
  Phase 12 candidate if proposal volume warrants it.
- **DEBT-008 line drift** ‚Äî the pre-existing mypy lambda error in
  `src/main.py` shifted from line 232 to 271 because of the helper
  insertion. The error is the same; only the line number moved.
  DEBT-008 entry references "line 220" which is now stale ‚Äî should
  be re-anchored when DEBT-008 is fixed (one-line fix candidate per
  the entry).
