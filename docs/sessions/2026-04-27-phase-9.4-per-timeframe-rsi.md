## Work Summary

Closed the user's original ask from earlier this session: "4시간봉 RSI, 15분봉 RSI" as separate baseline strategies. Phase 9.2 had shipped a single universal-cadence `rsi_mean_reversion` baseline that ran on whichever single timeframe the engine selected per cycle — meaning swing (4h) and scalp (15m) RSI behaviour couldn't fire side by side. With Phase 9.1 (multi-TF runtime) and Phase 9.3 (multi-TF backtester) both in place, splitting the cadences is now straightforward: two thin files, both reusing the existing `RSIMeanReversionStrategy` class, each declaring its own `timeframes` list. No framework changes. Signal-logic equivalence is automatic by construction.

The universal entry was renamed `rsi_mean_reversion` → `rsi_universal` for symmetry with the new siblings. Since the universal baseline is `status: experimental` and has no persisted performance records yet, the rename has no production cost.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Two thin wrapper files instead of parameterizing the loader | The strategy framework's contract is "one `TECHNIQUE_INFO` dict per file, one strategy class per file." Adding a "load-time parameter override" to skirt that would mean a framework change for what is essentially a metadata-only split. The wrapper approach respects the existing contract and keeps the loader simple. |
| Reuse `RSIMeanReversionStrategy` directly via `from strategies.rsi import RSIMeanReversionStrategy` | DRY — no math is duplicated, no new tests for the math are needed, and a future fix to the universal RSI logic propagates to all three variants automatically. The loader's "find any `BaseStrategy` subclass in the module" scan picks up the imported class. |
| Re-bind under a local alias (`RSI4hMeanReversionStrategy = RSIMeanReversionStrategy`) | Defensive: if a future module-level scan ever filtered by class name pattern (e.g. matching `*Strategy` only in module `__name__`), the local alias guarantees both the original and the alias resolve to the same class. Currently a no-op but cheap insurance. |
| Rename `rsi_mean_reversion` → `rsi_universal` | Symmetry with the new siblings (`rsi_4h` / `rsi_15m`). The universal variant is still useful as the "match whatever the engine picks" fallback; renaming makes its role explicit. The strategy is still `status: experimental` so no persisted performance records reference the old name yet. |
| Did not touch `ma_crossover.py` or `bollinger_bands.py` | The user's ask was specifically about RSI. Per-cadence splits of the other baselines may make sense later but are out of scope here — the universal-TF entry is sufficient until evidence says otherwise. |

## Potential Risks

- **`rsi_4h` and `rsi_15m` produce nearly-identical confidence values to `rsi_universal` on aligned candles**, so on charts where both fire the dashboard will show three "long" proposals from the same logical setup. The proposal engine ranks by composite score and emits the top-K, so duplicates will be deduped at top-K time — but if the same setup wins three slots, the user sees three identical-looking entries. Acceptable for now; a future "dedupe near-identical proposals" feature could collapse them.
- **Performance tracker fragmentation**: each variant has its own performance history. With three RSI entries instead of one, the early sample sizes per variant grow more slowly. Not blocking — the score formula's `sample_factor` already ramps with sample size — but worth flagging when looking at early dashboard numbers.
- **The rename of `rsi_mean_reversion` → `rsi_universal`** would invalidate any pre-existing performance records keyed by the old name. There are none today (the strategy is experimental + the file system tracker would just produce empty history for the new name). If someone has been running this in production already, they'd see history reset — flag in the next deployment notes.
