## Work Summary

Phase 11.3 closes the live-trading awareness gap carried forward
from Phase 10.1. Live mode runs unattended on Fly; the existing
notifier backends (Console + File) page nobody when a real-money
trade fires. The fix is a `SlackNotifier` posting to an
operator-configured incoming webhook — simplest push backend, no
OAuth, easy operator-side mute/redirect — implementing the existing
`Notifier` protocol so the Phase 6.3 dispatcher picks it up
unmodified.

The non-obvious choices were dependency shape, async shape, and
failure handling. Dependency: dev kept the zero-new-dep posture by
using `urllib.request.urlopen` instead of pulling `httpx` /
`aiohttp` for one POST. Async: `urllib` is sync, so the protocol's
async `send` is satisfied by `asyncio.to_thread` — it keeps the
event loop unblocked without a new dependency. Failure handling:
`SlackNotifier.send` deliberately does **not** swallow `HTTPError`.
Phase 6.3's `NotificationDispatcher` already wraps each channel's
`send` in `try/except` to keep one channel's failure from killing
the others (per-channel failure isolation contract); swallowing
inside the notifier would double-handle and hide signal in the
dispatcher's logs. The qa report specifically verified this with
`test_slack_http_failure_does_not_crash_dispatch`.

Settings shape: `slack_webhook_url` is `Optional[str] = None`. When
unset the notifier is silent / not registered; `main.py::build_engine`
appends `SlackNotifier()` to the dispatcher's notifier list only when
the URL is configured. The presence is logged at INFO; the URL itself
is never logged — `__repr__` redacts to `<redacted>`. Payload format
is exactly the spec in 11.3.b: a `text` line summary
(`{symbol} {side} score={c:.2f} entry={p}`) plus two `mrkdwn` blocks
— bolded summary and a code-fenced detail block with `proposal_id`,
`technique`, SL, TP, qty, leverage, rr. `test_build_slack_payload_text_matches_spec`
asserts the exact string.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| `urllib.request.urlopen` + `asyncio.to_thread` over a new HTTP dep | One POST does not justify pulling `httpx` / `aiohttp` into the runtime. `urllib` is in stdlib; `asyncio.to_thread` keeps the event loop unblocked without a new dependency. Zero-new-dep is the project's standing posture. |
| `SlackNotifier.send` does NOT catch `HTTPError` | Phase 6.3's `NotificationDispatcher` already wraps each channel's `send` in try/except for per-channel failure isolation. Swallowing inside the notifier would double-handle and hide failure signal from the dispatcher's logs. Verified by `test_slack_http_failure_does_not_crash_dispatch`. |
| `Settings.slack_webhook_url` Optional / default None | Non-breaking — existing deployments without the env var are unchanged; the notifier is simply not registered. `main.py::build_engine` is the one decision point. |
| Webhook URL never logged | `__repr__` redacts to `<redacted>`. Build-engine logs presence only. Webhook URLs grant write-access to the channel; they are credentials. Secret-hygiene grep verified by qa. |
| Payload format frozen by spec test | `test_build_slack_payload_text_matches_spec` asserts the exact string. Future drift requires updating both code and the spec test together — that visibility is the point. |

## Potential Risks

- **Webhook URL is a credential, not just config** — anyone with
  the URL can write to the channel. The codebase treats it as a
  secret (redacted in `__repr__`, logged only as presence) but
  operators must know not to commit `.env` or paste the value into
  PRs. `.env.example` documents the var; `docs/deployment.md` walks
  through incoming-webhook setup. No additional in-code mitigation
  beyond the redaction.
- **Single push backend** — Slack-only for now. Telegram / email
  remain future sub-tasks per the 11.3 spec. Operators without a
  Slack workspace have no push channel until those land. The
  fallback (Console + File) still runs, so detection is still
  possible via log scrape; just not push-style paging.
- **Spec test brittleness** — `test_build_slack_payload_text_matches_spec`
  asserts the exact payload string. Any cosmetic change to the
  message format (e.g. tweaking the rationale block layout) breaks
  the test. That brittleness is intentional — the test is the
  spec — but it means cosmetic tweaks must be co-changed with the
  test deliberately.
