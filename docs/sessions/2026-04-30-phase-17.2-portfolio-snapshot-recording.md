## Work Summary

Phase 17.2 wires `PortfolioTracker.record_snapshot` into the runtime
cycle so the Trading dashboard's equity surface actually moves in
production. Pre-17.2, `data/portfolio/` was empty on Fly: the
tracker had been built and unit-tested in isolation since Phase
4.x, the dashboard's Trading page read from it, but no caller in
`src/runtime/engine.py` ever wrote to it. The dashboard
consequently fell back to "Total PnL only" with no equity reading
— an operator looking at the Trading page saw realised P&L but no
sense of "what is the account worth right now". This sub-task
closes that gap: the engine records an `AssetSnapshot` at the end
of every cycle (balances + open-position marks), failures inside
the snapshot path are swallowed and logged so a flaky exchange
ticker query never breaks the trading loop, and the dashboard's
Summary row is rebuilt around Current Equity as the leading metric
card.

The snapshot path is implemented as a private
`_record_portfolio_snapshot(cycle_id)` method on `TradingEngine`,
called once at the end of `run_cycle` after monitoring completes
and before the `CYCLE_COMPLETED` activity event lands. The body
fetches balances via the new `Trader.get_balances()` async
protocol method (paper reads the in-memory ledger; live queries
the exchange), then per-symbol fetches a current ticker for each
open trade so the snapshot's mark-to-market math has live prices.
Both fetch surfaces (balance and ticker) are wrapped in
`try/except Exception` with an activity-log row emitted via
`MONITOR_ERRORED` carrying `{error, phase}` — the cycle continues
either way. The deliberate alternative considered and rejected was
making the snapshot path mandatory (raise on failure); rejected
because a missed snapshot is recoverable on the next cycle (60
seconds later) but a crashed cycle drops every active trade's
monitoring window. The "best-effort visibility, never break the
loop" framing is the same shape the Phase 8 activity-log
swallowing path uses for non-critical writes.

The `Trader` Protocol grew an async `get_balances() -> dict[str,
Decimal]` method so paper and live both feed the recorder through
the same surface. `PaperTrader.get_balances` is `async` purely
for protocol conformance (the in-memory ledger has no I/O);
`LiveTrader.get_balances` calls `self._exchange.get_balance()`
and projects to the `{currency: total}` shape the recorder wants.
`build_engine` in `src/main.py` always wires a fresh
`PortfolioTracker()` and passes the run mode (`paper` / `live`)
plus `quote_currency="USDT"` into the engine constructor. The
`mode: Mode = "paper"` and `quote_currency: str = "USDT"`
constructor kwargs default to safe values so existing test
constructions of `TradingEngine(...)` keep working without churn.

The dashboard's Summary card layout was rebuilt around Current
Equity as the leading metric, with Unrealized P&L as the
delta. Pre-17.2 the equity number lived in a caption row below
the metric cards — visually buried and easy to miss. Promoting
it to the leading card matches the operator question "what is
my account worth right now" that the page is supposed to answer.

## Key Decisions

| Decision | Rationale |
|---|---|
| Snapshot writes never raise — failures emit `MONITOR_ERRORED` and the cycle continues | A missed snapshot is recoverable on the next cycle (60s later); a crashed cycle drops monitoring on every active trade. The "best-effort visibility, never break the loop" framing is consistent with Phase 8's activity-log swallowing for non-critical writes. |
| `Trader` Protocol grows `get_balances()` instead of having the engine reach into trader internals | Keeps the protocol the single source of truth for the trader-engine boundary. Paper / live both implement the method; the engine's recorder path doesn't need to know which mode it's in. |
| `mode` + `quote_currency` injected as constructor kwargs rather than read from the trader | The trader implementation already knows its mode but the protocol intentionally hides that; passing the mode as an explicit constructor argument keeps the protocol thin and avoids leaking trader-implementation detail into the recorder. |
| `PortfolioTracker()` always wired in `build_engine` (no opt-out flag) | The dashboard reads from `data/portfolio/` unconditionally; making the recorder optional in production would just create a confusing "why is equity blank" failure mode. Tests that don't want the recorder pass `portfolio_tracker=None` explicitly. |

## Potential Risks

- **Per-cycle ticker fetches scale with open-position count.** The
  snapshot path fetches a current ticker for each open trade; with
  cross-cycle position cap of 5 (Phase 12.1) this is bounded to 5
  ticker calls per cycle in steady state, but a rate-limit-tight
  exchange could see the cumulative load matter. Caught
  individually by the per-trade `try/except continue` (one bad
  ticker doesn't kill the snapshot), but the cumulative latency is
  not measured.

- **`PortfolioTracker()` writes to `data/portfolio/` synchronously
  on the cycle thread.** A slow disk could add to per-cycle wall
  time. Atomicity across crash mid-write is the open question
  Phase 22.1 addressed (`atomic_write_text`), so on the 17.2
  baseline a crashed cycle could leave the snapshot file torn —
  recoverable on next cycle but flagged here for audit.
