## Work Summary

Phase 22.2 closes DEBT-027 (Medium — paper trader silently zeroes
balance instead of recording liquidation) and seals Phase 22 as a
whole. Pre-22.2, `PaperTrader.close_position` clamped
`balance.free = Decimal("0")` whenever an exit-fee + loss combo
would push free balance negative; the position closed
"successfully" with the loss capped, no liquidation event emitted,
no activity-log row, no negative-equity record. Operationally
equivalent to a margin call the system pretended didn't happen. A
leveraged drawdown that should have liquidated the account looked
identical to a normal close in the trade ledger; operators using
paper-mode to forecast live behaviour saw a softer drawdown
profile than reality. Phase 22.2 ships the contract DEBT-027's
"Suggested Resolution" prescribed: detect the under-water case via
a projected-free predicate (`projected_free = balance.free + (pnl
- exit_fee) < 0`); branch behaviour on it; default-emit a
structured `LIQUIDATED` activity event AND record true negative
equity; keep the legacy clamp behaviour available behind an
opt-out flag for testing scenarios that need a continuing run
after liquidation.

The under-water close detection moved from a balance-already-went-
negative check to a projected predicate evaluated *before* the
mutation is applied, so the branch can choose between recording
true negative equity (the new default) and clamping to zero (the
legacy opt-out behaviour). `ActivityEventType.LIQUIDATED` is the
new structured event with a documented payload contract: `symbol`,
`side`, `entry`, `exit`, `qty`, `realized_pnl`, `balance_before`,
`balance_after`. Both the negative-equity and the auto-deposit
branches emit the event — the difference is solely whether the
balance round-trips with its true value or gets clamped to zero
afterward. `PaperBalance.free`'s Pydantic constraint was relaxed
to drop the `ge=0` floor so the negative-equity round-trip
survives `validate_assignment`; lock / deduct / reserve paths
still enforce overdraw protection at their own boundaries (the
constraint relaxation is a permission to *report* negative
equity, not to silently underflow during normal operations).
`PaperTrader.__init__` gained two backward-compatible kwargs:
`activity_log` (the bus the `LIQUIDATED` event emits onto, defaults
to `None` so legacy callers without an activity log still work)
and `auto_deposit_on_liquidation` (defaults to `False`, the new
correctness-first behaviour). `EngineConfig` / `Settings` mirror
the flag as `paper_auto_deposit_on_liquidation` (env-overridable
via `PAPER_AUTO_DEPOSIT_ON_LIQUIDATION`); `.env.example` documents
it; `src/main.py::build_engine` plumbs `ActivityLog` and the flag
into `build_trader`.

Six regression tests pin the contract:
`test_under_water_close_emits_liquidated_event` (LIQUIDATED event
fires with structured payload on the under-water default branch),
`test_under_water_close_records_negative_equity` (balance round-
trips negative after liquidation),
`test_under_water_close_with_auto_deposit_clamps` (legacy clamp
behaviour preserved behind the opt-out flag — and still emits the
event),
`test_exit_fee_only_shortfall_emits_liquidated_event` (the
historical line-626 fee-shortfall branch — exit-fee alone pushes
balance negative without any pnl loss component — also takes the
liquidation path),
`test_normal_close_does_not_emit_liquidated_event` (happy-path
non-under-water close stays silent — the LIQUIDATED event fires
only when warranted), and a flag-on parity test pinning the event
payload identical between branches. Reviewers both ship-class
(qa 🟢, quant 🟢). Phase 22 seals here: 22.1 ✅ and 22.2 ✅.

## Key Decisions

| Decision | Rationale |
|---|---|
| Default behaviour = record true negative equity (auto-deposit OFF) | Closes the paper-vs-live divergence at the correctness boundary. Operators using paper mode to forecast live behaviour see the same drawdown profile they'd see live (where the exchange liquidates). The legacy clamp-to-zero behaviour is preserved as an opt-out for testing scenarios that need a continuing run after liquidation, but the default flips toward correctness. |
| Detection via projected-free predicate evaluated before mutation | The pre-22.2 code reacted *after* the balance had already gone negative (by clamping it back to zero), conflating detection and remediation. Splitting them — predicate first, branch second — lets the branch choose between two distinct remediation shapes (negative-equity round-trip vs. clamp) cleanly without the detection logic depending on the balance's post-mutation state. |
| `LIQUIDATED` event emitted on **both** branches | The event is the visibility surface — the activity log is what operators read to reconstruct what happened. The opt-out flag changes the *balance treatment*, not the *event semantics*. Suppressing the event under auto-deposit would re-introduce the silent-liquidation bug DEBT-027 named, just gated on a flag instead of being unconditional. The flag controls how the simulator continues; the event records that liquidation happened. |
| `PaperBalance.free` `ge=0` constraint relaxed (not removed structurally) | Dropping the floor permits negative-equity *reporting* through `validate_assignment`; the lock / deduct / reserve paths still enforce overdraw protection at their own boundaries (a strategy can't accidentally over-allocate via the normal flow). The relaxation is scoped to the round-trip surface — the model can hold negative free balance only because the liquidation branch deliberately set it. |
| `__init__` gains `activity_log` kwarg defaulting to `None` | Backward compatibility. Pre-22.2 callers constructed `PaperTrader` without an activity log; making it required would break those callsites. Default `None` means the LIQUIDATED event silently no-ops if no log is wired (the negative-equity round-trip still happens — the visibility-surface fallback is the trade ledger record, which always persists). `build_engine` always wires a real `ActivityLog`. |

## Potential Risks

- **Backtester remains asymmetric (DEBT-047).** Phase 22.2 closes
  the paper-vs-live divergence on the paper-trader side, but
  `Backtester` (`src/backtest/engine.py:371,396`) still does
  `balance += pnl_delta` with no margin lock, no clamp, no
  liquidation event. A backtest that "would have been liquidated"
  in a live or paper run continues simulating against arbitrarily
  negative equity, and the resulting equity curve cannot be
  distinguished from a strategy that drew down deeply but
  recovered. Operators reading backtest equity curves to forecast
  live behaviour now get a *paper-faithful* picture from
  `PaperTrader` and a *liquidation-blind* picture from
  `Backtester`. Registered as DEBT-047 (Medium) with two
  resolution shapes (a `BacktestConfig.liquidation_threshold`
  that emits a structural marker on `BacktestTrade` /
  `BacktestResult`, OR a conservative clamp-and-log when balance
  crosses a threshold). Open at phase seal; not blocking.

- **Constraint relaxation interacts with future model evolution.**
  Dropping `ge=0` from `PaperBalance.free` is a permission, not a
  contract: the field can now hold negative values because the
  liquidation branch deliberately sets them. A future contributor
  reading `PaperBalance` without context could mis-read the
  relaxation as "free balance is unconstrained" and accidentally
  introduce a path that allows silent overdraw outside the
  liquidation branch. Field docstring names the convention; lock
  / deduct / reserve paths still enforce their own bounds. Not
  actionable as a separate item; recorded as a methodology note.

- **Plan-text drift in DEBT-027 line references.** DEBT-027's
  description cited `src/trading/paper.py:619,626` as the under-
  water clamp sites, but by the time 22.2 shipped the actual
  liquidation branch lives in the rewritten `close_position`
  body around lines 656-720. Same pattern as the DEBT-024 stale
  line references and the DEBT-028 / Phase 22.1 path drift —
  long-form prose accumulates stale code references that the
  audit trail does not catch until someone tries to act on them.
  Recorded for future audits.
