## Work Summary

Added ``PerformanceAnalyzer`` that turns a ``BacktestResult`` into a
``PerformanceMetrics`` summary and a human-readable markdown report.
Metrics: win rate, total / annualized return, max drawdown (absolute
and as % of peak), per-trade Sharpe ratio (optionally annualized),
profit factor, expectancy, avg/largest win/loss, and total fees.

Metrics that are genuinely undefined (Sharpe with <2 trades or zero
variance, profit factor with no losses, annualized return on sub-day
runs) return ``None`` so UIs can render "n/a" rather than confusing
sentinel values.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Undefined metrics return `None`, not a sentinel like 0 or -1 | 0 is a valid Sharpe, 0 is a valid profit factor (all losses), so sentinel values mislead. `None` + Optional typing is unambiguous. |
| MDD walk sorts by `exit_time` before iterating | `BacktestResult` should arrive in order but callers re-loading from JSON or hand-crafting results shouldn't silently produce wrong MDDs if ordering drifted. |
| Per-trade Sharpe denominator uses running equity, not a fixed base | Matches how traders experience volatility — a 100-dollar move on a 10,000 account and a 100-dollar move on a 20,000 account have different *feel*. Using running equity is the risk-scaled view. |
| `trades_per_year` is an *optional* Sharpe annualization factor | We don't always know the sampling rate (a 3-trade backtest has no meaningful "per-year" frequency). Making it opt-in keeps the default honest (per-trade Sharpe) while letting callers with enough data annualize. |
| Annualized return returns `None` for runs shorter than 1 day | Compounding a tiny window to yearly is misleading. Under a day → n/a. |
| Annualized return returns `-100.0` when growth is non-positive | A wiped-out account doesn't have a well-defined compounded rate, but UIs need a number. −100% ("lost everything annually") is the least misleading. |
| Report format is a single markdown string with fixed section headers | Stable headers (`## Summary`, `## Returns`, etc.) make it trivial to render into the Streamlit dashboard or paste into a PR/issue. |
| `save_report` writes `report.md` inside the caller-supplied dir | Callers typically pass `backtester.data_dir / run_id` so engine's JSON and analyzer's markdown live side-by-side. But the analyzer doesn't import the engine's `data_dir` — it stays a pure consumer. |
| Profit factor returns `None` when there are *no losses* but `0.0` when there are no *wins* | `None` = undefined ratio (division by zero). `0.0` = gross_profit is legitimately zero and the ratio is well-defined. |

## Potential Risks

- **Sharpe semantics are per-trade by default** — consumers who
  expect the traditional annualized Sharpe must pass a sensible
  `trades_per_year`. Documented but still easy to misread in a
  dashboard that just says "Sharpe: 0.15" without clarification.
- **Currency hardcoded to "USDT"** in the markdown report. The
  `BacktestResult` doesn't yet carry the quote currency; when it
  does, the analyzer should read from it instead of assuming USDT.
  Not blocking for Phase 5.
- **No equity-curve output** — the analyzer walks equity internally
  for MDD but doesn't expose the curve. Phase 7 dashboard will want
  this; either the analyzer grows an `equity_curve()` method or the
  engine starts persisting balances per trade. Defer until 7.1.
