## Work Summary

Phase 17.1 lights up the strategy-evolution stack
(`StrategyImprover` → `Backtester` → `PerformanceAnalyzer` →
`RobustnessGate` → `FeedbackLoop._run_cycle` → `CandidateRecord`)
end-to-end for the first time. The components have shipped and been
unit-tested in isolation since Phase 5.5, but no caller had ever
exercised the full chain on Fly — `data/feedback/state/` and
`data/audit/` were empty in production, and `src/main.py` only
carried a FR-026 placeholder comment. Phase 17.1 closes the
**operator-driven** path first: a manual `python -m
scripts.auto_research_candidates [--picks N] [--dry-run]` invocation
turns the priority matrix's first-wave OHLCV-only picks into
`AWAITING_APPROVAL` candidate records, leaving promotion to the
operator per CON-003. Nightly auto-execution is deferred to a later
sub-task — operator control comes first.

The new `scripts/auto_research_candidates.py` parses
`docs/research/strategies/00-priority-matrix.md`'s first-wave Top-N
picks, dispatches each through `improver.generate_idea(context=<pick
description>)` to land an experimental strategy file, then
`loop.propose_new(...)` to run backtest → robustness gate → state
persistence. One pick failing raises an exception caught at the
per-pick boundary and recorded as `ERRORED` in the run snapshot —
the batch continues. After the batch, a JSON snapshot lands in
`data/research_runs/run_{YYYYMMDD-HHMMSS}.json` with per-pick
`{slug, status, candidate_id, error?}` plus a totals summary, and
the operator-facing stdout summary prints counts by status with each
pick's slug + final state on its own line, followed by an indented
continuation line carrying `decision_reason` + `robustness_summary`
so a DISCARDED pick's *why* is visible without opening the JSON.
`--dry-run` short-circuits before the feedback-loop call, generates
the experimental file under `strategies/experimental/dry_runs/` (so
it never mixes with real gated candidates), prints the planned-pick
list, but skips backtest / robustness gate / state persistence —
useful for validating catalog parsing + prompt output against
`claude -p` without paying the backtest cost.

The catalog-aware improver is the second half of the cycle.
`StrategyImprover.__init__` now accepts `catalog_path: Path | None
= None` defaulting to
`docs/research/strategies/00-priority-matrix.md`; a new private
`_load_catalog` helper reads the file at most once per improver
lifetime, caches the joined string on the instance, and fail-softs
(empty string, INFO log) when the path is missing. The cached
content is injected into `_build_new_idea_prompt` under a clearly
labelled `## Reference Catalog` section so Claude has the full
taxonomy in context instead of regenerating from-scratch ideas every
time. `_build_user_idea_prompt` deliberately does NOT receive the
catalog — the user has already described their idea, and injecting
the catalog would redirect Claude away from the user's intent. This
is a deviation from the original spec wording per
quant-trader-expert review Issue 4; the dev plan's test bullet (b)
was updated in this docs cycle to reflect the post-Issue-4 reality.
`_build_improvement_prompt` is left untouched on the same principle
— improvement is targeted failure-mode analysis on an existing
strategy's trace, not a fresh-idea exercise; the catalog menu would
be off-topic and waste the prompt budget.

The dev's commit `10bbd7f` shipped four scope-fixes from the
quant-trader-expert review in addition to the core feature: (a)
per-timeframe candle defaults bumped (`1h: 4380`, `15m: 8760`) so
the regime gate sees both bull and bear instead of ~6 months of
single-direction tape — Issue 1; (b) the operator stdout summary
surfaces `decision_reason` + `robustness_summary` so DISCARDED
reasons are terminal-visible — Issue 2; (c) dry-run output routes
under `strategies/experimental/dry_runs/` so it never mixes with
real gated candidates — Issue 6; (d) the catalog-injection
deletion from `_build_user_idea_prompt` described above — Issue 4.
Two issues were deferred to TECH-DEBT and surfaced in this docs
cycle: DEBT-013 (Issue 3 — `run_async` constructs its own
`FeedbackLoop` / `BinanceExchange` instead of accepting
caller-built ones; fine for the single-caller `main` today) and
DEBT-014 (Issue 5 — `loop.propose_new` called without a
`param_grid` so the sensitivity gate `SKIPPED`s for every Phase
17.1 candidate; matters more because it makes the robustness
verdict partial). No ADR — 17.1 wires existing components into
an operator script and extends one prompt; no new architectural
seam.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Per-timeframe candle defaults bumped (1h: 4380 = 6 months, 15m: 8760 = 3 months) — quant Issue 1 fix | The pre-fix defaults sized the backtest window so the regime gate saw effectively a single tape direction, which makes the "passes regime split" verdict structurally meaningless — a strategy can pass regime split if both halves are bull, but that doesn't generalise. Bumping the defaults so both halves contain real bull and real bear is the right floor; helper-based per-tf defaults (centralised in the `Pick` dataclass) over per-pick literals so a future operator adding picks doesn't have to re-derive the floor. The downside is longer backtest runtime per pick (4380 1h candles vs the prior smaller window), but the candidate path is operator-driven and not on the cycle's hot path. |
| Dry-run output under `strategies/experimental/dry_runs/` (vs the documented persistence path) — quant Issue 6 fix | The dry-run path generates a strategy file via the improver but does not run the backtest / robustness gate / state persistence. If the file lands directly under `strategies/experimental/`, a subsequent real run could pick it up as if it were a gated candidate, contaminating the gated set with un-gated ideas. Routing dry-run output under a `dry_runs/` subdirectory keeps the two clearly separated and makes "what's in `experimental/`" mean "every file here has been run through the gate at least once" without further guard logic. Cheaper than a sentinel field in the file and operationally the right shape. |
| Skip-vs-reframe for `_build_user_idea_prompt` catalog injection (skip) — quant Issue 4 fix | The original spec wording said the user-idea path should also receive the catalog. On reflection during dev review, this was wrong: when the user has described an idea, injecting the catalog redirects Claude toward the catalog's entries instead of anchoring on the user's text. The user is explicitly saying "I want THIS idea, refine it"; reframing into a catalog pick is editorial overreach. Skip the injection on this path; the new-idea path (where the user has NOT described an idea, so the catalog is the menu being chosen from) keeps it. Locked by `test_catalog_not_in_user_idea_prompt` regression guard. The dev plan's test bullet (b) was updated in this docs cycle to match — code-vs-spec drift goes the spec's way for editorial fixes. |
| `_build_improvement_prompt` also skips catalog injection — pre-existing, validated by 17.1 | Improvement is targeted failure-mode analysis on an existing strategy's trace; the catalog menu would be off-topic. The original spec already said improvement should NOT receive the catalog; 17.1 keeps this invariant and the new `test_catalog_not_in_improvement_prompt` is the regression guard. |
| INFO log (not WARNING) for missing-catalog fallback — code-vs-spec drift resolved spec-side | The original spec wording said the fail-soft path should `WARNING`. The code uses `logger.info(...)` because the missing-catalog case is a normal, operator-friendly fail-soft — the catalog is genuinely optional and a deployment without `docs/research/strategies/` (e.g. a CI box) shouldn't paint operator logs with WARNING noise. Editorial fix: the docs sweep softens the spec wording to `INFO` to match the code. If WARNING is operationally the right level (e.g. "operator should know the catalog isn't being seen"), that's a future-cycle code change, not a docs sweep — surfaced in the session log so it's not lost, but the immediate fix moves the spec, not the code. |
| Surface `decision_reason` + `robustness_summary` in operator stdout summary — quant Issue 2 fix | The original spec just printed slug + status. Without the why, an operator seeing 5 DISCARDED picks has no idea whether to look at the prompt (idea was bad), the data (window was too short), or the gate (parameter unstable) — they have to open every JSON. Indenting a continuation line under each pick row carrying `decision_reason` + `robustness_summary` puts the actionable info one keystroke away. Cheap, no extra IO, no separate flag. |
| No ADR | 17.1 wires existing components (`StrategyImprover`, `FeedbackLoop`, `BinanceExchange`) into an operator script and extends one prompt method (`_build_new_idea_prompt`) with a single new section. The `StrategyImprover.__init__` signature gains an optional kwarg, but the public contract is otherwise bytewise unchanged. ADRs are reserved for component-boundary or constraint decisions; this is operator-tooling glue + one prompt extension. |

## Potential Risks

- **Operator Fly verification still owed** — the script + improver
  changes are unit-tested with mocked Binance + Claude CLI, but no
  real-world run has ever exercised the full chain on Fly. The
  operator needs to invoke `flyctl ssh console --app crypto-master
  -C "python -m scripts.auto_research_candidates --picks 2"` and
  confirm: (a) at least one `CandidateRecord` lands under
  `/data/feedback/state/`, (b) at least one audit entry lands under
  `/data/audit/`, (c) the run snapshot under
  `/data/research_runs/run_{ts}.json` matches the operator-facing
  stdout summary. Without this, the entire Phase 17.1 chain is
  unproven against the production environment it targets.
  Highest-priority operator action for the next cycle.
- **Sensitivity gate `SKIPPED` for every Phase 17.1 candidate** —
  `loop.propose_new` is called without a `param_grid`, so the
  fourth robustness gate (parameter sensitivity) is `SKIPPED` per
  the gate's design. The OOS / walk-forward / regime gates still
  run, but the robustness verdict is partial — a candidate that
  passes the three running gates may be parameter-fragile in a way
  that goes undetected. Recorded as DEBT-014; matters more than
  DEBT-013 because it directly affects the gated-pass rate's
  trustworthiness. Mitigated short-term by the per-tf candle
  defaults from quant Issue 1 (longer windows + bull+bear coverage
  reduce, but don't eliminate, the parameter-fragility blind spot).
- **`evaluable_count` not surfaced in operator summary** — the
  robustness gate's verdict carries internal counts (e.g. how many
  walk-forward windows were actually evaluable vs SKIPPED for
  insufficient data) that the operator can't see from the stdout
  summary alone. The 4380 / 8760 candle floors mitigate by making
  the SKIP-from-insufficient-data case structurally rare, but if a
  pick lands DISCARDED with no obvious reason, the operator may
  have to open the JSON to see whether the gate actually ran or
  was effectively a no-op. Quant flagged this as a soft concern;
  not blocking because the candle-floor mitigation makes it a
  low-frequency case in practice.
- **`run_async` constructs its own `FeedbackLoop` /
  `BinanceExchange`** — currently fine because `main` is the only
  caller, but if a second caller (e.g. a dashboard hook or a
  scheduled job) wants to share an exchange client or a loop
  instance, the fan-out point is wrong. Recorded as DEBT-013;
  cheap fix when a second caller materialises (accept
  `loop=None` / `exchange=None` and construct on absence — the
  signature is already in place, just the production wiring needs
  adjustment). Low-priority because no second caller is on the
  near-term roadmap.
