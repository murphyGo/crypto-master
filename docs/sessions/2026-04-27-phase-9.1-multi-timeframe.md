## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Add explicit `requires_multi_timeframe: bool` to `TechniqueInfo` rather than infer from `len(info.timeframes) > 1` | Existing strategies (RSI, Bollinger, MA crossover, sample_prompt) use `timeframes` to mean "this strategy is *compatible* with any of these TFs". Chasulang uses it to mean "I need *all* of these simultaneously". An explicit flag is the only way to disambiguate without breaking the existing semantics. |
| Keyword-only kwargs (`*, ohlcv_by_timeframe=None, current_price=None`) on `BaseStrategy.analyze` | Backward compatible â€” existing single-TF subclasses don't need to update their signatures, since the engine only passes the kwargs in the multi-TF branch. Code strategies that opt into multi-TF can override with the same signature. |
| Primary timeframe = last entry in `info.timeframes` | Multi-TF templates list TFs macroâ†’micro (chasulang declares `["4h", "1h", "15m", "5m"]`). The smallest TF is the right source for `{ohlcv_data}`, the canonical `{timeframe}` label, and the `current_price` derivation (latest close = freshest tick available). |
| `{current_price}` formatted with `f"{x:f}"` | Decimal's default `__str__` falls into scientific notation for very small altcoin prices (e.g. `9.5E-7`). The `:f` spec forces fixed-point so the LLM gets a literal price string. The OHLCV CSV's per-cell rendering still uses default `__str__` â€” separate concern, out of scope for 9.1. |
| Backtester multi-TF support deferred | The `Backtester.run` walk-forward loop assumes a single pre-fetched candle stream indexed by `i`. Multi-TF needs aligned multi-stream iteration (different bar counts per TF, timestamp alignment). That's a non-trivial refactor and isn't required for the engine path that motivated 9.1. Single-TF backtests for every existing strategy + the 9.2 baselines remain fully functional. |
| Engine fetches TFs sequentially, not in parallel | Keeps the failure mode simple: any TF fetch that raises `ExchangeError` propagates up to the existing `propose_altcoins` per-symbol guard, so one bad TF skips the whole symbol cleanly. Parallelism (e.g. `asyncio.gather`) is a future optimization but adds error-aggregation complexity that's not needed today. |

## Potential Risks

- **Multi-TF strategies in the runtime engine will now make N exchange calls per cycle instead of 1** for that strategy's symbols. Chasulang declares 4 TFs Ă— 3 symbols (BTC/ETH/XRP) = 12 OHLCV calls per cycle just for that one strategy. At Binance public rate limits this is still well within budget, but worth watching once the production engine starts running it.
- **Code strategies cannot currently opt into multi-TF**. The framework supports it (the kwargs are on the abstract signature) but no `.py` baseline declares `requires_multi_timeframe=True` today. Adding one is straightforward â€” just override `analyze` to accept the kwargs and consume them. Not blocking 9.1.
- **The follow-up `rsi_4h.py` / `rsi_15m.py` split (9.2 deferred item) is now unblocked.** The infrastructure exists; the per-timeframe variants are a separate sub-task.
