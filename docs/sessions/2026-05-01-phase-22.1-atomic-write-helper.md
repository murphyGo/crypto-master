## Work Summary

Phase 22.1 closes DEBT-028 (Medium — non-atomic JSON persistence
across `TradeHistoryTracker` / `PortfolioTracker` /
`ProposalHistory` / Phase 18.1 stale-quote rewrite) ahead of
Phase 19.2's sub-account fan-out, which materially raises the
concurrent-writer rate against the same persistence files. The
shape is the one DEBT-028's "Suggested Resolution" prescribed: a
single helper `atomic_write_text(path, text)` in a new
`src/utils/io.py` that writes to a tmp path then `os.replace`s
into the destination, plus mechanical one-line replacements at
each named site. The implementation chose a uuid-suffixed tmp
file (`path.with_suffix(path.suffix + f".{uuid4().hex}.tmp")`)
rather than the simpler `path.with_suffix(path.suffix + ".tmp")`
DEBT-028 sketched, on the grounds that concurrent writers
sharing a fixed `.tmp` suffix would step on each other on the
tmp side even though the `os.replace` step itself is atomic.
Cleanup-on-exception is wired so a raise between tmp-write and
`os.replace` removes the orphan tmp rather than leaving it on
disk for the next operator to wonder about.

Five named load → mutate → save sites were migrated:
`PerformanceTracker._save_records` (`src/strategy/performance.py:
439`), `PerformanceTracker._update_summary` (`:494`),
`TradeHistoryTracker._save_trades` (`:1077`),
`PortfolioTracker._save_snapshots` (`src/trading/portfolio.py:
407`), and `ProposalHistory.save` (`src/proposal/interaction.py:
245`). The sixth named DEBT-028 surface —
`src/runtime/engine.py::_record_stale_quote_rejection` — is
covered transitively: the function calls `ProposalHistory.save`
which is now atomic, so `_record_stale_quote_rejection`'s
write-side is atomic by construction without a direct migration.
A doc comment at the call-site names the transitive coverage so
a future contributor doesn't add a redundant direct route.

The plan text itself needed a one-line correction. Phase 22.1's
spec line in `docs/development-plan.md` and DEBT-028's
description both pointed at `src/proposal/history.py` as the
location of `ProposalHistory.record` / `update`, but
`ProposalHistory` actually lives in `src/proposal/interaction.py`
— `src/proposal/` ships `engine.py`, `interaction.py`, and
`notification.py`; there is no `history.py`. Senior-developer
flagged this during implementation (the migration could not
proceed against the cited path because the file does not exist).
The plan reference was corrected in-place at the Phase 22.1
sub-task block.

Reviewers both returned ship-class (qa 🟢, quant 🟢) and surfaced
three follow-up debts that the lead emphasised as load-bearing
for the Phase 19.2 prereq question. **DEBT-044** (Low) — senior-
developer caught `FeedbackLoop.save_state` (`src/feedback/loop.
py:444`) using the same load → mutate → save shape but outside
DEBT-028's named scope; mechanical one-line fix using the new
helper. **DEBT-045** (Low) — quant noted `Backtester._save_result`
(`src/backtest/engine.py:1057`) is a single-write (no load →
mutate cycle) but still benefits from atomicity because a
backtest run that crashes during result persistence corrupts a
finished result file even without a race. **DEBT-046** (Medium,
**hard prereq for Phase 19.2**) — the durability-vs-concurrency
caveat: `atomic_write_text` resolves last-writer-wins durability
(destination is either fully old or fully new, never partial)
but does **not** solve concurrent-mutation loss. Two workers
each doing `data = json.loads(path.read_text());
data.append(record); atomic_write_text(path, json.dumps(data))`
in the same wall-clock window will both see the same prior
state, both write atomically, and the loser's mutation is
silently dropped. Single-engine deployment is safe today (one
writer per file); Phase 19.2's sub-account fan-out introduces
parallel workers and requires either a per-file lock helper
(`fcntl.flock` on POSIX) layered over `atomic_write_text`, or
per-account file partitioning aligned with 19.2's
`{sub_account_id}` path layout. The Phase 19.2 spec page now
carries an explicit "Prerequisites:" line citing DEBT-046 so
the planner can't ship 19.2 without picking a resolution shape.

## Key Decisions

| Decision | Rationale |
|---|---|
| uuid-suffixed tmp file rather than fixed `.tmp` suffix | DEBT-028's "Suggested Resolution" sketched a fixed `.tmp` suffix, but concurrent writers sharing the same fixed suffix would step on each other on the tmp side (each writer's tmp file is the same path; the second writer truncates the first's in-progress tmp before its own `os.replace`). The atomicity of `os.replace` itself protects the destination, but the tmp side becomes a collision surface that the `cleanup-on-exception` block can no longer reason about correctly. uuid-suffixed tmp eliminates the collision by construction at zero meaningful cost (16 bytes of filename, one `uuid4()` call per write). |
| `_record_stale_quote_rejection` covered transitively, not directly migrated | The function's persistence path *is* `ProposalHistory.save` (the function calls into `ProposalHistory` to record the rejection event). Once `ProposalHistory.save` is atomic, `_record_stale_quote_rejection`'s write-side is atomic by construction. A direct migration would either duplicate the persistence path (bypass `ProposalHistory.save` and write the rejection record directly) or wrap an already-atomic call in another atomic call (no-op). A doc comment at the call-site names the transitive coverage so a future contributor refactoring `_record_stale_quote_rejection` doesn't lose the atomicity guarantee silently. |
| Plan text corrected in-place, not via planner cycle | The DEBT-028 description and the Phase 22.1 spec line both pointed at `src/proposal/history.py`, which does not exist in the codebase. The migration could not proceed against the cited path. The correction is a one-line factual fix (path drift, not a design change), and the Phase 22.1 cycle was already mid-flight; routing through a planner cycle would have blocked the implementation for a typo. Corrected in-place at the Phase 22.1 sub-task block in `docs/development-plan.md`; recorded in the Resolved-DEBT-028 entry and this session log. |
| Three follow-up debts registered, not absorbed into Phase 22.1 scope | `FeedbackLoop.save_state` (DEBT-044) and `Backtester._save_result` (DEBT-045) are mechanical one-line fixes but were outside DEBT-028's named scope; absorbing them would expand the cycle's review surface beyond the spec. The durability-vs-concurrency caveat (DEBT-046) is a fundamentally different problem class — atomicity ≠ concurrency-safety — and resolving it requires a planner-level decision between per-file locking and per-account file partitioning, which is Phase 19.2's territory. Registering them as separate DEBT items keeps Phase 22.1 narrowly-scoped and surfaces the Phase 19.2 prereq cleanly. |

## Potential Risks

- **DEBT-046 — atomicity ≠ concurrency-safety, hard prereq for
  Phase 19.2.** This is the headline risk and is captured as a
  separate DEBT item plus a Prerequisites line on the Phase 19.2
  spec page. Today (single-engine deployment, one writer per
  file) the risk is dormant. The moment 19.2's sub-account
  fan-out lands without addressing DEBT-046, parallel workers
  doing load → mutate → save against the same persistence files
  can silently drop each other's mutations — atomic writes
  prevent torn files but not lost updates. Resolution shape is
  planner territory (per-file lock via `fcntl.flock` OR
  per-account file partitioning aligned with 19.2's
  `{sub_account_id}` layout); both are viable, both are sketched
  in DEBT-046's Suggested Resolution.

- **Two adjacent-scope sites left unmigrated by design.**
  `FeedbackLoop.save_state` (DEBT-044) and
  `Backtester._save_result` (DEBT-045) use direct
  `Path.write_text(...)` without going through the new helper.
  The risk profile is the same as the migrated sites
  (crash-mid-write corrupts the file) but lower frequency
  (`save_state` is single-writer in single-engine mode;
  `_save_result` is single-write per backtest run). Picking up
  both is mechanical — the helper exists, the call sites are
  one-line — but they were outside Phase 22.1's named scope.
  Registered as DEBT items so a future cycle (or operator-
  reported corruption) has a place to start.

- **Plan-text drift caught by chance.** DEBT-028's description
  and the Phase 22.1 sub-task spec line both pointed at a
  non-existent `src/proposal/history.py`. The drift was caught
  during 22.1 implementation (the migration could not find the
  cited file), not by the audit pass that wrote DEBT-028 in the
  first place. This is a soft signal that path references in
  long-form prose (DEBT items, spec blocks) drift over time and
  the audit trail does not catch them until someone tries to
  act on the path. Not actionable as a separate item; recorded
  as a methodology note for future audits.
