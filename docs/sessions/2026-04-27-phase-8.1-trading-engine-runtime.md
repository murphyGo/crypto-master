## Work Summary

Built the long-running headless engine that wraps the existing
proposal / paper-trader / notification stack into a real production
loop. New `TradingEngine` runs cycles of: scan with `ProposalEngine`
→ auto-decide via injected callback → notify → execute on
`PaperTrader` → monitor open positions for SL/TP → sleep. Every step
is recorded to a new append-only JSONL `ActivityLog` so the
dashboard (8.2) and operators have a clean event stream to read.

Key wiring choices:

- The engine **reuses `ProposalInteraction`** rather than building a
  parallel decision path. It overrides the interaction's
  `_decision_callback` with `_auto_decide` (composite ≥ threshold).
  Result: every proposal lands in `data/proposals/` with the same
  shape as a manually-decided one — the dashboard's existing
  Proposals view works unchanged.
- New `ProposalHistory.attach_trade(proposal_id, trade_id)` lets the
  engine link the executed trade to its proposal at open time
  without lying about the realized outcome (which `attach_outcome`
  always sets `outcome_recorded_at` on). Realized P&L is filled in
  by `attach_outcome` later when the position closes.
- Loop is **interruptibly sleeping** via
  `asyncio.wait_for(stop_event.wait(), timeout=...)`. SIGINT / SIGTERM
  in `main.py` flip the event so `fly deploy` rolling restarts give
  the engine a clean exit instead of a kill.
- `_find_proposal_for_trade(trade_id)` scans
  `ProposalHistory.list_all()` to map closed trades back to their
  proposals. Linear in record count — fine at current scale; can grow
  an index if ever bites.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Engine reuses `ProposalInteraction` (overrides its callback) | One persistence path → same `data/proposals/` shape for headless and manual modes; dashboard works unchanged. |
| Separate `attach_trade` method (not just calling `attach_outcome` with `pnl=0.0`) | `attach_outcome` semantically means "outcome is realized" — sets `outcome_recorded_at`. Calling it at open time would lie about realization. The new method only fills `trade_id`; outcome stays pending until the close path runs. |
| Interruptible sleep via `wait_for(stop_event.wait(), timeout=N)` | Single primitive handles both normal sleep (TimeoutError = path) and shutdown (event set wakes immediately). No threads, no signals propagating into the loop. |
| Single-cycle errors logged but don't crash the loop | Trading engines that crash on first error are useless. Cycle wrapper catches everything, writes `CYCLE_ERRORED`, sleeps, retries. The error path is exercised in tests via `ExchangeAPIError` injection. |
| Engine logs every step to its own activity log (not the existing `AuditLog`) | `AuditLog` is the feedback-loop's audit trail (CON-003 traceability for technique adoption). Mixing engine cycle events would make both noisier. Two logs, one purpose each. |
| Proposal-to-position translation is a private helper, not a method on `Proposal` | Keeps `Proposal` headless (no trader-side dependencies). The engine owns the translation since only it needs it today. |

## Potential Risks

- **Scan-error retry has no backoff.** If Binance is rate-limiting,
  every 5-minute cycle hits the same 429. Acceptable for a personal
  paper deploy; if it bites in production, add exponential backoff
  to `_scan` keyed off `SCAN_ERRORED` event count.
- **`_find_proposal_for_trade` is linear.** Loads every record from
  disk on every closed trade. Fine for tens-to-hundreds of records,
  not for thousands. If proposal volume grows, add a `trade_id →
  proposal_id` index file or a retention policy. Already flagged on
  the existing carry-forward list.
- **`monitor_interval_seconds` is unused right now.** The engine
  monitors once per cycle. If 5-minute cycles miss tight SL/TP
  windows, split monitoring into its own task with the shorter
  cadence. Out of scope for 8.1 (KISS).
- **`main.py` only builds Binance or Bybit, picks Binance first.**
  No way to choose at runtime today. A `TRADING_EXCHANGE` env var
  would be a small follow-up if it matters.
- **Activity log grows unbounded.** No rotation. Same risk as the
  audit log and other JSONL stores; revisit with a retention policy
  before live trading.
