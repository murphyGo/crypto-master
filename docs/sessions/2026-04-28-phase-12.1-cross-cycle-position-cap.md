# Session Log: 2026-04-28 - Phase 12.1 - Cross-Cycle Position Cap

## Overview

- **Date**: 2026-04-28

## Work Summary

Phase 12.1 closes the real-money risk surfaced by the 2026-04-28 Fly
redeploy that produced two BNB short positions in 14 minutes (05:40:51
and 05:54:30). Phase 10.6's per-cycle dedup operates *within* a cycle
only — across cycles, nothing in the engine consulted
`trader.get_open_trades()` before opening another position on the same
symbol, so 4× cycle = 4× position concentration on a single pair, 4×
the intended `risk_percent`. Quant flagged this risk during Phase 10.6
design ("Option (b) cap at TradingEngine — defensible
belt-and-braces, follow-up sub-task") but it was deferred. With the
runtime now actively trading real money, deferring further is a
genuine concern.

The fix is a hard cap at the engine layer, not the proposal-engine
layer: proposal generation continues unchanged; cap operates at the
execution gate. `EngineConfig.max_open_positions_per_symbol` (default
1, `ge=1` validator) is env-overridable as
`ENGINE_MAX_OPEN_POSITIONS_PER_SYMBOL` via
`Settings.engine_max_open_positions_per_symbol` (Phase 10.2 pattern).
`TradingEngine._handle_proposal` checks `trader.get_open_trades()`
filtered by symbol *after* the composite-accept gate; on cap-hit it
increments `proposals_rejected`, logs `PROPOSAL_REJECTED` with reason
`"symbol X cap N reached (M open)"` plus structured details (`cap`,
`open_count` keys), and skips `_execute`. The activity log re-uses the
existing `proposal_rejected` event shape so the dashboard timeline
surfaces it without UI churn. Phase 10.6 within-cycle dedup is
untouched — the two checks are properly orthogonal.

Backward-compat: cap=1 reproduces the pre-12.1 effective behaviour for
single-position-per-symbol deployments. The default of 1 is the safe
bet for live mode; operators who want concurrent positions on the
same symbol (e.g. for laddered entries) can raise the cap by env. The
ge=1 validator prevents a misconfigured cap=0 from silently blocking
all execution.

5 new tests in `tests/test_runtime_engine.py` cover the default value,
env wiring, cap-hit rejection, cap-not-reached execution, and the
other-symbol-doesn't-block path. 1104 → 1109 tests. ruff clean; mypy
14 errors all pre-existing in entry-point chain (will be addressed in
12.2). Quant's review caught one test gap: the cap correctly blocks
the long+short same-symbol case (cap counts trades regardless of
side, which is the safe behaviour preventing accidental synthetic
hedge), but the test suite does not explicitly cover that path.
Recorded as DEBT-010 (Low) — recommend adding
`test_cap_blocks_opposite_side_same_symbol` in a follow-up cycle.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Cap layered after composite-accept gate, not before | The proposal still gets recorded for audit; only execution is blocked. Matches the spec ("Hard cap at the engine layer, NOT at the proposal-engine layer — proposal generation continues unchanged; cap operates at the execution gate"). |
| Re-use `PROPOSAL_REJECTED` event shape | Dashboard timeline surfaces the cap rejection without UI churn. Reason strings (`"composite below threshold"` vs `"symbol X cap N reached (M open)"`) are distinguishable for downstream filtering. |
| Symbol-scoped count via `trader.get_open_trades()` filtered by symbol | `get_open_trades` returns only `status=open` trades (verified by quant during review), so the cap's denominator is correct without further status-filter logic in the engine. |
| Default cap = 1, `ge=1` validator | Safe default for live mode (reproduces pre-12.1 effective behaviour for single-position-per-symbol deployments). `ge=1` prevents a misconfigured cap=0 from silently blocking all execution. |
| Cap counts trades regardless of side (long+short both count) | Prevents accidental synthetic hedge — opening a long on a symbol that already has an open short is an unintended risk-doubling, not a hedge. Quant confirmed this is the correct behaviour; test gap recorded as DEBT-010. |
| Phase 10.6 dedup untouched | Within-cycle (Phase 10.6) and across-cycle (Phase 12.1) dedup are properly orthogonal: 10.6 keeps the highest-composite winner per symbol within a single proposal batch; 12.1 blocks execution when a prior cycle's position is still open. Both run; neither subsumes the other. |

## Potential Risks

- **Long+short same-symbol test gap (DEBT-010)** — the cap correctly
  blocks the long+short same-symbol case (cap counts trades regardless
  of side, which is the safe behaviour preventing accidental synthetic
  hedge), but the test suite does not explicitly cover that path.
  Implementation is correct; absence of an explicit test means a
  future refactor that splits the count by side could regress the
  protection without anything failing red. Recommend adding
  `test_cap_blocks_opposite_side_same_symbol` in a follow-up cycle.
- **Pre-existing mypy errors in entry-point chain** — 14 mypy errors
  remain in the entry-point chain. Phase 12.1 introduced no new
  errors. Phase 12.2 (Residual mypy Sweep) will close these as part
  of the planned mini-sweep cycle.
- **Cap = 0 silently disables execution if validator slips** — `ge=1`
  is the only thing standing between an operator typo
  (`ENGINE_MAX_OPEN_POSITIONS_PER_SYMBOL=0`) and a silently dead
  trading runtime. The validator catches this at `Settings`
  construction time so the failure is loud and early; worth keeping
  on the radar if the validator is ever relaxed.
