## Work Summary

Phase 13.1 closes three small carry-forward TECH-DEBT items that
accumulated across Phases 11 and 12. None of them shifted runtime
behaviour; the value of the batch is that it removes a CI footgun
(DEBT-009), pins a safety invariant the implementation already
upholds (DEBT-010), and drops a small typing lie that propagated to
every consumer of `build_summary_metrics` (DEBT-011). Bundling the
three into one cycle keeps the audit overhead per cleanup low — none
of them individually warranted a phase entry, but the batch
collectively closes Phase 11.1's QA-flagged lint script issue and
Phase 12.1/12.2's "ship with note" follow-ups.

DEBT-009 split `scripts/lint.sh` into two scripts: the original
filename now runs `ruff check src tests && mypy src` (no `--fix`,
report-only — safe for CI / pre-commit) and a new
`scripts/lint-fix.sh` keeps the dev-convenience `--fix` form. Both
are executable. The split is the explicit shape the DEBT-009 entry
suggested (vs the alternative of dropping `--fix` outright); two
scripts means dev workflow doesn't change while CI gets a script it
can wire without auto-mutating the working tree. Any docs that
reference `scripts/lint.sh` now mean the report-only variant
unambiguously.

DEBT-010 added `test_cap_blocks_opposite_side_same_symbol` in
`tests/test_runtime_engine.py`. Setup: 1 BNB long open + BNB short
proposal at composite=2.0; cap=1 → `positions_opened=0`, no
`open_position` call, `PROPOSAL_REJECTED` logged with the BNB symbol
and "cap 1 reached" reason. The test setup matches `engine.py:394-417`
cap-branch exactly so a future refactor that splits the count by
side regresses red instead of silently letting through a synthetic
hedge. Implementation was already correct in Phase 12.1 — the cap
counts trades regardless of side; this test just locks the
invariant against future regression.

DEBT-011 replaced `dict[str, object]` returns from
`build_summary_metrics` with per-page `TypedDict`s
(`TradingSummaryMetrics` in `src/dashboard/pages/trading.py`,
`EngineSummaryMetrics` in `src/dashboard/pages/engine.py`). Per-page
TypedDicts (rather than a shared `_types.py` declaration) chosen
because the two shapes differ — `trading.py`'s metrics dict has
trading-specific keys (PnL, position counts) while `engine.py`'s has
engine-cycle-specific keys (cycles count, last-cycle status); a
shared TypedDict would be a union that consumers still need to
narrow. Consumer-side `cast(int, ...)` / `cast(float, ...)` calls
dropped at every access site; mypy infers the narrowed type from the
TypedDict. No `from typing import cast` left in either file. Pure
refactor — no behavioural change.

## Potential Risks

- **`engine.py:213` docstring drift** (QA-flagged 🟡, cosmetic) — the
  docstring still says "Returned dict keys" though the return type is
  now `EngineSummaryMetrics` TypedDict. Pure cosmetic; runtime is
  correct, the TypedDict's field names are self-documenting at the
  declaration site, and the docstring description still narrates what
  the keys mean. Optional follow-up; not worth a cycle on its own.
- **Per-page TypedDicts vs shared `_types.py`** — by choosing
  per-page declarations the dashboard does have two TypedDict
  definitions instead of one. If a third page ever adds a
  `build_summary_metrics` with overlapping fields, the
  per-page-vs-shared tradeoff should be re-evaluated; for now the
  shapes diverge enough that per-page is the cleaner choice.
- **DEBT-010 test specificity** — the test pins the exact
  `engine.py:394-417` cap-branch path including the symbol-cap
  reason string `"cap 1 reached"`. A future reformat of the rejection
  reason string could fail this test on a string match even though
  the behavioural invariant holds. Acceptable trade-off (the string is
  operator-visible and worth pinning); just noting the coupling.
