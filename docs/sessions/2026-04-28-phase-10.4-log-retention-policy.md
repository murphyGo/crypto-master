## Overview

## Work Summary

Phase 10.4 closes the unbounded-growth gap on `data/audit/feedback.jsonl`,
`data/runtime/activity.jsonl`, and `data/proposals/`. Phase 5 / 7 / 8
risk lists all flagged this — current volumes are tiny but post-deploy
the files balloon at a few MB/day. This is the last 10.x sub-task; with
it shipped, Phase 10 is complete.

The new `JsonlRotator` (`src/runtime/jsonl_rotator.py`, 259 lines) wraps
an append-only JSONL file with **time-based monthly rotation**. Writes
go to `<base>.YYYY-MM.jsonl` derived from `datetime.now()` per append,
so a process that runs across a month boundary correctly switches files
without restart. Reads merge the active month plus the most-recent
`Settings.log_retention_months` archives in **timestamp order** (records
without a parseable `timestamp` field sort to the end in encounter
order). Malformed lines are skipped tolerantly with a warning — the
same policy as the un-rotated predecessors. Legacy un-rotated
`<base>.jsonl` files (anything written before this change landed) are
treated as the oldest available archive: they are read on merge but
never written. Dev's rationale for the read-as-legacy strategy over a
first-read migration: migration is destructive (a botched rename loses
history), the legacy records carry their own `timestamp` so merging by
timestamp produces correct ordering, and operators can rename the file
by hand once they confirm the new rotated files are accumulating.

`AuditLog` and `ActivityLog` now compose the rotator. The non-obvious
piece is backward compat: existing tests pass `tmp_path / "audit.jsonl"`
or `tmp_path / "activity.jsonl"` — the `__init__` strips the trailing
`.jsonl` to derive the rotator base, so the public `self.path`
attribute keeps the `.jsonl` form callers reference for display while
the rotator works off the suffix-stripped base. `append` round-trips
the Pydantic event through `model_dump(mode="json")` so the rotator
receives a uniformly-serializable dict (datetimes → ISO strings).
`read_all` no longer cares whether the file exists — the rotator
handles missing files as empty.

`ProposalHistory` is the odd one out: it already uses one JSON file per
proposal, so monthly rotation doesn't fit. Dev gave it a sibling
`purge_old(now, retention_months)` method instead, which walks
`<data_dir>/*.json`, parses each proposal's `created_at`, and for any
record older than `now - 30 * retention_months` days moves the file to
`<data_dir>/archive/<YYYY-MM>/<original_filename>` keyed on the
proposal's *own* creation month (so the archive bucket lines up with
the matching audit / activity rotations). Idempotent — re-running with
the same retention does nothing because the archive lives under a
subdirectory the top-level glob ignores. `purge_old` is
operator-callable: the engine does **not** call it on every write. A
startup hook or CLI command can invoke it; for Phase 10.4 the method
exists, is tested, and is left wired up to nothing. Adding the startup
hook is a deliberate separate sub-task if the operator wants it.

`Settings.log_retention_months` defaults to 12, validated `ge=1`
(zero or negative retention is meaningless), loadable via the
`LOG_RETENTION_MONTHS` env var, documented in `.env.example`.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Time-based monthly rotation, not size-based | Per-month layout is human-readable, makes retention semantics ("12 months back") trivial, and the rotation token is derived from `datetime.now()` per append (cross-month-safe without a restart). Size-based rotation needs a state file or a stat-on-every-write which isn't worth the complexity for the volumes involved. |
| Legacy un-rotated file is read but never written | Migration on first read would be destructive — a botched rename loses history. The legacy file's records carry their own `timestamp`, so merging by timestamp produces correct ordering. Operators can rename the file by hand once they confirm the rotated files are accumulating. |
| Strip trailing `.jsonl` to derive rotator base | Existing tests pass `tmp_path / "audit.jsonl"` and inspect `self.path`. Stripping the suffix internally lets the rotator work off `<base>.YYYY-MM.jsonl` while preserving the `self.path` contract callers depend on for display / log lines. Zero call-site churn. |
| `ProposalHistory.purge_old` is operator-callable, no startup hook | The engine doesn't need to purge on every write — purge is a one-shot operator action triggered when archives accumulate. Adding a startup hook is a separate sub-task if wanted. Phase 10.4 ships the method + tests; wiring is deferred. |
| `purge_old` keys archive on the proposal's own creation month, not on `now` | The archive bucket lines up with the matching audit / activity rotations — an operator browsing `archive/2024-01/` sees only proposals created in January 2024. If we keyed on `now`, all archived records would land in a single bucket per purge invocation, defeating the chronological browseability. |
| Retention default 12 months, `ge=1` floor | 12 months matches the prior phase risk-list expectations and gives operators a year of merged history by default. `ge=1` prevents zero/negative configs that would either drop everything or behave undefined. |
| `30 * retention_months` for the cutoff (calendar approximation) | A calendar-exact cutoff via `relativedelta` would be sharper but the approximation is well within the operational tolerance of an "old enough to archive" judgement. QA flagged this as "acceptable but sharper with `relativedelta`"; auditor agreed not worth recording as debt. |

## Potential Risks

- **Calendar approximation in `purge_old`** — uses `30 * retention_months`
  days for the cutoff rather than `relativedelta(months=retention_months)`.
  At 12 months default retention this is a ~5-day approximation
  (12 × 30 = 360 vs ~365). QA flagged "acceptable but sharper with
  `relativedelta`"; auditor judged not worth tracking. If a future
  operator hits a near-cutoff edge case, swap in `relativedelta` —
  one-line change.
- **No startup hook for `purge_old`** — the method exists and is tested
  but nothing in the runtime calls it. Operators have to invoke it by
  hand (or via a CLI / scheduled cron). Until then, archive growth is
  bounded only by retention reads, not by file count on disk. The
  legacy un-rotated files also stay forever unless an operator deletes
  them.
- **Legacy file slot in retention window** — when the legacy
  un-rotated file exists, it counts as "the oldest archive" and slots
  in front of the rotated archives. If an operator has 12 rotated
  months *and* a legacy file, the merged read window is effectively
  13 sources (rotated[-12:] + legacy). Minor; matters only if the
  legacy file is huge.
