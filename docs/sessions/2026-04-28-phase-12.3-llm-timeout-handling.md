## Work Summary

Phase 12.3 closes the reliability gap surfaced by Fly monitoring on
2026-04-28 — `chasulang_ict_smc` failing twice within 12 minutes with
`Claude CLI timed out after 120.0 seconds`. The error was logged but
the strategy silently dropped out of that cycle's multi-technique
scan with no fallback, no retry, and no operational visibility
beyond the log line. As LLM strategies multiply this becomes a
reliability concern. Phase 12.3 wraps the existing `subprocess.run(...,
timeout=...)` call in `src/ai/claude.py` with a configurable
retry-on-timeout loop (1.5× backoff) and emits an
`ActivityEventType.LLM_TIMEOUT` event on final exhaustion so the
dashboard can show LLM reliability over time.

The non-obvious piece is the exception hierarchy. `ClaudeTimeoutError`
now multiply-inherits `ClaudeError + StrategyError` so the engine's
existing `StrategyError` catch handles it cleanly without a
dedicated except block at every call site, while still preserving
the `ClaudeError` identity for callers that want it. The flip side:
`PromptStrategy.analyze` had to be taught to re-raise
`ClaudeTimeoutError` UNWRAPPED — every other `ClaudeError` subtype
still wraps into `StrategyError(...)` per the pre-existing contract,
but unwrapping the timeout case is what lets the engine emit the
`LLM_TIMEOUT` event with the original `timeout_seconds` detail
intact (a wrapped `StrategyError` would lose that). One unwrap
test (`test_unwrap_propagation`) locks the contract.

`Settings.claude_cli_timeout_seconds` (default 120, `ge=10`) and
`claude_cli_max_retries` (default 1, `ge=0`) both env-overridable
following the Phase 10.2 pattern. Max retries 0 = single shot (no
retry); default 1 = original timeout + one retry at 1.5× = 120 →
180s. Retry only on `asyncio.TimeoutError` — any other exception
propagates immediately, verified via
`test_non_timeout_errors_do_not_trigger_retry`
(`mock_exec.call_count == 1`). Process cleanup is per-attempt so
there's no zombie risk if the retry loop fires multiple times.

`ProposalEngine` accepts an optional `activity_log` (None default
preserves backward-compat) and emits `LLM_TIMEOUT` with
`strategy_name`, `version`, `symbol`, and `timeout_seconds` on final
exhaustion. `build_engine` creates one `ActivityLog` and shares it
between both `ProposalEngine` and `TradingEngine` so timeline events
from both sources land in the same JSONL stream.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| `ClaudeTimeoutError` multiply-inherits `ClaudeError + StrategyError` | Lets the engine's existing `StrategyError` catch handle it without a new except block at every call site, while preserving `ClaudeError` identity for callers that want it. MRO `[ClaudeTimeoutError, ClaudeError, StrategyError, Exception, ...]` confirmed. |
| `PromptStrategy.analyze` re-raises `ClaudeTimeoutError` UNWRAPPED | Wrapped `StrategyError(...)` would lose the original `timeout_seconds` detail. Unwrapping lets the engine emit `LLM_TIMEOUT` with the original payload intact. Other `ClaudeError` subtypes still wrap per the pre-existing contract. Locked by `test_unwrap_propagation`. |
| 1.5× escalation per retry attempt (not 2×) | Verified arithmetic 120 → 180 → 270 — gives a meaningful additional headroom on retry without doubling the worst-case cycle latency. Default `max_retries=1` keeps the 95th-percentile cost bounded at 300s for the timeout path. |
| Retry only on `asyncio.TimeoutError` | Any other exception is a real bug — retrying would mask it. Locked by `test_non_timeout_errors_do_not_trigger_retry` (`mock_exec.call_count == 1`). |
| `Settings.claude_cli_max_retries` default 1, `ge=0` (0 = single shot) | Conservative default — one retry is enough to clear transient CLI hangs without compounding cycle latency. `ge=0` allows operators to disable retry entirely (useful for backtesting / CI / debugging). |
| `ProposalEngine.activity_log` optional with `None` default | Backward-compat preserved — pre-12.3 callers don't need to pass an `ActivityLog`. The engine no-ops the `LLM_TIMEOUT` emission when `activity_log is None`. |
| `LLM_TIMEOUT` event details include `strategy_name`, `version`, `symbol`, `timeout_seconds` | Four required fields cover the dashboard's reliability-over-time view: which strategy, at what version, for which symbol, with what cumulative timeout budget. Verified by QA. |
| `build_engine` shares one `ActivityLog` between proposal + trading engines | Single JSONL stream for timeline events from both sources keeps the dashboard's filtered view coherent (one cycle's `LLM_TIMEOUT` and the resulting `PROPOSAL_REJECTED` land adjacent). |

## Potential Risks

- **Retry compounds cycle latency** — `max_retries=1` at 1.5×
  escalation worst-case is 300s for the timeout path (120 + 180).
  Acceptable given the cycle interval and the alternative
  (silent strategy drop-out + zero observability). Operators can
  disable via `CLAUDE_CLI_MAX_RETRIES=0` if a deployment needs
  bounded latency over reliability.
- **Multi-inheritance `ClaudeTimeoutError` is subtle** — future
  callers writing `except ClaudeError` might be surprised that a
  catch on `StrategyError` also fires. The MRO is documented and
  test-locked, but worth keeping on the radar when adding new
  exception subtypes in `src/ai/exceptions.py`.
- **`ProposalEngine.activity_log` Optional means missing emit if
  caller forgets** — the `None` default is backward-compat-preserving
  but does mean a caller wiring up a fresh `ProposalEngine` without
  threading an `ActivityLog` will silently lose `LLM_TIMEOUT`
  events. `build_engine` does it correctly today; the contract is
  "if you want timeline events, pass an `ActivityLog`."
