## Work Summary

Phase 10.3 closes the "no numbers" gap on `docs/baselines.md` by shipping
the operator script that fetches historical OHLCV from Binance, runs the
existing `Backtester.run_for_strategy` per baseline, runs
`PerformanceAnalyzer`, and persists artefacts under
`data/backtest/baselines/<strategy>/`. This is operator tooling, not a
service — the script is invoked by hand whenever the user wants to refresh
the reference numbers. Phase 9.3's multi-TF backtester is the engine doing
the actual work; 10.3 is purely the glue.

`scripts/backtest_baselines.py` (620 lines) does the fetch + run + persist
flow. The non-obvious piece is pagination: Binance's klines endpoint caps
at ~1500 candles per call, but the swing baselines need 2160 candles
(3 months × 1h) and `rsi_15m` needs 2880 (1 month × 15m). `BaseExchange.get_ohlcv`
doesn't accept a `since` parameter, so the script reaches past the
abstraction into `BinanceExchange._client` to paginate. QA judged this
acceptable — the reach-around is gated to actually-needed cases and
recorded as TECH-DEBT for a future framework extension. Per-baseline
artefacts are `result.json` (raw `BacktestResult` round-trip),
`analysis.md` (PerformanceAnalyzer output with header), and
`summary.json` (compact metric dict for the doc updater). Idempotent
overwrite — re-running replaces previous numbers.

The doc updater is the load-bearing piece. `update_baselines_doc` reads
`docs/baselines.md`, finds the reference-numbers table by regex anchor,
and replaces only the metric cells for baselines whose `summary.json`
exists; period labels are filled in too. `_TBD_` remains for any baseline
the script hasn't run yet, so partial runs don't invent numbers. A
`--no-update-doc` CLI flag gates the doc edit for dry-run / inspection
flows. The script does not run any baselines today — the user will run it
once they've redeployed Fly + the multi-technique fix from 10.6 has
accumulated trade history (live trading numbers), or they can run it
standalone right now to populate just the indicator-baseline numbers
(no live trading needed; only Binance public OHLCV).

The smoke test (`tests/test_scripts_backtest_baselines.py`, 382 lines, 6
tests) covers `render_table` contract, `update_baselines_doc`
TBD-replacement, the short-window fetch path, full `run_all` end-to-end
with the artefact layout + `summary.json` schema + `BacktestResult`
round-trip + analysis.md header, idempotent overwrite, and
`--no-update-doc` gating. Mocks the exchange so the test doesn't depend
on Binance availability.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Reach past `BaseExchange.get_ohlcv` into `BinanceExchange._client` for pagination | The framework's `get_ohlcv` doesn't accept a `since` parameter; the swing baselines need 2160 candles (3mo × 1h) and `rsi_15m` needs 2880 (1mo × 15m), both exceeding Binance's ~1500 cap. Adding `since` to the abstraction is real framework work that this operator-tooling cycle isn't scoped for. The reach-around is gated to actually-needed cases and recorded as TECH-DEBT. |
| Persist `summary.json` alongside `result.json` + `analysis.md` | The doc updater needs a compact metric dict it can read without re-loading the full `BacktestResult`. Keeping the three artefacts side-by-side means `result.json` stays the canonical raw output (round-trippable) while `summary.json` is the doc-updater contract. |
| Idempotent overwrite — re-runs replace previous numbers | The whole point of the script is "refresh the table". Append-only would force the user to manage staleness themselves. The script writes are atomic per artefact directory, so a partial failure leaves the previous run intact for the baselines that didn't run. |
| `--no-update-doc` flag gates the `docs/baselines.md` edit | Operators may want to dry-run the script (or run a single baseline for diagnosis) without modifying the doc. Default behaviour is to update — the common case. |
| Metric cells stay `_TBD_` until operator runs the script | Dev was disciplined here: no synthesised numbers, no faked baselines. The script is shipped; the numbers come from operator action. The doc explicitly tells the user how to populate them. |
| No ADR needed | Operator script following existing project patterns (one-off scripts under `scripts/`, idempotent file outputs under `data/`, doc-updater pattern from prior session-log generators). No new design ground broken. |

## Potential Risks

- **Reach-around into `BinanceExchange._client`** — the script paginates
  by accessing the underlying ccxt client because
  `BaseExchange.get_ohlcv` doesn't accept `since`. If a future phase
  extends the abstraction (real framework work, not this cycle's scope),
  the script should drop the reach-around. Recorded as DEBT-004 follow-up.
  Acceptable today because the reach is gated to the actually-needed
  multi-page fetch case.
- **Doc updater anchored on a regex** — if `docs/baselines.md`'s
  reference-numbers table moves or changes shape, the regex anchor will
  fail to match. The script raises rather than silently writing nothing,
  so failures are visible. Mitigation: the smoke test pins the anchor
  contract.
- **mypy nit at lines 241/248** — `seen: set[int]` vs `seen.add(candle[0])`
  where `candle[0]` is typed `float`. Runtime is correct because ccxt
  returns int ms; the type system doesn't know that. One-line fix
  (`seen.add(int(ts))` or retype). Recorded as DEBT-004.
