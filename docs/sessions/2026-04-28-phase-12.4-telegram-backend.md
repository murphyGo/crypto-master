## Work Summary

Phase 12.4 closes the "single push backend" carry-forward that the
Phase 11 cross-check flagged twice. Phase 11.3 shipped Slack via
incoming webhook; Phase 10.1's "notification redundancy for live
mode" follow-up listed Slack OR Telegram OR email as candidates and
Telegram is the second logical addition (also webhook-style, also no
OAuth dance, easier setup than email infrastructure). With both
Slack and Telegram in place, an operator without a Slack workspace
finally has a push channel for live mode.

`TelegramNotifier` mirrors `SlackNotifier`'s shape one-for-one — same
stdlib `urllib.request.urlopen` + `asyncio.to_thread` pattern (zero
new dep), same per-channel failure-isolation contract owned by the
dispatcher (Phase 6.3), same opt-in activation (silent / not
registered when env vars are missing). The on-the-wire content of
Slack and Telegram alerts is kept in sync via `_build_telegram_text`
collapsing Slack's two-block payload into one Markdown string —
operators reading either backend get the exact same fields
(proposal_id, technique, SL, TP, qty, leverage, rr).

The non-obvious piece is the activation gate: Telegram needs **both**
`TELEGRAM_BOT_TOKEN` and `TELEGRAM_CHAT_ID` — partial config is a
deploy mistake, not an intentional configuration. `build_engine`
checks `if settings.telegram_bot_token and settings.telegram_chat_id`
and silently skips the notifier construction otherwise. The three
partial-config scenarios (token-only, chat-id-only, neither) are
explicitly covered by `test_telegram_notifier_silent_when_either_missing`
to lock the silent-on-partial-config contract.

`__repr__` masks both the bot token AND the chat id — the chat id is
treated as a secret on its own because it identifies the operator's
destination channel; anyone with the chat id can drive a payload at
that channel via any compromised bot. This is a tighter contract than
Slack (which only redacts the URL), reflecting the multi-secret shape
of the Telegram Bot API. No log line ever emits either value;
`build_engine` logs presence only ("Telegram push notifier enabled.").

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Stdlib `urllib.request` + `asyncio.to_thread` (no new dep) | Mirrors `SlackNotifier`'s pattern from Phase 11.3 exactly. Zero new dependency, the dispatcher's event loop is not blocked, and the same shape across both backends keeps the maintenance cost flat. |
| Both `TELEGRAM_BOT_TOKEN` and `TELEGRAM_CHAT_ID` required for activation; partial config silent | Half-configured deploys are deploy mistakes, not intentional configurations. Failing loud at runtime would crash a Fly machine on boot for what's recoverable; failing silent would mask the misconfiguration entirely. The chosen shape (both required, both missing → silent) matches Slack's `Optional[str] = None` shape and is explicitly tested via `test_telegram_notifier_silent_when_either_missing`. |
| `__repr__` masks BOTH token AND chat id (Slack only masks URL) | Tighter contract than Slack reflects the multi-secret shape of Telegram's Bot API. The chat id is sensitive on its own because anyone with it can drive a payload at that channel via any compromised bot. Log lines never emit either value; `build_engine` logs presence only. |
| Single Markdown string (collapsed Slack two-block) via `parse_mode=Markdown` | Telegram has no "blocks" concept. Collapsing both into one Markdown string with a bolded headline + code-fenced detail keeps the on-the-wire content of Slack and Telegram alerts in sync — operators reading either backend get the exact same fields (proposal_id, technique, SL, TP, qty, leverage, rr). Easier to maintain a single payload spec than to diverge across backends. |
| `send` does not swallow `HTTPError` | Mirrors Phase 11.3 `SlackNotifier`'s deliberate "dispatcher owns failure isolation" contract from Phase 6.3. Locked by `test_telegram_http_failure_does_not_crash_dispatch`. |
| Tests cover all 3 partial-config scenarios (token-only, chat-id-only, neither) | The activation gate is `bool(token and chat_id)` — three orthogonal failure modes warrant explicit coverage so a future refactor that breaks the gate (e.g. flipping to `or`) gets caught in red. |
| Telegram message format kept in sync with Slack's | One payload spec across both backends keeps the maintenance cost low. If the spec ever diverges deliberately, the in-sync invariant should be re-evaluated explicitly. |

## Potential Risks

- **Bot token + chat id are both credentials, not just config** —
  anyone with the token can drive the bot, and anyone with the chat
  id can identify the operator's destination channel. Codebase treats
  both as secrets (`__repr__` masks them, `build_engine` logs presence
  only) but operators must know not to commit `.env` or paste either
  value into PRs. Same operational discipline as Phase 11.3's Slack
  webhook URL, with the additional surface area of two secrets per
  backend instead of one.
- **Single payload spec across Slack + Telegram** —
  `_build_telegram_text` and `_build_slack_payload` are kept in sync
  by hand. A cosmetic format tweak that lands in only one of them
  silently diverges the operator-visible content. The intent is "one
  payload spec, two backends"; if a deliberate divergence is ever
  warranted (e.g. Telegram-only Markdown link to a dashboard),
  the spec invariant should be re-asserted explicitly in both
  builders.
- **Partial-config silence is invisible** — if an operator sets
  `TELEGRAM_BOT_TOKEN` but forgets `TELEGRAM_CHAT_ID` (or vice
  versa), nothing surfaces; the notifier just isn't registered.
  Silent failure is the right call (loud crash on boot would be
  worse), but it does mean an operator expecting Telegram alerts and
  not getting them needs to check `build_engine`'s INFO log line as
  the diagnostic surface. `docs/deployment.md` calls this out.
