## Work Summary

Phase 17.2 closes DEBT-019 — the 9-hour-hang failure mode the
operator hit on 2026-04-30 when the first real run of `python -m
scripts.auto_research_candidates --picks 5` produced exactly one
candidate (`donchian_turtle_system_2_20260430_002157.md`) and then
looped forever invoking Claude per-bar. The candidate file itself
was well-formed (frontmatter + falsifiable hypothesis + Donchian
55/20 rules), but `frontmatter.technique_type: prompt` meant the
backtester invoked Claude per-bar expecting strict JSON, and the
generated body lacked the `## Output Contract` block that
`chasulang_ict_smc.md` carries. Claude returned conversational text,
`_extract_json` failed, the engine logged "Claude response did not
contain parseable JSON" — and continued, bar after bar, producing
1.5 MB of warning output without ever advancing the cursor or
erroring out. Phase 17.2 ships DEBT-019's **Options A + C** (the
minimum to unbreak the workflow); Option B (code-type steering for
deterministic catalog picks) is deferred to Phase 17.3 as the
cleaner long-term path.

The senior-developer's primary delivery is a runtime contract
injection plus a backtest circuit breaker. **Option A** —
`src/ai/improver.py::_build_new_idea_prompt` gains a new
`_new_idea_output_contract()` instruction block requiring the
generated `prompt`-type technique body to include a `## Output
Contract` section carrying the chasulang JSON schema verbatim
(`signal: long|short|neutral`, `entry_price`, `stop_loss`,
`take_profit`; one JSON object per bar; no prose around the fenced
JSON block). The instruction is **only** injected into
`_build_new_idea_prompt` — `_build_user_idea_prompt` and
`_build_improvement_prompt` deliberately untouched per Phase 17.1's
Issue-4 deviation (user-idea anchors on the user's text;
improvement is failure-mode analysis on an existing strategy whose
`original_source` already carries the contract). Pinned by two
regression-guard tests in the new `TestNewIdeaOutputContract`
class. **Option C** — `src/backtest/engine.py::Backtester.run`
(single-TF) and `_run_multi_timeframe` (multi-TF) both gain a
per-bar `asyncio.wait_for(...)` timeout wrapping the
`strategy.analyze(...)` call plus a consecutive-failure counter
that increments on `ClaudeParseError` / `StrategyExecutionError` /
`StrategyLoadError` / `asyncio.TimeoutError`. When the counter hits
`max_parse_failures` (default 5), the run aborts via a new
`BacktestAbortedError(reason, candle_index)` propagating through
`Backtester.run_for_strategy` → `RobustnessGate` → the existing
`FeedbackLoop._run_cycle` exception handler at
`src/feedback/loop.py:529`, landing the candidate as
`LoopStatus.ERRORED` with `decision_reason` carrying the abort
reason. A single non-error bar resets the counter so a flaky one-off
doesn't trip the breaker. `BacktestAbortedError` is exported from
`src/backtest/__init__.py`. The 9-hour-hang failure mode is now
bounded to ~minutes on the structural failure path.

The implementation refinement worth flagging: ``StrategyValidationError``
(a ``StrategyError`` subclass that semantically means "data not ready
yet") is caught **separately** and skipped without incrementing the
breaker counter. Otherwise any strategy whose internal warmup floor
exceeds ``BacktestConfig.warmup_candles`` (default 20) would trip
the breaker on bar ~20 — `rsi_universal`'s `period * 3 = 42`
warmup floor is the immediate exemplar; the breaker would have
fired at bar 25 (5 consecutive validation errors past the 20-bar
warmup) every time `rsi_universal` ran through `Backtester`. That's
a footgun, not a circuit breaker. The skip-only refinement is
defensive (cheap, no harm), but it papers over a deeper contract
mismatch — recorded as DEBT-021 (Medium) for the long-term
`BaseStrategy.minimum_candles` declared-floor fix that lets the
engine compute `max(warmup_candles, strategy.minimum_candles)` as
the effective floor.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Ship A + C; defer B to Phase 17.3 | A + C is the minimum to unbreak the auto-research workflow. A (mandatory `## Output Contract` block in generated `prompt`-type bodies) directly fixes the broken-output failure mode — the next generated candidate carries the schema chasulang ships with. C (per-bar timeout + N-consecutive-parse-failures circuit breaker) is the safety net that bounds any future contract drift to ~minutes instead of ~hours. Both are independent, both are testable in isolation, both can land without touching the deterministic-catalog-picks code path. B (code-type steering — generate Python `BaseStrategy` subclasses for deterministic picks instead of `prompt`-type markdown) is the cleaner long-term fix for the 5+ deterministic picks in `TOP_PICKS` (Donchian, Supertrend, Z-score, Larry Williams, NR7), but it's a meatier change requiring a new `Pick.steering` flag and a new code-generating prompt branch. Splitting reduces blast radius and keeps Phase 17.2 focused on "stop the bleeding". |
| `StrategyValidationError` skip-only refinement (does NOT increment breaker counter) | Otherwise any strategy whose internal warmup exceeds `BacktestConfig.warmup_candles=20` (e.g. `rsi_universal`'s `period * 3 = 42` floor) would trip the breaker on bar ~25 — 5 consecutive validation errors past the 20-bar warmup, every run, deterministically. That's a footgun, not a circuit breaker. The skip respects the validation error's semantic meaning ("data not ready yet, try the next bar") while keeping the breaker focused on the genuine contract-failure modes (`ClaudeParseError`, `StrategyExecutionError`, `StrategyLoadError`, `asyncio.TimeoutError`). The skip is defensive and cheap, but it papers over a deeper contract mismatch — `BaseStrategy.minimum_candles` (or `TechniqueInfo.min_warmup_candles`) is the declared-floor fix that lets the engine compute `max(warmup_candles, strategy.minimum_candles)` as the effective floor. Tracked as DEBT-021 (Medium). |
| `per_bar_timeout` default 60 → 600 (DEBT-020 same-cycle fix) | The 60s default that initially shipped derived from a stale dev-plan rationale ("240s ceiling, multi-bar amortised") — actual values are 480s per-`analyze()` (chasulang frontmatter line 10) and the override is per-call NOT amortised (`src/strategy/loader.py:226-232`). 60s would have aborted any chasulang backtest on the first slow bar. 600 = 480 + 120s headroom for parsing/validation/disk I/O. | `connect()` / `disconnect()` lifecycle in `scripts/auto_research_candidates.py` recorded as session-log line item, not a new DEBT entry | The change is behaviorally correct (the script was leaking the http client; previously errored "Not connected. Call connect() first." during the DEBT-019 hung-Phase-B session), well-tested (12/12 in `tests/test_scripts_auto_research_candidates.py` still pass), and was the operator's own fix during the original session that birthed DEBT-019 — defensible scope-drift, not a stealth feature. The cleaner alternative (path b — DEBT-024 "scope-drift connect/disconnect added without dedicated sub-task", Resolved same cycle) would be mostly traceability noise; the session log is the right traceability surface. |

No new component, no new architectural seam, no new constraint that future work must respect. ADRs are reserved for component-boundary or constraint decisions; this is a contract hardening + safety-net add. |

  represents 50% Claude API spend wasted. Low-frequency in practice
  (the immediate DEBT-019 failure mode is "all bars fail", which
  the consecutive counter handles), but the asymmetry argues for a
  secondary cumulative-rate guard when the auto-research workflow
  next exercises a strategy with intermittent parse-failure
  behaviour.

  `_build_improvement_prompt` deliberately doesn't re-inject the

  breaks the JSON schema match, the resulting refined strategy
