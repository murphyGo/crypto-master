## Overview

## Work Summary

Phase 20.1 closes DEBT-024 (High) — the leverage double-application
the 2026-04-30 3-agent comprehensive audit surfaced.
`Backtester.calculate_position_size` already returns a leverage-
neutral notional (margin-adjusted quantity from
`risk / abs(entry - stop_loss)`), but the per-trade PnL writer in
`_close_trade` then multiplied the trade's PnL by `leverage` again,
so every recorded PnL double-counted the multiplier.
`Portfolio.calculate_unrealized_pnl` carried the same shape on the
unrealized side; `PaperTrader.close_position` recorded realized PnL
without the second multiplication, so the convention diverged across
the two ledgers and operator comparison ("did Fly's results match
the baseline?") was broken. Phase 20.1 picks the canonical shape —
leverage already baked into position size, paper-trader's existing
shape is the reference — extracts a single helper
(`pnl_for_trade(entry, exit, qty, side)`) into a new
`src/utils/trading_math.py` module, and routes every PnL site
through it. The mathematical change is one removal in two files;
the discipline change is "every PnL site goes through the helper".

The cycle was driven by the 3-agent audit's High-priority finding
and the planner's Phase 20 spec; team-lead picked 20.1 as the
self-contained unit (helper + math fix + symmetry refactor on the
already-correct paper-trader path) so 20.2 can layer the
docstring sweep / dashboard prose / broader grep audit on top of a
green math base. The dev's `pnl_for_trade` ships as a side-dispatched
helper: longs compute `(exit - entry) * qty`, shorts compute
`(entry - exit) * qty`; `qty` already reflects the levered notional
from `calculate_position_size`, so leverage is deliberately **not**
a parameter — making it a parameter would invite a future caller to
pass it again and reintroduce the bug. The helper's module-level
unit tests pin both signs against fixed numeric fixtures (long
profit / long loss / short profit / short loss / zero-pnl
boundary / decimal precision / etc.), 11 cases total. The
backtester's `_close_trade` (`src/backtest/engine.py` ~line 948-960)
drops the `* leverage` and routes through the helper;
`Portfolio.calculate_unrealized_pnl` does the same against the
current mark price; `PaperTrader.close_position` was already
correct shape but routed through the helper for symmetry (no
behaviour change — same bytewise output, same test invariants).

Quant-trader-expert review during the cycle surfaced one in-scope
extension: `src/strategy/performance.py::TradeHistory.calculate_pnl`
(lines ~797-839) carried the same `* self.leverage` bug on **both
branches** of its long/short dispatch, and additionally inflated
`pnl_pct` by `leverage` (a more visible operator metric than raw
`pnl` since dashboard cards key off it). The dev absorbed this
extension into 20.1 with the lead's approval — both branches now
drop `* self.leverage` from `pnl`, and `pnl_pct` is reformulated as
leverage-neutral (`(exit - entry) / entry` for longs, with sign
inversion for shorts). Two new persistence-layer tests
(`test_close_trade_persisted_pnl_routes_through_helper` +
`_short`) lock the persisted-record path against the helper's
output, so a future regression that diverges the persistence
convention from the helper fails the suite. The scope extension
absorbs a chunk of what Phase 20.2's "every PnL surface" sweep was
originally going to touch — see the planner reconciliation note
under Phase 20.2 below.

QA verdict: 🟢 Ship. Quant verdict: 🟢 Ship. Code is green
(1226 tests passing, ruff/mypy clean). Cascaded test assertion
updates rippled across `tests/test_paper_trading.py` (8 across 7
methods), `tests/test_portfolio.py` (5 methods), and
`tests/test_strategy_performance.py` (3 `calculate_pnl` methods)
because the prior assertions encoded the inflated PnL — purely
mechanical fixture corrections, no behaviour change in the test
contract, and the new numeric expectations are the correct
post-fix values. Two new fixtures in `test_backtest_engine.py`
(`TestPnLConventionAlignment`, 4 cases — long/short numeric
equality between backtester and paper-trader on a fixed (entry,
exit, qty, leverage) input + 2 originals) lock the cross-ledger
parity that DEBT-024 named as the operator-facing acceptance test.
No ADR — 20.1 extracts a helper into a new module and updates four
callers; the public contract of `Backtester` / `Portfolio` /
`PaperTrader` / `TradeHistory` is unchanged (same signatures, same
return types, same persistence shapes — only the numeric output
shifts to the correct convention).

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Canonical convention: paper-trader's existing shape (leverage already in position size; PnL = `(exit - entry) * qty` with sign-by-side) | Three of the four call sites already used this shape; the backtester and portfolio were the outliers. Aligning to the majority convention minimises code churn and minimises the operator-visible numeric shift (only backtester / portfolio / persistence-layer figures change; live paper-trader history is bytewise-preserved). The alternative (lift everyone to "include leverage at PnL time") would have inverted the bug rather than fixed it — every paper trade since Phase 4.3 would re-encode under the new convention, breaking historical comparison against the existing trade ledger. |
| `leverage` deliberately NOT a parameter on `pnl_for_trade` | `qty` already reflects the levered notional from `calculate_position_size`. Making `leverage` a parameter would invite a future caller to pass it again and reintroduce DEBT-024 verbatim. The helper's signature is the contract: "if you have entry, exit, qty, side, you have everything you need." Documented in the module docstring. |
| Scope extension into `TradeHistory.calculate_pnl` (originally 20.2's territory) | Quant-trader-expert review during the cycle flagged that the persistence-layer branch carried the same bug on both long and short paths AND additionally inflated `pnl_pct` (more visible operator metric than raw `pnl` since dashboard cards key off it). Leaving it for 20.2 would have shipped a half-fix: the math source is correct, but the persisted records still inflated. The lead approved the absorption mid-cycle; the trade-off is a slightly larger 20.1 diff in exchange for a coherent end-to-end fix and a smaller 20.2. See planner reconciliation below. |
| `pnl_pct` reformulated as `(exit - entry) / entry` (long) / sign-inverted (short), leverage-neutral | The original formula was `((exit - entry) / entry) * leverage`, which compounded the bug at the percentage level (5× leverage made a 1% price move read as 5% pnl_pct, which then multiplied through dashboard cards). The leverage-neutral form is the operator's natural reading: "this trade made 1% on the underlying." Operators looking at leverage-amplified equity curves get that signal from the pnl-in-USDT line and from the position-sizing layer, not from `pnl_pct` itself. |
| `PaperTrader.close_position` routed through the helper despite already being correct | Pure symmetry refactor — the four PnL sites in the codebase now share one source of truth, so a future reviewer can grep `pnl_for_trade(` and find every site, rather than greping for `* leverage` and missing the one path that didn't have it. Bytewise-identical output, locked by the existing paper-trading test suite. |
| No ADR | 20.1 extracts a helper into a new utility module and updates four callers. Public contracts of `Backtester` / `Portfolio` / `PaperTrader` / `TradeHistory` are unchanged (same signatures, same return types, same persistence shapes). The numeric output shifts to the correct convention; that's a bug fix, not an architectural decision. ADRs are reserved for component-boundary or constraint decisions; this is a math primitive lifted into a single source of truth. |

## Potential Risks

- **Persisted-history numeric shift on next read** — the
  `TradeHistory.calculate_pnl` extension is a behavioural fix, not
  a refactor. Any operator dashboard / report that reads existing
  paper-trader history under the new code path will see different
  numbers than before for trades whose stored fields drive the
  recompute. The fix is correct; the discontinuity is the point.
  Operators monitoring the dashboard immediately post-deploy may
  see equity-curve / pnl-pct cards shift; the right reaction is
  to verify the new numbers match the stored entry / exit / qty
  expectations under the leverage-neutral convention, not to
  reach for an env override (there is no override — DEBT-024 is a
  bug, not a tunable). Phase 20.3's baseline re-computation will
  formally restate the per-strategy artefacts; the dashboard
  shift is the same fix at runtime.
- **Persistence-layer extension absorbed mid-cycle** — the
  scope extension into `TradeHistory.calculate_pnl` was approved
  by the lead during quant review, but it does mean Phase 20.1's
  diff is wider than the original spec listed. The reconciliation
  here is structural, not semantic: 20.2's grep-audit and
  docstring sweep are reduced (see Phase 20.2 reconciliation
  below), so the total Phase 20 work is unchanged — only the
  sub-task boundary moved. If a future reader is reconstructing
  why 20.1's diff includes `src/strategy/performance.py`, this
  log + the change-history reconciliation row are the audit
  trail.
- **Operator comparison "did Fly match baseline" remains broken
  until Phase 20.3 lands** — Phase 20.1 closes the math source;
  Phase 20.3 re-runs every persisted baseline figure under the
  corrected math. Until 20.3, `docs/baselines.md`'s operator
  table still quotes pre-fix numbers (inflated by leverage). An
  operator running `--picks` against the new code and comparing
  to the old baseline table would see what looks like a
  performance regression but is in fact the bug going away. The
  20.3 priority just rose by one notch as a result.
