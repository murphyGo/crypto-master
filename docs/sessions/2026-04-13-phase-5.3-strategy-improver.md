## Work Summary

Added ``StrategyImprover`` — a single entry point for three Claude-
driven flows:
1. **Improvement** (FR-022): "here's an underperforming technique and
   its recent results; give me a better version".
2. **New idea** (FR-023): "invent a technique from scratch".
3. **User idea** (FR-024): "turn this freeform user text into a
   structured technique".

All three flows funnel through the same pipeline: build a prompt →
call ``ClaudeCLI.complete`` (new method for raw-text completions) →
extract a fenced ``markdown`` block → parse YAML frontmatter →
optionally persist under ``strategies/experimental/``.

Files are named ``{slug}_{utc_timestamp}.md`` so repeat generations
never clobber each other, giving the feedback loop a stable
filesystem-backed audit trail of every candidate Claude produced.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Added `ClaudeCLI.complete()` returning raw stdout instead of forcing `analyze()`'s JSON shape | Technique files are markdown, not JSON. Stuffing an md file inside a JSON wrapper would be fragile parsing and noisy for the user to edit later. A separate method keeps `analyze()` as the strict JSON path while letting text flows stay text. |
| Inject `ClaudeCLI` via constructor with a default factory | Makes testing trivial (`AsyncMock(spec=ClaudeCLI)`) and avoids fighting asyncio for subprocess fakes. Production code still gets a working default. |
| Fenced `markdown` block convention for Claude's output | Claude reliably respects explicit code-fence instructions and the `_MARKDOWN_BLOCK_PATTERN` regex is trivial. Fallback: if the model forgets the fence, the improver treats the whole response as the technique — better graceful than strict. |
| Frontmatter parsed with `yaml.safe_load`; parse errors fall back to the `fallback_name` per flow | The generated content is always preserved verbatim regardless of frontmatter validity. We only read frontmatter for metadata — if it's broken, the file still lands on disk, a human can read/edit it, and the improver doesn't crash. |
| Filename = `{slug}_{timestamp}.md` | Same technique name generated twice should both be kept. Collision-by-name would silently lose Claude's earlier suggestion. Slugging strips anything outside `[A-Za-z0-9_-]` to survive odd model outputs. |
| Empty user-idea text raises `StrategyImproverError` before hitting Claude | Saves a CLI round-trip for a known-bad input and gives a precise error. The typed error distinguishes user-facing config problems from Claude-side failures. |
| `GeneratedTechnique.content` is the extracted block; `raw_response` keeps Claude's full reply | `content` is the file you write; `raw_response` is a debugging breadcrumb. Keeping both lets the feedback loop log exactly what the model said when a candidate later flames out in backtests. |
| `.gitkeep` in `strategies/experimental/` | Directory needs to exist for the improver's relative default path to work on a fresh clone, even though no experimental techniques are checked in. |

## Potential Risks

- **Claude's output format is advisory, not enforced**. If the model
  ignores the "fenced ``markdown`` block" instruction AND emits
  something that's not parseable as a technique at all, the improver
  still returns a `GeneratedTechnique` whose content is the raw
  response. The feedback loop should treat "does the candidate pass
  a loader roundtrip?" as a separate validation gate before
  promoting a file.
- **No deduplication across generations**. Running the same flow
  repeatedly against identical inputs will produce N near-identical
  files in `experimental/`. Phase 5.4 can add a content-hash dedupe
  pass if this becomes noisy.
- **`raw_response` kept on the model means generated techniques
  carry a trailing copy of Claude's full reply in memory**. Not
  persisted to disk, but don't pickle these objects carelessly.
- **`.gitkeep` is checked in but generated files land in the same
  directory and are gitignored elsewhere**: confirm `.gitignore`
  excludes `strategies/experimental/*.md` before committing real
  experimental output.
