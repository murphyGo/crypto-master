## Work Summary

Added the operator-facing view of the runtime engine. The new
`Engine` page reads `data/runtime/activity.jsonl` (Phase 8.1's
output) and surfaces:

- **Summary cards** — total cycles, last cycle started + status,
  avg cycle duration, errored-cycle count, total positions
  opened/closed in the window.
- **Recent Cycles table** — per-cycle aggregation (start, duration,
  status, proposal/open/close counts) for the most recent 25 cycles.
- **Cycle Duration bar chart** — durations of completed cycles in
  chronological order; spikes flag scan/monitor latency issues.
- **Activity Timeline** — newest-first event table with a
  multi-select filter on `ActivityEventType` so the operator can
  zoom in on (e.g.) just `POSITION_*` events.

The cycle-aggregation logic is a pure function (`aggregate_cycles`)
that walks events grouped by `cycle_id`, picks the start/completion
events, and counts proposal/position event types. It tolerates
truncated logs (missing `CYCLE_STARTED`), in-flight cycles
(no completion event yet), and unsorted input. Three pure
DataFrame helpers (`build_summary_metrics`,
`build_cycles_dataframe`, `build_cycle_duration_dataframe`,
`build_timeline_dataframe`) keep the data shaping testable without
Streamlit.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Cycle aggregation lives in the page module, not the engine | The engine writes events; this is a read-side concern. Putting `aggregate_cycles` in `src/runtime/` would couple the engine to a Pandas-shaped output it never produces. The page owns its presentation. |
| `CycleSummary` is a `dataclass`, not a Pydantic model | Read-only DTO that only needs to ride between aggregator and DataFrame builder. No validation needed; dataclass is lighter. |
| `build_cycle_duration_dataframe` skips running cycles | A bar with `None` height is misleading; a missing bar communicates "in flight" honestly. The Recent Cycles table still surfaces the running cycle row. |
| Timeline newest-first; cycle-duration chart oldest-first (chronological) | Two different audiences: timeline reader wants fresh activity at the top; chart reader wants to see how durations evolve left-to-right. Each ordering matches its purpose. |
| Multiselect filter defaults to all event types | "Showing nothing" on first render would surprise the operator. Default to everything so noise reduction is opt-in. |
| Status "running" vs "ok" vs "errored" derived in summary, not stored | Three-way label is presentation-only. Storing it on the dataclass would duplicate state already implicit in `errored` + `completed_at`. |

## Potential Risks

- **No pagination on the timeline**. The page reads the entire log
  and tails the last `DEFAULT_TAIL_LIMIT=300` events for the table.
  At realistic scan cadence (5-min cycles, ~10 events each) the log
  hits 300 events in ~2.5 hours. After that the table covers a
  shrinking window. Carry forward — pagination or `st.cache_data`
  with date-range filter is a logical Phase 9 follow-up.
- **`aggregate_cycles` is O(n²) over events with the same cycle id**
  due to the per-cycle `_first_event` walks. With realistic cycles
  (≤30 events each) this is negligible; flagged here only because
  the log file is unbounded. If it bites, switch to a single-pass
  builder that walks once and accumulates per-cycle state.
- **No live-refresh.** The page only re-reads the log on a manual
  Streamlit rerun (sidebar nav click, browser refresh). For an
  always-on dashboard, an `st.autorefresh`-style helper would make
  the page truly live.
