## Work Summary

Closed the most-explicit deferral the Phase 8 cross-check carried forward: live trading was wired in code but the production entrypoint always built `PaperTrader`. Live-money deployment was blocked.

The Phase 8 plan called this a "small one-file follow-up" â€” that turned out not to be true. The runtime engine was tightly coupled to `PaperTrader`'s sync API, while `LiveTrader` was naturally async (network I/O, confirmation callbacks). Bridging the two required introducing a `Trader` Protocol that both implementations satisfy and converting PaperTrader's open/close methods to async (~50 test call sites updated as a result).

The user explicitly chose option **B** (Trader Protocol) over the easier-but-uglier alternatives (engine-level branching, sync LiveTrader adapter). The result is that `TradingEngine` no longer knows or cares which trader it talks to â€” `src/main.py` picks the implementation based on `Settings.trading_mode` and hands it in.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Trader Protocol with mixed async/sync surface | State-mutating operations (`open_position` / `close_position`) genuinely need async on the live path (network I/O, possibly a confirmation prompt). Read-only helpers (`get_open_trades` / `check_exit_conditions`) query in-memory state on both implementations â€” async would be a paper-tax for no live benefit. Mixed surface is honest about the actual cost model. |
| Made PaperTrader's open/close `async def` rather than wrapping it in an adapter | An adapter class doubles the moving parts (PaperTrader + AsyncPaperTraderAdapter) for marginal gain. The `async def` keyword has zero runtime cost when the body never awaits. Cost is paid in test churn â€” but that's a one-time bill, and the resulting tests more closely model the runtime's actual call shape. |
| `LiveTrader` skips confirmation on `stop_loss` / `take_profit` reasons | The user already authorized those bounds when opening the position. Re-prompting on auto-exit is operator-hostile and breaks the whole "headless production runtime" goal. Manual closes (`reason="manual"`) still go through the callback â€” explicit user intent gets explicit confirmation. This matches the policy `LiveTrader.monitor_positions` already had; we just made `close_position` consistent with it. |
| Live confirmation callback wired to auto-approve in production | The engine's `auto_approve_threshold` gate has already filtered the proposal by composite score before the trader ever sees it. Asking again at confirmation time would gate the same decision twice. The auto-confirm shim documents this explicitly so a future operator knows where to swap in `default_confirmation` for interactive sessions. |
| Renamed `Settings.paper_initial_balance` was *not* changed | The field still drives PaperTrader sizing in paper mode. Live mode doesn't read it â€” sizing comes from real exchange balances + `risk_percent`. Kept the field name for backward-compatibility with existing `.env` files; the deployment doc clarifies its meaninglessness in live mode. |

## Potential Risks

- **Live-mode rollback does not auto-close open positions.** Flipping `TRADING_MODE` back to `paper` stops new live entries but leaves existing live positions on the exchange â€” they need manual close. The deployment doc's step 9 calls this out explicitly. A future "graceful live-mode exit" feature could close-or-monitor outstanding positions on shutdown, but it's not in scope here.
- **Auto-confirmation in live mode means the engine's threshold gate is the only gate.** A misconfigured `auto_approve_threshold` (e.g. accidentally 0.0) would auto-open every proposal Claude generates. The deployment doc emphasizes "start with 1.5â€“2.0 and ratchet down" but it's a soft control, not a hard cap. Phase 10.5 candidates: notification dedup or a per-cycle position cap.
- **PaperTrader's async conversion is technically a breaking API change.** Any external code that imported `PaperTrader` directly and called `open_position` / `close_position` synchronously will need to be updated. Today there are no external consumers â€” the only code that touches PaperTrader directly is `src/main.py`, the tests, and ApplicationName-internal modules â€” all updated. Worth flagging in the next release notes if a public API stabilizes.
- **`_engine_auto_confirmation` accepts a `position: object`** rather than `Position` â€” chosen for typing flexibility (the callback signature in `LiveTrader` is `Callable[[Position, str], Awaitable[bool]]` but our shim doesn't actually need the `Position` type). Mostly cosmetic; flagged so a future tightening pass knows the option exists.
- **No live integration test runs against a real exchange.** All "live" testing is unit-level with mocked clients. The first live deploy is the first real validation. Phase 10.5/10.6 candidates include adding a one-off `scripts/live_smoke.py` against a tiny balance.
