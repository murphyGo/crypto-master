## Work Summary

Closed the offline gap for multi-TF strategies: `Backtester` now has a `run_multi_timeframe` method, `RobustnessGate` threads multi-TF data through every gate that needs it, and `FeedbackLoop` carries `ohlcv_by_timeframe` end-to-end. `chasulang_ict_smc` (and any future multi-TF technique) can now go through baseline backtest → OOS / walk-forward / regime / sensitivity gates → `AWAITING_APPROVAL` — the full feedback-loop pipeline that was blocked after Phase 9.1.

The existing single-TF code paths are unchanged: `Backtester.run` keeps its signature, all 58 prior backtest-engine tests still pass untouched, and the 9.2 baselines / sample strategies / `chasulang` (offline) all backtest exactly as before. Multi-TF is opt-in via `strategy.info.requires_multi_timeframe`, dispatched through a new `run_for_strategy` shim.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Separate `run_multi_timeframe` instead of overloading `run` | Mutually-exclusive `ohlcv: list[OHLCV] \| dict` signature is gross — keeps the type checker honest and the docstring readable. The `run_for_strategy` dispatcher is the polymorphic entry point. |
| `run_for_strategy(strategy, ohlcv, ..., *, ohlcv_by_timeframe=None)` as the dispatcher | Single call shape for the loop and validator; they don't need to branch on `strategy.info.requires_multi_timeframe`. Raises early when a multi-TF strategy has no dict — fail-fast at the call site, not after a half-completed run. |
| Bisect-based per-TF cursor instead of naive `[c for c in higher if c.ts <= cur_ts]` | Naive slicing is O(N·M) per-step → O(N²·M) total. Bisect is O(log M) per step. For 1y of 5m primary (≈105k bars) × 4 TFs the difference is hours vs seconds — worth doing right the first time. |
| Warmup gates every TF, not just primary | ICT/SMC top-down analysis is meaningless without a full higher-TF context window. A 4h slice of 1 candle isn't "macro structure" — it's noise. Same `config.warmup_candles` applied to every TF. Per-TF warmup config is a noted follow-up. |
| Primary timeframe derived from caller's `timeframe` arg, not `strategy.info.timeframes[-1]` | Validator gates already pass a `timeframe` string through every layer. Reusing it as the primary TF in multi-TF mode keeps the call shape uniform. The `[-1]` convention applies upstream (engine populates `timeframe` from `strategy.info.timeframes[-1]`). |
| `BacktestResult.timeframe` stores the primary TF as a single string | Same convention as Phase 9.1's `Proposal.timeframe`. Downstream consumers (PerformanceTracker, analyzer, dashboard) all read `.timeframe` as a string — no schema drift. Multi-TF metadata is a feature-creep risk; defer until something actually needs it. |
| `_gate_regime` unchanged | The regime gate operates entirely on the baseline trades' entry timestamps + primary OHLCV's SMA. Higher TFs don't enter the calculation. Touching it would be no-op churn. |

## Potential Risks

- **Multi-TF backtests are slower than single-TF** by a constant factor — the per-step bisect is cheap but every analyze call now serializes a per-TF dict. For chasulang (4 TFs × Claude API call) the bottleneck is still the LLM, not the slicing. For deterministic multi-TF code strategies (which don't yet exist) the overhead is negligible.
- **The sensitivity gate's `strategy_factory` is a strategy-author concern** — when sweeping params for a multi-TF strategy, the factory must produce strategies whose `info.requires_multi_timeframe` matches the original. Not enforced by the framework. Documented in the gate's docstring.
- **`Proposal.timeframe` and `BacktestResult.timeframe` both use single strings** for the primary TF. If a future feature needs to surface "this strategy used these 4 TFs" in the dashboard, a separate metadata field will be needed. Not blocking 9.3.
- **No live engine validation against real Claude yet** — the integration is correct in unit tests but a Claude-backed multi-TF backtest hasn't been run against historical OHLCV. Manual validation should be done before promoting any chasulang variant to active.
