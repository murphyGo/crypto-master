## Work Summary

Phase 18.1 closes the first of the live-trading-quality defects the
2026-04-30 production review of `/data/trades/paper/trades.json`
surfaced (1W/8L over the post-redeploy window, -78.50 USDT,
EV -8.73/trade). The smoking-gun trade is
`5d51cba3-900f-4415-a401-096df391860a` — ETH long, proposal
`6ef8c07e...` — created at `14:43:21` with `entry=2323` /
`SL=2305`; chasulang/Claude CLI took **3 min 13 sec** to return;
auto-approval + fill at `14:46:34` recorded the position at the
stale `entry=2323` even though live had already crossed past the
SL by then; the position closed `0.48s` later at `2300`. The
runtime engine (`src/runtime/engine.py::_execute` →
`_proposal_to_position`) was copying `proposal.entry_price` into
the opened `Position` with no live-price sanity check, so any LLM
latency spike turned into a guaranteed-loss fill at a price the
market had already moved through. Phase 18.1 inserts a sanity gate
between auto-approval and `trader.open_position`: fetch a fresh
ticker, reject the fill if live has crossed the SL or drifted
beyond a configurable slippage tolerance, otherwise fill at
`proposal.entry_price` exactly as before (no silent switch to live
price — that would defeat the proposal's R/R math). Ticker fetch
failure falls back to fill so a transient exchange hiccup doesn't
silently disable trading; the WARN log is the operator's signal.

The cycle was driven by the user's direct brief on the negative-PnL
/ 11% win-rate Fly run; team-lead's diagnosis identified the
instant-stopout class of losers (positions closing at SL within
seconds of fill) as the dominant failure mode and traced the root
cause to chasulang/Claude CLI proposal-to-fill latency. The fix
ships in `src/runtime/engine.py` as a new `_stale_quote_gate`
helper called from `_execute` before `trader.open_position`. Two
defaults are deliberately conservative: `fill_slippage_tolerance`
defaults to `Decimal("0.005")` (50 bps) and
`reject_if_past_stop_loss` defaults to `True`, so the smoking-gun
bug closes without an env flip. Env overrides
`ENGINE_FILL_SLIPPAGE_TOLERANCE` and
`ENGINE_REJECT_IF_PAST_STOP_LOSS` follow the Phase 10.2 / 13.2
pattern; `Settings.engine_fill_slippage_tolerance: Decimal` and
`Settings.engine_reject_if_past_stop_loss: bool` parity-asserted
against `EngineConfig` defaults via the existing
`test_settings_defaults_match_engine_config` style. The past-SL
check is side-dispatched off `proposal.signal` (longs:
`live <= SL`; shorts: `live >= SL`); the slippage check is
symmetric (`abs(live - entry) / entry > tolerance`). Order matters:
past-SL runs first (more specific signal), only then the slippage
check.

The rejection activity event includes the existing
`_proposal_summary` payload plus three explicit numeric fields
(`proposal_entry`, `proposal_stop_loss`, `live_price`) and a
computed `drift_bps` for post-mortem reconstruction. The rejection
also overwrites the `ProposalRecord` decision to `REJECTED` (load
+ `model_copy` + save) so dashboard / post-mortems reading
`ProposalHistory.list_all(decision=REJECTED)` see the stale-quote
rejections with structured fields. This is the first rejection
path that rewrites the proposal record — Phase 12.1's cap
rejection only emits an activity event without a record rewrite,
so two rejection paths now follow two persistence patterns;
recorded as a Medium tech-debt for unification (see DEBT-015).
Quant validated the 50-bps default against 1H BTC/ETH expected
drift over a 4-min latency window (~26 bps ≈ 2σ at 50 bps),
confirmed the smoking-gun ETH case (`drift = abs(2300 - 2323) /
2323 ≈ 99 bps` past SL by `15 bps`) is comfortably caught with
49 bps headroom, and flagged that `bollinger_band_reversion` will
see a visible rejection-rate uptick under the new gate (intended
behaviour, not a regression — the gate is doing its job on a
strategy that proposes near-edge fills).

QA verdict was 🟡 ship-with-note. Code path is correct; four
non-blocking observations were filed as tech-debt for follow-up
cycles (semantic divergence between rejection paths, `CycleResult`
counter contract docs, activity-event field redundancy, test-suite

(`test_proposal_executes_when_cap_not_reached`,
`test_cap_counts_only_matching_symbol`) whose **inverted-SL

No ADR — 18.1 inserts one helper into an existing component

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Conservative defaults — `reject_if_past_stop_loss=True`, `fill_slippage_tolerance=0.005` (50 bps) — on by default | The smoking-gun ETH bug is a real-money loss on a proposal that crossed its own SL before fill. The gate has to close that path without an operator opt-in step, otherwise the fix isn't a fix in production. 50 bps is conservative enough to catch the smoking-gun case (99 bps drift past SL — 49 bps headroom) and tolerant enough to absorb legitimate fast-market drift over a 4-min latency window (~26 bps expected at 1H BTC/ETH ≈ 2σ at 50 bps per quant validation). The tolerance is env-overridable for operators who want to tighten / loosen per strategy class — `bollinger_band_reversion` will see a visible rejection-rate uptick at the default, which is intended behaviour, not a regression. |
| No silent switch to live price on near-tolerance fills — fill stays at `proposal.entry_price` | The proposal's composite score, R/R math, position sizing, and SL/TP distances are all predicated on `entry_price`. Mutating entry at fill would corrupt every downstream metric (risk-per-trade, RR, P&L attribution) and turn the proposal record into a fiction. The gate is a binary: either the live price is close enough to fill at the proposed entry, or the proposal is rejected. No "fill at live + adjust SL/TP proportionally" — that's a rebalance, not a fill, and it would obscure the fact that the proposal was stale. |
| Past-SL check runs first, then slippage check | More specific signal first. Past-SL is the unambiguous guaranteed-loss case (live has crossed the proposal's stop, so filling at proposed entry plus the original SL would eat the entire risk budget on a single tick); slippage is the precautionary "live moved more than we're comfortable with" case. The past-SL `decision_reason="stale_quote_past_sl"` is more actionable than the generic `"slippage_exceeds_tolerance"` for post-mortem triage, so it should win when both fire. |
| Ticker fetch failure → fall through to fill, log WARN | The alternative (fail-closed: reject when ticker fetch errors) would have a transient exchange hiccup silently disable trading. Logging WARN with a structured `stale_quote_check_failed` reason gives operators the signal to investigate without locking the engine. Aligns with the established "exchange errors are transient by default" pattern from Phase 11.2 / Phase 12.3. |
| Rejection rewrites `ProposalRecord` (load + model_copy + save) | Dashboard / post-mortems reading `ProposalHistory.list_all(decision=REJECTED)` need the stale-quote rejections to show up as `REJECTED` with structured fields — that's the operator-visible audit trail. Phase 12.1's cap rejection took the lighter "activity event only" path; this divergence is recorded as DEBT-015 (Medium) for follow-up unification (probably converging on the rewrite, since it preserves a single source of truth for the decision). |
| No ADR | 18.1 inserts a helper into an existing component between two existing call points and adds two `EngineConfig` fields with the established Phase 10.2 / 13.2 env-override pattern. The `Trader` Protocol contract is unchanged. ADRs are reserved for component-boundary or constraint decisions; this is a guard inserted into an existing seam, not a new seam. |

## Potential Risks

- **`bollinger_band_reversion` rejection-rate uptick** — quant
  flagged that the strategy's near-edge fill pattern means it will
  see a visibly higher rejection rate under the new gate at the
  50-bps default. This is the gate doing its job, not a
  regression: a strategy that proposes fills near the edge of a
  fast-moving range *should* be rejected when live has moved past
  the edge by the time the LLM returns. Operators monitoring
  Fly logs after redeploy may see the uptick and reach for the
  env-override; the right reaction is to look at the strategy's
  proposal-to-fill latency and the rejection's `drift_bps`, not
  to widen the tolerance reflexively. The gate is the floor;
  loosening it should be the last lever, not the first.
- **Rejection-path semantic divergence (DEBT-015)** — Phase 18.1's
  rewrite of `ProposalRecord` to `REJECTED` is the right shape for
  operator-visible audit, but it diverges from Phase 12.1's
  activity-event-only pattern for cap rejections. Dashboard /
  post-mortems reading
  `ProposalHistory.list_all(decision=REJECTED)` will see
  stale-quote rejections but miss cap rejections. The fix is
  cheap (apply the rewrite to the cap path), but the divergence
  is a live inconsistency until then and could mislead an
  operator running a "show me all rejected proposals" query.
- **Counter-semantics gap (DEBT-016)** — when stale-quote or cap
  rejection fires, `CycleResult.proposals_accepted` and
  `proposals_rejected` both increment for the same proposal (the
  composite gate accepts; the post-acceptance gate rejects).
  Cycle summary log `accepted+rejected` will not sum to
  `proposals_processed`. The rejection paths are correct; the
  documentation is what's missing — operators reading the cycle
  summary may mis-attribute "where did the proposal go" until the
  contract is documented in the field docstrings.
- **Test-suite gap on simultaneous-counters assertion (DEBT-018)**
  — the 4 new rejection tests don't assert
  `result.proposals_accepted == 1`. The simultaneous-counters
  contract isn't locked in by the suite, so a future regression
  that drops the accept-counter increment on the rejection path
  wouldn't be caught. One-line follow-up.
- **Operator Fly redeploy still owed** — the gate is unit-tested
  with mocked exchange tickers, but no real-world run has
  exercised it on Fly yet. The redeploy + log monitoring is the
  next operator action (this cycle, after commit). Without it,
  the smoking-gun ETH case isn't proven closed in production.
  Highest-priority operator action for the next cycle.
