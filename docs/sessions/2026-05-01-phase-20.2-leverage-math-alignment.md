## Work Summary

Phase 20.2 finishes the DEBT-024 follow-up that Phase 20.1 left open
after the persistence-layer math extension was absorbed mid-cycle.
20.1 fixed every PnL math site (backtester `_close_trade`, portfolio
`calculate_unrealized_pnl`, paper-trader `close_position`,
`TradeHistory.calculate_pnl` both branches + `pnl_pct`) and routed
all four sites through the new `pnl_for_trade(...)` helper. 20.2
turns the runtime fix into a discipline lock: a `grep -rn "leverage"`
audit across `src/backtest/`, `src/trading/`, `src/strategy/` to
confirm no other `* leverage` site exists outside the four
already-routed callers; field-level docstring sweep on every PnL
field naming the leverage-neutral convention so future contributors
don't reintroduce the bug; and a regression-guard test that fails
the suite if a future commit adds a raw `* leverage` line to any
PnL math file. Mathematically the cycle is a no-op — green stays
green, no numeric fixture moves. The value is in the audit trail
(grep table) + the contractual docstrings + the regex guard pinning
the convention forward.

The grep audit produced an 8-row table: 4 margin / position-sizing
sites kept (`calculate_position_size` and friends in
`src/backtest/engine.py` + `src/trading/portfolio.py` + risk math
modules) where `* leverage` is the correct shape (margin = notional
/ leverage; capital deployment scales with leverage), and 4 PnL
sites confirmed already routed through `pnl_for_trade(...)` per
20.1. No missed `* leverage` anywhere in the PnL surface — the
sweep's outcome is the verification the spec called for, with zero
hits requiring action. The docstring sweep added convention prose
to `AssetSnapshot.unrealized_pnl`, `Portfolio.unrealized_pnl`
(field listings), `TradeHistory.pnl`, `TradeHistory.pnl_percent`,
and `Position.calculate_pnl` (method) — each docstring states the
leverage-neutral convention explicitly and points at the helper
("PnL is computed against leveraged notional via `pnl_for_trade`;
do not re-multiply by `leverage` downstream"). The regression-guard
file `tests/test_leverage_pnl_no_double_apply.py` ships 5 tests:
4 file scans (one per PnL math file) using a regex that flags any
`pnl.*\\* *leverage` or `\\* *self\\.leverage` pattern outside the
4 known correct margin sites, plus 1 self-test of the guard regex
against a synthetic positive (the regex itself does what the test
file claims it does — meta-test, prevents the guard from rotting
silently into a no-op).

Both reviewers returned 🟡 ship-with-note. Quant-trader-expert
noted the regression-guard regex is text-pattern based, so an
indirect alias (`mult = leverage; pnl *= mult`) would slip past it.
The lead addressed this with a module docstring on
`tests/test_leverage_pnl_no_double_apply.py` acknowledging the
limitation explicitly: the guard catches direct `* leverage`
reintroduction (the historical bug shape from DEBT-024), not
arbitrary aliasing — defence-in-depth alongside the unit tests
that pin numeric output, not a sole gate. QA-reviewer noted two
cosmetic single-line collapses in `src/trading/portfolio.py` at
lines 366 and 381 that crept in via black during the docstring
edit pass — harmless reformats (multi-line argument list collapsed
to single-line where it fit), no behaviour change, no diff to
field semantics; flagged here for audit-trail completeness. Both
notes are addressed in the working tree; gates green
(pytest 1231 passed, ruff/mypy/black clean).

The cycle scope was deliberately narrow because Phase 20.1's
absorption already shipped the persistence-layer math fix
(`TradeHistory.calculate_pnl` both branches + `pnl_pct`

## Potential Risks

- **Regression-guard alias gap (acknowledged)** — the regex-based
  guard catches `* leverage` and `* self.leverage` patterns in the
  four PnL math files but a contributor writing
  `mult = leverage; pnl *= mult` would slip past it. The module
  docstring states the limitation explicitly. Mitigation is
  defence-in-depth: Phase 20.1's `TestPnLConventionAlignment`
  pins numeric output (cross-ledger parity catches any
  reintroduction regardless of how it's written); the regex guard
  catches the historical bug shape and acts as a fast-failing
  smoke test. Neither layer alone is sufficient; together they
  cover both shapes. Future contributors writing math should
  prefer routing through `pnl_for_trade(...)` rather than
  open-coding the multiplication in any form.
- **Cosmetic reformats in `src/trading/portfolio.py`** — lines 366
  and 381 single-line collapsed by black during the docstring
  edit pass. Behaviourally a no-op; the diff carries the cosmetic
  noise. Not a follow-up item; flagged here so a future audit
  reader reviewing the 20.2 commit doesn't read meaning into the
  reformat.
- **Phase 20.3 still gates the operator-comparison closure** —
  the math is correct as of 20.1, the discipline is locked as of
  20.2, but `docs/baselines.md` and the persisted per-strategy
  artefacts still quote pre-fix numbers. An operator running
  `--picks` against the new code and comparing to the old
  baseline table sees what looks like a regression but is the bug
  going away. 20.3's mechanical re-baseline closes this; until it
  lands, operator comparisons stay broken.
