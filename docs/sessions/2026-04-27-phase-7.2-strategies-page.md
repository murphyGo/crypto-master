## Work Summary

Added the first content page to the dashboard. `Strategies` lists every
registered analysis technique with its aggregate performance and lets
the user pick one to see a cumulative-P&L trend over time.

The page reuses existing components without re-implementing them:
`load_all_strategies()` for the technique list and `PerformanceTracker`
for both per-technique aggregates and historical records. The two
data-shaping helpers (`build_summary_dataframe`, `build_trend_dataframe`)
are pure functions — testable without spinning up Streamlit — and the
`render` function is the only Streamlit-runtime concern.

The page is wired into the chassis nav under a new "Sections" group;
7.3 and 7.4 will add their pages alongside it. The Home view's
Strategies card was promoted from `st.info` ("Coming in Phase 7.2") to
`st.success` ("Available now").

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Two pure helpers (`build_summary_dataframe`, `build_trend_dataframe`) separated from `render` | Lets us assert on data shaping without Streamlit, then leave only the rendering for AppTest smoke. Same separation pattern as `format_proposal` vs `default_decision_prompt` in Phase 6.2. |
| Use `exit_timestamp` for the trend X-axis, fall back to `analysis_timestamp` | A trade's *closing* time is when its P&L is realized — that's when the cumulative line should move. Falling back to analysis time keeps records that lack an exit timestamp from being dropped. |
| Skip pending / no-P&L records from the trend chart | Plotting them as zero would misrepresent flat performance during quiet periods; dropping them keeps the cumulative curve faithful to realized gains/losses. |
| Stored `pnl_percent` takes precedence over `calculate_pnl()` | If the tracker recorded a P&L explicitly (e.g. with fees applied), that value is more accurate than re-deriving from prices. Recompute only when stored value is missing. |
| Page accepts `strategies_dir` and `tracker` overrides | Tests can render the page in isolation against `tmp_path`; production code falls through to the defaults. |
| Inject Strategies under a new "Sections" nav group | Future pages (Trading, Feedback Loop) belong in the same group; preempting the structure now avoids a navigation rewrite at 7.3. |

## Potential Risks

- **No caching.** Every navigation to the page re-reads `strategies/`
  and re-loads every record file. Fine at current scale; revisit with
  `@st.cache_data(ttl=...)` once the performance directory grows or
  the loader becomes more expensive. Carry forward.
- **Trend chart treats every closed record as one tick.** A backtest
  with thousands of trades will render slowly; a future iteration
  could resample by day/week. Not urgent — the underlying data is
  small in real-world use.
- **`load_all_strategies` failures are silently logged.** A broken
  `.md` file in `strategies/` won't show up in the table at all,
  which could mask configuration drift. The chassis logger captures
  it; the dashboard could display a "X strategies failed to load"
  banner in a future iteration.
