## Motivation

Phases 5.1–5.4 produced four discrete capabilities — backtester,
performance analyzer, Claude-driven strategy improver, robustness
validation gate — but no glue. Until 5.5, evolving a technique
required the operator to manually invoke the improver, save the
output, run a backtest, run the gate, then decide whether to promote.
Every manual step was an opportunity to drop a candidate, skip a
gate, or promote something that should have been discarded.

Phase 5.5 closes the loop with a single orchestrator that takes a
candidate from generation through gating and (with explicit user
approval) into the active strategy set. The gate is the
non-negotiable; the user-approval step is the embodiment of CON-003.

## Work Summary

### `src/feedback/audit.py` (new)

Append-only JSONL audit log under `data/audit/feedback.jsonl`. Every
state transition in the loop emits an `AuditEvent` with the candidate
ID, technique name/version, actor, and free-form details (gate
verdicts, error messages, paths). Format choice:

- **JSONL** — one event per line, append-only, crash-safe (a partially
  written final line is the only damage a crash can cause; `read_all`
  skips it with a warning).
- **Stateless** — every `append()` opens, writes, closes, so concurrent
  writers do not need coordination beyond the OS-level append guarantee.
- **Greppable** — operators can answer "why did we discard candidate
  X?" without booting Python.

`AuditEventType` covers the lifecycle: GENERATED, BACKTESTED,
GATE_PASSED, GATE_FAILED, APPROVED, REJECTED, PROMOTED, DISCARDED,
ERRORED.

### `src/feedback/loop.py` (new)

`FeedbackLoop` is the orchestrator. Public surface:

- **Generation entry points** — `improve_existing`, `propose_new`,
  `from_user_idea`, `reevaluate`. Each runs the full cycle:
  improver call → load saved file → baseline backtest → robustness
  gate → decision. Errors are caught, audited as ERRORED with the
  exception message, the state file is persisted, then the exception
  is re-raised so the caller sees what happened.
- **User-decision API** — `approve(candidate_id, approver)` and
  `reject(candidate_id, approver, reason)`. Both refuse to act on
  candidates not currently in `AWAITING_APPROVAL`, so a re-evaluation
  must precede approval if a candidate has gone stale. `approve`
  re-reads the experimental file, rewrites frontmatter
  (`status: active`, fresh `updated_at`), writes the result to
  `active_dir`, then unlinks the source. The loop refuses to overwrite
  an existing active file at the target path.
- **State persistence** — one JSON file per candidate at
  `data/feedback/state/<candidate_id>.json`, written after every
  transition. `list_pending()` returns every candidate currently in
  `AWAITING_APPROVAL`. We persist state but don't try to auto-resume
  from a crash mid-cycle in this phase; the caller can inspect the
  state file and rerun manually.

`CandidateRecord` is the persisted snapshot — last-known status,
backtest run id, robustness summary, failed-gate names, decision
reason, timestamps. The full event history lives in the audit log;
the state file holds only the latest snapshot.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| JSONL for the audit log | Append-only, crash-safe, greppable, no schema migrations needed when fields evolve. |
| State snapshot per candidate, not a single rolling file | Avoids write contention if multiple cycles ever run concurrently; one file per candidate is the natural unit. |
| Promotion target defaults to `strategies/` root, not `strategies/active/` | Existing techniques (`sample_prompt.md`, `chasulang_ict_smc.md`) live at the root. Inventing a new `active/` subdir would have created two conventions for "an in-use technique." Frontmatter `status: active` already captures the lifecycle. The `active_dir` ctor argument lets a future phase override without touching the loop. |
| Manual resumption only (no auto-resume) | The `list_pending()` + `load_state()` pair is enough for an operator to pick up where a crashed run left off. Auto-resume needs a richer state machine (which step to retry, idempotency keys per backtest run); not worth the complexity for this phase. |
| Approve refuses non-pending candidates | Forces explicit re-evaluation before promotion if a candidate has gone stale (e.g. fresh OHLCV, new robustness config). Prevents "approve a discarded candidate by mistake." |
| Gate errors audited & re-raised; improver errors only re-raised | Improver errors happen before the cycle is entered, so there is no candidate ID to audit against yet. Gate errors happen mid-cycle when a candidate already exists, so we record ERRORED on it. |

## Potential Risks

- **No automated cleanup of old state files.** `data/feedback/state/`
  grows monotonically. Not a problem at current scale (a few candidates
  per session); a future phase may want a retention policy.
- **No file lock around state writes.** Concurrent cycles for the
  *same* candidate would race. Today there's only one entry point per
  candidate, so this is not exploitable; flag it if multi-tenant
  orchestration is added.
- **Promotion is not transactional.** If the OS crashes between
  `target.write_text()` and `source_path.unlink()`, the candidate
  ends up in both `experimental/` and `strategies/`. Operators can
  resolve this manually by inspecting frontmatter `status`. A more
  robust implementation would write to a temp path + atomic rename;
  not worth the complexity yet.
