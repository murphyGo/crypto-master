## Work Summary

Phase 14.2 closes DEBT-012 â€” the `EmailNotifier` SMTP_SSL alternative
deferred during Phase 13.4. Phase 13.4 shipped the email push backend
with STARTTLS-only (`smtplib.SMTP` + `starttls()` on port 587) because
Gmail / SendGrid / Mailgun / Office 365 / most corporate relays all
support STARTTLS as the recommended path. A handful of providers â€”
Yahoo Mail, AT&T, ProtonMail â€” only offer SMTP-over-SSL on port 465
with no STARTTLS option; on those, calling `starttls()` against an
already-encrypted channel raises `SMTPNotSupportedError`. Operators on
those providers had no path to enable email push without an
intermediary relay. DEBT-012 (Low) tracked it; Phase 14.2 closes it.

Implementation is a four-touch wedge with strict back-compat: (1)
`Settings.email_use_ssl: bool = Field(default=False)` in
`src/config.py` â€” env `EMAIL_USE_SSL=true` activates the SSL path,
default `False` keeps the existing STARTTLS path bytewise unchanged.
(2) `EmailNotifier.__init__` accepts a keyword-only `use_ssl: bool =
False` flag stored on `self._use_ssl`. (3) The inner `_send` closure
branches at send-time: `use_ssl=True` constructs `smtplib.SMTP_SSL(host,
port, timeout=...)` and skips the `starttls()` call (the channel is
already encrypted on connect); `use_ssl=False` constructs the existing
`smtplib.SMTP(host, port, timeout=...)` and calls `starttls()` before
login. Both paths still call `with smtp:` for socket cleanup, `login`,
and `send_message` identically. (4) `src/main.py::build_engine` reads
`settings.email_use_ssl` and forwards it to `EmailNotifier(...)`;
`.env.example` and `docs/deployment.md` document `EMAIL_USE_SSL` with
provider-specific guidance (Yahoo / AT&T / ProtonMail use
`EMAIL_USE_SSL=true` + `EMAIL_SMTP_PORT=465`).

Test coverage adds 2 net new (1158 â†’ 1160) in
`tests/test_proposal_notification.py`:
`test_email_notifier_uses_smtp_ssl_when_flag_set` patches both
`smtplib.SMTP_SSL` (to `_FakeSMTP`) and `smtplib.SMTP` (to a
`_wrong_constructor` stub that raises `AssertionError`) so the test
fails loudly if the SSL path accidentally hits the wrong constructor;
asserts `smtplib.SMTP_SSL` was called with `host`/`port=465`,
`starttls_called is False`, login was called with the right
credentials, and `send_message` fired exactly once.
`test_email_notifier_uses_starttls_when_flag_unset` is the mirror
image: patches `smtplib.SMTP` to `_FakeSMTP`, patches `smtplib.SMTP_SSL`
to a raising stub, and asserts the existing STARTTLS path is intact
with the default `use_ssl=False`. The cross-protection (raise on the
OTHER constructor) is the critical piece â€” without it, a regression
where both branches accidentally call `smtplib.SMTP` would silently
pass.

No ADR â€” extends Phase 13.4's `EmailNotifier` with one config branch;
nothing about component boundaries changes. The dispatcher still owns
per-channel failure isolation; the `Notifier` protocol contract is
unchanged.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| New `email_use_ssl: bool = Field(default=False)` (vs renaming the existing port field, vs auto-detecting on port==465) | Default `False` is bytewise-equal to pre-14.2 behaviour for every existing deployment â€” strict back-compat. Explicit boolean rather than implicit "if port==465" because RFC-strict port assignment is operator-controllable; some corporate relays put SMTP_SSL on non-standard ports and STARTTLS on 465, so port-sniffing would surprise. The flag is opt-in, named for intent, and matches the DEBT-012 Suggested Resolution verbatim. |
| Branch at send-time (inside `_send`) rather than constructing two `EmailNotifier` subclasses | One config flag, one notifier class. A subclass-per-transport split would force `build_engine` to choose between `EmailNotifier` and `EmailNotifierSSL` and would fragment the protocol implementation surface. The branch is 4 lines and the `with smtp:` cleanup, `login`, and `send_message` are shared by both paths. |
| Tests patch BOTH `smtplib.SMTP` AND `smtplib.SMTP_SSL` and raise on the wrong one | Without the cross-protection, a regression where both branches accidentally call the same constructor would silently pass â€” both tests would still find their `_FakeSMTP` instance. Raising `AssertionError` from the wrong-constructor stub turns "accidentally called the wrong thing" into a hard failure with a clear message. Belt-and-suspenders for a transport-selection invariant that has no other observable side-effect. |
| `use_ssl` is keyword-only on `EmailNotifier.__init__` | Matches the existing `timeout` keyword-only convention; positional ordering is irrelevant for a transport-selection flag, and keyword-only forces calls to read self-documenting at the call site. |
| No ADR | Extends the Phase 13.4 `EmailNotifier` with one config branch. Nothing about component boundaries changes; the `Notifier` protocol and the dispatcher's failure-isolation contract are untouched. ADRs are for component-shape decisions, not config-flag additions. |

## Potential Risks

- **`smtplib.SMTP_SSL` connect timeout and `starttls()` skip are
  silent transport-layer differences** â€” operators flipping
  `EMAIL_USE_SSL=true` against a STARTTLS-only server (port 587) will
  see a connection-level TLS handshake failure, not a clear "wrong
  transport for this server" message. The deployment doc flags the
  pairing requirement (`EMAIL_USE_SSL=true` + `EMAIL_SMTP_PORT=465`)
  but it's still on the operator to read and apply. A future
  enhancement could detect the mismatch and log a warning, but it's
  not worth the complexity for an opt-in flag that defaults off.
- **Body parity invariant across three push backends remains
  by-hand** (12.4 + 13.4 + 14.2 carry) â€” the SMTP_SSL path doesn't
  touch `_build_email_body` (still delegates to
  `_build_telegram_text`); content stays in sync with Slack and
  Telegram unchanged. But no mechanism *enforces* the invariant. If a
  future Telegram-only tweak lands, email + Slack would silently
  diverge. Out of scope for 14.2; flagged in the Phase 13 cross-check
  as a "low priority â€” only worth if a divergence is proposed" item.
- **Cross-protection in the new tests only fires if the wrong
  constructor is called at the `smtplib.<X>(...)` call site** â€”
  doesn't catch a regression where the SSL path accidentally also
  calls `starttls()` (which would raise `SMTPNotSupportedError` at
  runtime against a real server but is happily ignored on `_FakeSMTP`
  whose `starttls()` is a no-op flag flip). The
  `assert fake.starttls_called is False` check on the SSL path tests
  the *intent* but not the *runtime constraint*. Acceptable: the
  intent is what's load-bearing and the unit test pins it.
