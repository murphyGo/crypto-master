# Brownfield Execution Plan

## Purpose

This plan adapts the existing chronological Crypto Master development history
into AI-DLC unit-oriented construction tracking. The legacy plan archive remains
in `docs/legacy/development-plan.md`; this document defines how future work
should be routed by unit.

## Stage Strategy

| Stage | Applies When | Output |
|-------|--------------|--------|
| Functional Design | Public behavior, module contract, or workflow changes | Unit-specific design notes under `aidlc-docs/construction/<unit>/functional-design/` |
| NFR Requirements | Reliability, trading safety, data integrity, observability, security, or performance changes | Unit-specific NFR notes |
| NFR Design | NFR implementation approach is non-trivial | Unit-specific design notes |
| Infrastructure Design | Fly.io, credentials, process lifecycle, or external integration topology changes | Unit-specific infra notes |
| Code Generation | Any source/test/script change | Application code in workspace root, summary in construction docs when useful |
| Build and Test | After code changes or before sealing a unit change | Test/lint evidence in session logs and cross-checks |

## Legacy Phase Handling

Existing phases are considered completed historical work. They should not be
renumbered or moved out of `docs/legacy/development-plan.md`. Future changes
should reference both:

- the target unit from `aidlc-docs/inception/units/unit-of-work.md`
- any related legacy phase/debt/session/cross-check records

## Cross-Check Policy

Cross-checks should be generated by unit for new work. A unit cross-check should
include:

- related FR/NFR IDs from `docs/requirements.md`
- files touched
- tests run
- relevant technical debt added/resolved
- whether legacy behavior was preserved or intentionally changed
